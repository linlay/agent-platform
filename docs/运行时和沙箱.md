# 运行时和沙箱

## 当前状态

运行时目录默认由 `AP_RUNTIME_DIR` 派生，低频子目录由 `configs/runtime.yml` 的 `paths` 节控制。沙箱能力通过 Container Hub 接入；只要 agent 声明 `runtimeConfig.environmentId`，运行时就会为该 agent 注入 sandbox context，并在工具执行时使用 Container Hub session。

宿主机侧工具和 sandbox 工具有明确边界：host bash / FileTools 访问宿主机受本地权限配置约束，sandbox bash 通过 Container Hub 在远端或容器环境执行。

KBASE 的 `kbase-lance-engine` 是 Go runtime 直接管理的本地伴随进程，不是 Container Hub sandbox tool。Go 主程序仍以 `CGO_ENABLED=0` 构建，LanceDB 本地数据读写由随包分发的 Rust sidecar 承担。

## 核心流程

```text
agent.yml runtimeConfig
  -> environmentId / workspaceRoot / env / hostAccess / sandboxMounts
  -> session builder
  -> Container Hub mounts
  -> sandbox tool execution
```

默认基础挂载最多包括 `/workspace`、`/root`、`/skills`、`/pan`、`/agent`、`/owner`、`/memory`；其中 `/skills` 仅在 `run/agent` 级别默认挂载。`runtimeConfig.sandboxMounts` 可覆盖默认挂载模式或新增非默认目标路径，`platform: skills-market` 可显式挂载 `/skills-market`。

## 配置与接口

- `configs/runtime.yml` 的 `container-hub` 节：Container Hub 运行时行为主入口；代码默认值仅作为兜底。
- `AP_RUNTIME_DIR`：runtime 总目录。
- `AP_RUNTIME_REGISTRIES_DIR`：provider / model / mcp server registry 根目录。
- `AP_RUNTIME_CHATS_DIR`：chat 与上传资源目录。
- `AP_RUNTIME_MEMORY_DIR`：memory 数据目录。
- `AP_RUNTIME_KBASE_DIR`：KBASE runtime 索引目录。
- `AP_RUNTIME_PAN_DIR`：pan 目录映射。
- `configs/runtime.yml` 的 `paths.*`：低频运行态子目录，例如 owner、agents、teams、root、automations、skills-market、tools。
- `AP_CONTAINER_HUB_BASE_URL`：非空即启用 Container Hub。
- `container-hub.default-environment-id`：默认 environment id。
- `runtimeConfig.workspaceRoot`：host workspace 边界，可为绝对路径或 `@chat`。
- `runtimeConfig.hostAccess`：host-side `bash/file_*` 额外读写根，不影响 sandbox mounts。
- `runtimeConfig.env`：字面量字符串 map，注入 host bash 或 Container Hub。
- `runtimeConfig.sandboxMounts`：沙箱挂载扩展。

## KBASE Lance sidecar

runtime 只在存在 enabled KBASE capability 的 startup refresh 或第一次 Lance 操作时启动一个进程级 sidecar，不为每个 Agent 启动独立进程。可执行文件优先使用显式 `AP_BUILTINS_BIN`，未设置时解析 release bundle `bin/kbase-lance-engine[.exe]`、support package registry，最后才是开发/容器覆盖 `AP_KBASE_LANCE_ENGINE`。普通安装不需要用户安装 Rust、Python、GitHub 插件或数据库服务。

启动协议：

- Go 生成 256-bit 随机 token，只通过 `KBASE_LANCE_TOKEN` 子进程环境传入；所有私有 HTTP 请求必须携带 Bearer token。
- sidecar 必须绑定 `127.0.0.1:0` 随机 loopback 端口，stdout 只输出一行 protocol/version/address ready handshake，运行日志写 stderr 并由 Go 加上 `[kbase-lance]` 前缀。
- handshake 超时 10 秒；健康检查超时 2 秒；启动/健康失败会按 1、2、4 秒退避重试，最终将 KBASE 标记为 unavailable。
- Go 传入 parent PID，sidecar 有父进程存活 watchdog。runtime 关闭时调用私有 shutdown，最多等待 5 秒后强制终止。
- sidecar 缺失不阻止普通 Agent、chat、memory、automation 启动；enabled KBASE 路径明确返回 engine unavailable，不会回退 SQLite 检索。存在 required `mode: KBASE` 时健康检查失败；只有普通 Agent optional capability 时健康检查报告 degraded 但仍返回 ready。

Docker 镜像使用 Debian bookworm slim/glibc，Go runtime 与 sidecar 同镜像、同一非 root UID 运行，KBASE runtime 目录必须可写。当前容器 `HEALTHCHECK` 只验证 runtime 进程与 sidecar 可执行文件存在；真实 protocol handshake/health 通过 KBASE status 和 supervisor 完成，不应把容器 probe 误解为已执行 Lance 查询。

sidecar 由相邻 `agent-platform-builtins/kbase-lance-engine` 的 versioned archive 提供。`./scripts/sync-local-builtins.sh` 每次在隔离目录按本地 `VERSION` 重建 dbx/httpx/sidecar 与 Poppler launcher/archive，并将校验后的内容原子写入 `build/builtins/<host>/`；`rg` 仅复用校验后的预编译 artifact，Poppler native runtime 仅校验并重新打包。同步使用临时本地 checksum lock且不镜像到 `release-local/bin/`；cache 激活后，精确 native host 的更高干净版本经 `yes` 成为组件目标，落后平台在 VERSION/commit 匹配后自动跟随。正式 schema v2 lock 同时保存组件目标 release 和每个 target 的实际 release，runner 只允许改自己的 target；固化 archive 失败、同版本不同 commit/SHA 或 lock 并发变化均不写正式 lock。`make run` 只引用本机 cache，`make release-program` 只消费 cache 中已验证的发布内容，二者都不下载或构建 Rust。Docker 从同一 cache 的 `linux-<arch>` 目录复制 sidecar。six-platform archive、签名/notarization 与真机 smoke 仍必须由外部 sidecar release CI/对应 runner 验证。

## 约束与注意事项

- `runtimeConfig.env` 不支持 `${VAR}` 展开。
- env key 不能为空，不能包含空白字符或 `=`。
- catalog API 不回显 `runtimeConfig.env`。
- `sandbox` 不再属于 `contextConfig.tags`，由 `runtimeConfig.environmentId` 控制。

目录型 agent 的 `.config/` 是工具的专属 XDG 配置根。host bash 与 WebSocket terminal 使用 `<agentDir>/.config`；Container Hub 通过默认只读 `/agent` mount 使用 `/agent/.config`。因此沙箱能读取静态工具配置，但不能把登录态或其他可写 state 写回 agent 定义目录。
- `configs/` 目录固定为项目根下 `configs/`。
- `kbase-lance-engine` 的 HTTP 协议是 loopback 私有协议，不向客户端、sandbox 或网关暴露。
