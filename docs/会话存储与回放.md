# 会话存储与回放

## 当前状态

Chat 数据以本地文件系统为主，根目录由 `AP_RUNTIME_CHATS_DIR` 控制。运行事件、StepLine、raw messages、上传资源、归档与搜索都围绕 chat store 组织；chat/run summary 由 `agentKey/teamId` 身份对区分公开 owner：`agentKey` 为空且 `teamId` 非空时为 orchestrated Team，其余为 agent-owned。

## 核心流程

```text
query request
  -> ensure chat
  -> stream events
  -> StepWriter 聚合 StepLine
  -> <chatId>.jsonl
  -> chat summary / archive / replay
```

默认布局：

```text
<AP_RUNTIME_CHATS_DIR>/
  chats.db
  <chatId>.jsonl
  <chatId>/
    <uploaded-file>
    .btw/
      <btwId>.jsonl
    .tools/
      results/
        <toolId>.json
      state/
        file-versions.json
      planning/
        <planningId>.md
      plan-tasks/
        <runId>_plan.json
```

Chat JSONL 使用 UTF-8 文本，每条物理行必须且只能包含一个 JSON object。换行符可为 LF 或 CRLF，最后一行可以没有换行符；空行、多行 object、同行多个 JSON 值、数组、标量和语法错误都会使整次读取失败。每个 object 必须包含 `_type`，顶层 `type` 不作为替代字段。

当前允许的 `_type` 只有 `query`、`react`、`react-tool`、`event`、`steer`、`submit`、`compact.checkpoint`、`compact.tool`。其他值均违反存储 schema。`event.type:"awaiting.ask"` 与 `event.type:"planning.snapshot"` 不落盘：awaiting 只写入 step 的 `awaiting[]`，planning snapshot 只在回放结果中合成。

JSONL 中，`_type:"query"` 可携带单个 system init 快照；`_type:"react".seq` / `_type:"react-tool".seq` 是模型调用分组 id，不是 JSONL 物理行号。`PLAN-EXECUTE` agent mode 也只写 `react` / `react-tool`，用顶层 `stage` 表达 planning、execute 或 summary 阶段；`plan-execute` 不是行类型。`plan-execute:*` system cache key 保持不变。

system init 使用 query 顶层单数 `system`，step 只保存 `systemRef`：

```json
{"_type":"query","chatId":"chat-1","runId":"run-1","updatedAt":1782720096241,"query":{"agentKey":"coder","role":"user","message":"实现功能"},"system":{"agentKey":"coder","cacheKey":"coder:planning","fingerprint":"sha256:...","systemMessage":{},"tools":[],"model":{}}}
{"_type":"react","chatId":"chat-1","runId":"run-1","updatedAt":1782720096242,"systemRef":{"agentKey":"coder","cacheKey":"coder:planning","fingerprint":"sha256:..."},"messages":[]}
```

system 身份由 `agentKey + cacheKey + fingerprint` 共同确定；`cacheKey` 只表达模式和阶段，不拼接 agent。相同快照已在 chat 中注册时，新 query 可以省略 `system`，后续 step 继续引用已注册快照。`system` 与 `systemRef` 出现时三项均为必填；step 的 `systemRef` 只能包含这三个字段，并按三元组精确命中已注册快照，未命中即返回 not found。`query.systems`、query 的 `systemRef`、`step.systems`、step 内联 `system`、缺失标识或错误字段类型均返回 `chat_storage_schema_violation`，错误不回显 system prompt。

## Team owner、事件与历史视图

legacy Team 延续原有存储：`agentKey` 为当前成员，同时保留 `teamId`。orchestrated Team 仅保存 `teamId`，公开 summary/run 的 `agentKey` 为空；委派到不同成员不会修改 chat 根 owner。`CHATS` 与 `RUNS` 只保存这对身份字段；含已删除 `OWNER_TYPE_` 列的数据库不再受支持，必须清空后重建。

`CHATS.AGENT_MODE_` 表示 chat 当前（最后一次公开 owner）的 mode，`RUNS.AGENT_MODE_` 保存每次 run 不可变的 mode。普通 agent 保存规范 API mode；orchestrated Team 固定保存 `TEAM`；legacy Team 随实际成员切换而更新 chat 摘要，但已有 run 保留原 mode。归档与恢复会原样保留 chat/run mode，派生 chat 会复制已纳入历史的 mode。active 与 archive 仅创建包含这些字段的最新 SQLite schema，不会为历史记录补列或回填。

orchestrated Team 的成员流仍写入同一个 `<chatId>.jsonl`，通过 `taskSubAgentKey`、task 元数据和 stream actor 信息区分：

- 所有 `agent_delegate` 成员正文都保持 task-scoped，并以实际成员标记；不会拼接到根 run 最终正文。
- 根 run 的 `AssistantText`、summary 和非流式 `content` 只保存隐藏总控生成的唯一 Team 最终回答。
- 总控 reasoning 与隐藏 `agent_delegate` 事件不进入客户端事件流；结构化成员结果只作为内部模型上下文参与本次调度。

raw history 按消费方构建，不直接把整份 JSONL 交给每个成员：

- 根历史跳过所有带 `taskId/subAgentKey` 的 child query 和带 `taskSubAgentKey` 的 child step，避免内部任务 prompt、reasoning 与工具结果进入协调器后续上下文。
- 成员读取 Team member 专用历史：自己的 step 保留完整 tool chain；其他成员只转换为带 `[Team member <key>]` 标记的最终正文，并去除 reasoning/tool calls。
- 根 Team 最终正文对成员可见。
- 自定义 task 仍继承该成员可见的 Team 历史；task prompt 和其他成员中间执行链不会进入后续共享 raw history。
- 历史读取兼容旧 `team_delegate/team_invoke` 名称，但只生成 `agent_delegate agentKeys=...` 安全摘要；新 run 不再接受旧工具调用。

`/api/chat` replay 通过 `taskSubAgentKey/subAgentKey` 保留成员 task 归属；live stream 中更完整的 `actor/presentation` 是展示元数据。archive、restore 与 derive 同步保留 `agentKey/teamId`。orchestrated Team 的隐藏执行 key 不是 chat 或 run 的公共 owner；客户端应始终使用 `teamId` 操作该 run。

CODER planning 的 planning 与 execute system 分别注册在初始 query 和“执行规划” synthetic query。PLAN_EXECUTE 的 execute、summary 新快照通过 `kind:"system-init"`、`role:"system"`、`hidden:true` 的存储专用 query 注册；HITL continuation 若系统快照变化也先注册该 hidden query，再写带精确 `systemRef` 的模型调用 step。orchestrated Team 的持久化 system `agentKey` 使用 `team:<teamId>`，不会写入 `__team__:` 隐藏协调器 key；这些行参与 system cache 和 LLM chat 重建，但不进入 live 客户端事件、chat 回放、raw messages、搜索或导出。

`POST /api/btw` 创建隐藏只读分支时，会在 FileStore 写锁内把当前 `<chatId>.jsonl` 原子复制为 `<chatId>/.btw/<btwId>.jsonl`；之后复用普通 StepWriter，只把 query/step/event/submit 行追加到该副本。续聊从同一文件加载 raw messages 与 system init，因此能看到创建时的父会话快照和之前的 BTW 消息，但不会自动合并父 chat 后续新增内容。

同一 assistant turn 产生多个 tool call 时，JSONL 只持久化一条 assistant tool-call message，多个调用合并在同一个 `tool_calls[]` 中。平台内部 id 写在每个 call 内部；assistant message 外层只保留 `_msgId` 和 `ts`，不再写 `_toolId` / `_actionId`：

```json
{
  "role": "assistant",
  "tool_calls": [
    {
      "id": "call_00",
      "type": "function",
      "function": { "name": "file_read", "arguments": "{}" },
      "_toolId": "call_00"
    },
    {
      "id": "call_01",
      "type": "function",
      "function": { "name": "file_glob", "arguments": "{}" },
      "_toolId": "call_01"
    }
  ],
  "_msgId": "m_xxx",
  "ts": 1782720096241
}
```

`role:"tool"` 结果消息在外层保留 `_toolId` / `_actionId` / `tool_call_id`。`rawMessages` 直接使用该结构。

CODER planning mode 中，模型调用 `finalize_planning` 后，后端只把 Markdown 正文写入 `<chatId>/.tools/planning/<planningId>.md`；规划确认的 `awaiting.ask(mode=planning)` 必须在 `planning` 对象中同时携带 `planningId` 与 `planningFile`，并随 `_type:"react"` step 的 `awaiting[]` 持久化。awaiting mode 只允许 `question`、`approval`、`form`、`planning`。JSONL 会保留原始 assistant `tool_calls`，并在用户 approve/reject 后追加同一个 `tool_call_id` / `name:"finalize_planning"` 的 `_type:"react-tool"` / `role:"tool"` 结果。`GET /api/chat` 按 `planningFile` 读取 Markdown，并把最新一条放到顶层 `planning` 字段；同时会在对应 `awaiting.ask` 前合成 replay-only 的 `planning.snapshot` 事件供时间线使用。

`plan_add_tasks` / `plan_update_task` 的当前任务列表会按 run 写入 `<chatId>/.tools/plan-tasks/<runId>_plan.json`；`/api/chat` 从该目录读取最新任务快照作为顶层 `plan`。重启后同一 `chatId` 的新 query 不复活旧 run / SSE，但在可使用 `plan_get_tasks` 或 `plan_update_task` 且不是 planning 阶段时，会从最新任务快照恢复当前 run 的 plan task 内存状态，并把当前任务上下文注入 system prompt；后续 `plan_update_task` 会写入新 run 自己的快照。`plan_add_tasks` 默认追加到当前 plan；仅当调用方显式传入 `mode:"new"` 时，服务端创建新的 `planId`，并以本次传入 tasks 作为新 plan 的完整初始任务列表，旧 snapshot 文件保留为历史。

## 约束与注意事项

- `GET /api/chat` 默认返回 `events`，`includeRawMessages=true` 时附带 `rawMessages`。
- `GET /api/chats?mode=REACT,TEAM&limit=20` 可按 chat 当前 mode 筛选并限制返回条数；mode 支持逗号分隔、重复参数、大小写无关和 `PLAN_EXECUTE` 别名，集合内部为 OR，与 `agentKey`、`lastRunId` 为 AND。`limit` 省略时返回全部，传入时必须是正整数，且在筛选和 `UPDATED_AT_ DESC, CHAT_ID_ DESC` 排序后截断。WebSocket `/api/chats` 使用同名 `mode`、`limit` 字段；旧 `agentMode` 会返回 400。
- `POST /api/chat/derive` 会从一个无 active run / 无 pending awaiting 的 active chat 派生出独立新 chat。派生时截取到目标已完成 run 的最后一行 JSONL，复制上传资源、`.tools/results`、planning 文件和 plan task 快照，并重写标准 `chatId` / `runId` / `requestId` 以及平台生成的 resource URL / 本地 chat 目录路径；`.tools/artifacts.json` 仅保留已派生 run 的条目，并重写其中的 `chatId`、`runId` 与 resource URL；`.tools/state` 不复制，原 chat 与新 chat 后续互不关联。
- Team chat 派生、归档和恢复会保留原 `agentKey/teamId`；legacy Team 同时保留成员 `agentKey`，orchestrated Team 的公开 `agentKey` 保持为空。
- `.btw` 不进入 `chats.db`、chat 列表、搜索、Markdown 导出、未读统计或 `/api/chat/jsonl`；`/api/resource` 也拒绝访问该内部目录。客户端持有 `btwId` 继续对话，v1 不提供 BTW 列表接口。
- archive/restore 会随整个 `<chatId>/` 目录移动 BTW 文件，删除 chat 会一起删除；derive 新 chat 时明确跳过 `.btw`，避免复制仍引用原 chat/run 的隐藏历史。
- `GET /api/chat/jsonl` 返回 `<chatId>.jsonl` 原文；WS `/api/chat/jsonl` 在 `response.data` 返回同一原始字符串；active chat 不存在时读取归档库中的 `JSONL_CONTENT_`。active、archive、BTW、derive、compact、search、replay 与 raw history 共用同一 JSONL schema 入口，任一行非法时整次读取失败，不返回部分结果。schema 错误返回 HTTP/WS `422 chat_storage_schema_violation`；时间字段错误继续返回 `422 time_contract_violation`。
- `chat_storage_schema_violation` 的安全数据字段为 `code`、`field`、`location`、`expected`、可选 `actual`、`status:422`、`retryable:false`。`location` 使用 1-based 物理行号；错误不包含整行 JSON 或 system prompt。
- 文件资源落在 chat 目录下，resource ticket 只保护访问，不改变落盘位置。
- 大工具结果会以 tool call 为粒度写入 `<chatId>/.tools/results/<toolId>.json`；live event 与 JSONL 只保留预览和 `resultRef`。
- `plan_add_tasks` / `plan_update_task` 的任务快照写入 `<chatId>/.tools/plan-tasks/<runId>_plan.json`，是 `/api/chat` 顶层 `plan` 的事实源。`plan_add_tasks mode:"new"` 会生成新的 `planId` 并让新 tasks 成为当前 plan；各 run 的 snapshot 文件独立保存。
- 已发布 artifact 的功能性事实源是 `<chatId>/.tools/artifacts.json`。manifest 原子写入 `chatId` 与按发布顺序保存的项目；每项包含公开 artifact 元数据，以及派生/排序所需的 `runId`、`publishedAt`。实际 artifact 文件仍在 chat 根目录或 `artifacts/<runId>/`，绝不放入 `.tools`。
- `.tools` 不通过 `/api/resource` 暴露；展开完整工具结果使用 `/api/tool-result`。
- `file-tools.read-before-write-scope: chat` 时，跨 run 文件版本证明写入 `<chatId>/.tools/state/file-versions.json`，只保存路径、sha、mtime、size 等元数据。
- 每条 query 最多写一个顶层 `system`；快照首次出现或完整 payload 变化时写入，未变化时省略。step 不内联 `system` / `systems`，只写精确的 `systemRef`。
- child `_type:"query"` 与 child step 仍会为了 system-init 定位和 replay 写入 JSONL，但标准 `LoadRawMessages` 不会把它们误当成共享对话；Team member 历史必须使用专用过滤视图。
- `awaiting.ask` 发出时会立即 flush 当前 canonical step，并把完整等待项写入 step 的 `awaiting[]`；`CHATS.AWAITING_*` 只作为可恢复等待状态旗标，不再为 `awaiting.ask` 写 `_type:"event"`。
- 如果同一组 `tool_calls[]` 中任意调用需要 awaiting / approval，`_type:"react"` 行只写 assistant `tool_calls[]` 与 `awaiting[]`，确认完成前不写任何 sibling tool result。
- awaiting 解除后，本组所有 tool result 写入同 `seq` 的 `_type:"react-tool"` continuation；`messages[]` 按原始 `tool_calls[]` 顺序排序，不按实际完成时间排序。
- OpenAI-compatible 请求直接使用当前消息顺序；该层不重排 tool-call/result、不补齐缺失 result，也不删除孤立 tool message。
- `source.publish` 是工具结果派生的 UI 来源卡片。标准写入挂到对应 `react-tool` step 的顶层 `sources.items[]`，不进入 `messages[]`，因此不会参与下一轮 LLM 上下文；`GET /api/chat` replay 会按 `toolId` 在对应 `tool.result` 后合成 `source.publish` 事件。当前 `_type:"event"` 的 `source.publish` 也可回放。
- `artifact.publish` 的 SSE payload 带 `toolId`，子任务归属明确时也带 `taskId`。对应 `react-tool` step 的 `artifacts.items[]` 仅保留本次 `artifact_publish` 调用的审计 sidecar（包含 tool/run、可选 task、批次项目、timestamp、liveSeq），不累计、不在 replay 时合成事件，也不参与 `/api/chat` 的 artifact 恢复。`GET /api/chat.data.artifact` 只读取 manifest。
