# 会话存储与回放

## 当前状态

Chat 数据以本地文件系统为主，根目录由 `CHATS_DIR` 控制。运行事件、StepLine、raw messages、上传资源、归档与搜索都围绕 chat store 组织。

## 核心流程

```text
query request
  -> ensure chat
  -> stream events
  -> StepWriter 聚合 StepLine
  -> <chatId>.jsonl
  -> chat summary / archive / replay
```

默认布局：

```text
<CHATS_DIR>/
  index.json
  <chatId>.jsonl
  <chatId>/
    <uploaded-file>
    .tools/
      results/
        <toolId>.json
      state/
        file-versions.json
      plans/
        <planningId>.md
```

StepLine 快照中，`_type:"query"` 可携带 system init 快照；`_type:"react".seq` 是 react mode 的模型调用分组 id，不是 JSONL 物理行号。

CODER planning mode 中，`planning_write` 成功后只写 Markdown 文件；计划确认的 `awaiting.ask(mode=plan)` 会在 `plan` 对象中携带 `planningId` 与 `planningFile`，并随 `_type:"react"` step 的 `awaiting[]` 持久化。Markdown 正文只保存在 `<chatId>/.tools/plans/<planningId>.md`。`GET /api/chat` 回放时按 `awaiting[].plan.planningFile` 读取文件，并把最新一条放到顶层 `planning` 字段；旧 `_type:"planning"` / `planning.snapshot` 索引行仍兼容读取。

## 约束与注意事项

- `GET /api/chat` 默认返回 `events`，`includeRawMessages=true` 时附带 `rawMessages`。
- `GET /api/chat/jsonl` 返回 `<chatId>.jsonl` 原文；WS `/api/chat/jsonl` 在 `response.data` 返回同一原始字符串；active chat 不存在时读取归档库中的 `JSONL_CONTENT_`，不做 replay、格式化或脱敏。
- 文件资源落在 chat 目录下，resource ticket 只保护访问，不改变落盘位置。
- 大工具结果会以 tool call 为粒度写入 `<chatId>/.tools/results/<toolId>.json`；live event 与 JSONL 只保留预览和 `resultRef`。旧 `<chatId>/.tool-results/<toolId>.json` 继续可通过 `/api/tool-result` 读取。
- `.tools` 与旧 `.tool-results` 不通过 `/api/resource` 暴露；展开完整工具结果使用 `/api/tool-result`。
- `file-tools.read-before-write-scope: chat` 时，跨 run 文件版本证明写入 `<chatId>/.tools/state/file-versions.json`，只保存路径、sha、mtime、size 等元数据。
- `system` 快照首次出现必写，后续仅在 model/messages/tools/stream 变化时写入。
- `awaiting.ask` 发出时会立即 flush 当前 canonical step，并把完整等待项写入 step 的 `awaiting[]`；`CHATS.AWAITING_*` 只作为可恢复等待状态旗标，不再为 `awaiting.ask` 写 `_type:"event"`。
