# 会话存储与回放

## 当前状态

Chat 数据以本地文件系统为主，根目录由 `AP_RUNTIME_CHATS_DIR` 控制。运行事件、StepLine、raw messages、上传资源、归档与搜索都围绕 chat store 组织。

## 核心流程

```text
query request
  -> ensure chat
  -> stream events
  -> StepWriter 聚合 StepLine
  -> <chatId>.jsonl
  -> chat summary / archive / replay
```

默认布局：

```text
<AP_RUNTIME_CHATS_DIR>/
  chats.db
  <chatId>.jsonl
  <chatId>/
    <uploaded-file>
    .tools/
      results/
        <toolId>.json
      state/
        file-versions.json
      plans/
        <planningId>.md
      plan-tasks/
        <runId>_plan.json
```

StepLine 快照中，`_type:"query"` 可携带 system init 快照；`_type:"react".seq` / `_type:"react-tool".seq` 是 react mode 的模型调用分组 id，不是 JSONL 物理行号。`react-tool` 只表示 submit/answer 后补写的 tool result continuation，并复用对应 `_type:"react"` 的 `seq`。

同一 assistant turn 产生多个 tool call 时，JSONL 只持久化一条 assistant tool-call message，多个调用合并在同一个 `tool_calls[]` 中。平台内部 id 写在每个 call 内部；assistant message 外层只保留 `_msgId` 和 `ts`，不再写 `_toolId` / `_actionId`：

```json
{
  "role": "assistant",
  "tool_calls": [
    {
      "id": "call_00",
      "type": "function",
      "function": { "name": "file_read", "arguments": "{}" },
      "_toolId": "call_00"
    },
    {
      "id": "call_01",
      "type": "function",
      "function": { "name": "file_glob", "arguments": "{}" },
      "_toolId": "call_01"
    }
  ],
  "_msgId": "m_xxx",
  "ts": 1782720096241
}
```

`role:"tool"` 结果消息仍在外层保留 `_toolId` / `_actionId` / `tool_call_id`。`rawMessages` 直接输出当前新形态，不兼容旧 JSONL 中 assistant 外层 `_toolId` / `_actionId` 的拆分形态。

CODER planning mode 中，模型调用 `finalize_planning` 后，后端只把 Markdown 正文写入 `<chatId>/.tools/plans/<planningId>.md`；计划确认的 `awaiting.ask(mode=plan)` 会在 `plan` 对象中携带 `planningId` 与 `planningFile`，并随 `_type:"react"` step 的 `awaiting[]` 持久化。JSONL 会保留原始 assistant `tool_calls`，并在用户 approve/reject 后追加同一个 `tool_call_id` / `name:"finalize_planning"` 的 `_type:"react-tool"` / `role:"tool"` 结果。`GET /api/chat` 回放时按 `awaiting[].plan.planningFile` 读取文件，并把最新一条放到顶层 `planning` 字段；同时会在对应 `awaiting.ask` 前合成 replay-only 的 `planning.snapshot` 事件供时间线使用。

`plan_add_tasks` / `plan_update_task` 的当前任务列表会按 run 写入 `<chatId>/.tools/plan-tasks/<runId>_plan.json`；`/api/chat` 优先从该目录读取最新任务快照作为顶层 `plan`，仅在缺少快照时兼容读取历史 JSONL StepLine 的 `plan`。`plan_get_tasks` 仍只读当前 run 的内存状态，不从该快照恢复。

## 约束与注意事项

- `GET /api/chat` 默认返回 `events`，`includeRawMessages=true` 时附带 `rawMessages`。
- `GET /api/chat/jsonl` 返回 `<chatId>.jsonl` 原文；WS `/api/chat/jsonl` 在 `response.data` 返回同一原始字符串；active chat 不存在时读取归档库中的 `JSONL_CONTENT_`，不做 replay、格式化或脱敏。
- 文件资源落在 chat 目录下，resource ticket 只保护访问，不改变落盘位置。
- 大工具结果会以 tool call 为粒度写入 `<chatId>/.tools/results/<toolId>.json`；live event 与 JSONL 只保留预览和 `resultRef`。
- `plan_add_tasks` / `plan_update_task` 的任务快照写入 `<chatId>/.tools/plan-tasks/<runId>_plan.json`，作为 `/api/chat` 顶层 `plan` 的回放事实源；历史 JSONL 顶层 `plan` 只作兼容 fallback。
- `.tools` 不通过 `/api/resource` 暴露；展开完整工具结果使用 `/api/tool-result`。
- `file-tools.read-before-write-scope: chat` 时，跨 run 文件版本证明写入 `<chatId>/.tools/state/file-versions.json`，只保存路径、sha、mtime、size 等元数据。
- `system` 快照首次出现必写，后续仅在 model/messages/tools/stream 变化时写入。
- `awaiting.ask` 发出时会立即 flush 当前 canonical step，并把完整等待项写入 step 的 `awaiting[]`；`CHATS.AWAITING_*` 只作为可恢复等待状态旗标，不再为 `awaiting.ask` 写 `_type:"event"`。
- 如果同一组 `tool_calls[]` 中任意调用需要 awaiting / approval，`_type:"react"` 行只写 assistant `tool_calls[]` 与 `awaiting[]`，确认完成前不写任何 sibling tool result。
- awaiting 解除后，本组所有 tool result 写入同 `seq` 的 `_type:"react-tool"` continuation；`messages[]` 按原始 `tool_calls[]` 顺序排序，不按实际完成时间排序。
- `source.publish` 是工具结果派生的 UI 来源卡片；新写入不再单独写 `_type:"event"` 行，而是挂到对应 `react-tool` step 的顶层 `sources.items[]`，不进入 `messages[]`，因此不会参与下一轮 LLM 上下文。`GET /api/chat` replay 会按 `toolId` 在对应 `tool.result` 后合成 `source.publish` 事件；历史 event line 仍兼容读取。
