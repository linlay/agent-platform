# 配置化说明

Go runtime 的配置事实源分三层：

```text
代码默认值 < configs/*.yml < .env allowlist / 启动参数
```

`.env` 现在只作为启动/部署注入层。除本文列出的 allowlist 外，旧环境变量不再作为 runtime 配置入口；高级和低频配置统一写入 `configs/*.yml`，修改后需要重启服务。现场排障只保留明确列出的 debug allowlist。

## `.env` Allowlist

根 `.env.example` 只展示以下 key，也只有这些 key 会作为 `.env` 正式入口：

| 环境变量 | 默认值 | 说明 |
|---|---|---|
| `SERVER_PORT` | `11949` | 本地 `make run` 与 compose 外部入口端口；`--port` 启动参数优先级更高 |
| `AP_RUNTIME_DIR` | `runtime` | runtime 总目录 |
| `AP_RUNTIME_REGISTRIES_DIR` | `${AP_RUNTIME_DIR}/registries` | provider / model registry 根目录 |
| `AP_RUNTIME_CHATS_DIR` | `${AP_RUNTIME_DIR}/chats` | chat 落盘目录 |
| `AP_RUNTIME_MEMORY_DIR` | `${AP_RUNTIME_DIR}/memory` | memory 落盘目录 |
| `AP_RUNTIME_KBASE_DIR` | `${AP_RUNTIME_DIR}/kbase` | KBASE runtime 索引目录 |
| `AP_RUNTIME_PAN_DIR` | `${AP_RUNTIME_DIR}/pan` | pan 目录映射 |
| `AP_CONTAINER_HUB_BASE_URL` | 空 | Container Hub 服务地址；非空即启用 Container Hub |
| `AP_CHAT_RESOURCE_TICKET_SECRET` | 空 | resource ticket 签名密钥；设置后启用 `/api/resource` ticket 校验 |
| `AP_DEBUG_LLM_CONSOLE` | `request,usage` | 可选 LLM console 日志分类，逗号分隔 |
| `AP_DEBUG_LLM_CHAT_RECORD` | `false` | 是否记录 LLM chat completion JSON trace 并发送 `debug.llmChat` 事件 |

不在 allowlist 里的旧环境变量会被忽略。resource ticket TTL 也不再有环境变量入口，请使用 `configs/runtime.yml -> resource.ticket-ttl-seconds`。

## Server

本地运行约定：

- `make run` 会先加载根目录 `.env`
- `make run` 的端口链路为 `SERVER_PORT -> 11949`
- 直接执行 `go run ./cmd/agent-platform` 不会自动加载 `.env`
- 直接 `go run` 时，未设置 `SERVER_PORT` 或 `--port` 则代码默认监听 `8080`
- 服务端口不从 `configs/runtime.yml` 读取，最终优先级是代码默认 `8080` -> `SERVER_PORT` -> `--port`

## Runtime YAML

`configs/runtime.yml` 是运行时主配置入口，模板见 `configs/runtime.example.yml`。当前支持的顶层配置包括：

| YAML 节 | 说明 |
|---|---|
| `paths` | runtime 目录，例如 `tools-dir`、`owner-dir`、`agents-dir`、`teams-dir`、`root-dir`、`automations-dir`、`kbase-dir`、`skills-market-dir`；高频目录仍可由 `.env` allowlist 覆盖 |
| `skills` | skills prompt 拼接限制，例如 `max-prompt-chars` |
| `query` | query 消息构造策略，例如 `advanced-user-prompt` |
| `auth` | `/api/**` JWT 鉴权开关、JWKS 地址、issuer 和 JWKS cache |
| `resource` | resource ticket TTL 等非敏感运行策略 |
| `budget` | run、model、tool、HITL/awaiting 超时和调用上限 |
| `automation` | automation orchestrator 开关、默认时区和 dispatch 并发 |
| `memory` | memory store 文件名、召回条数、上下文字符上限、搜索默认 limit 和混合检索权重 |
| `kbase` | KBASE 新建 agent 默认回答模型配置、workspace refresh debounce 与兜底 reconcile 间隔 |
| `container-hub` | Container Hub URL、token、默认 environment、请求超时、sandbox 级别和回收策略 |
| `desktop` | desktop action / CDP bridge 地址和超时 |
| `cors` | `/api/**` CORS 策略 |
| `billing` | 计费展示货币 |

### Query

`query.advanced-user-prompt` 默认关闭。开启后，native Go LLM 请求会把 provider-side user message 包装为 `advanced_user_prompt`，其中 `run_context` 只包含 runId、requestId、agentKey、teamId、role、当前时间、时区和 scene 摘要；真实用户请求位于 `user_message`。该开关不修改 `/api/query` 的原始 `message`，也不改变 PROXY / ACP CODER 的上游转发 payload。

### Auth / Ticket

Auth 默认开启，默认使用 `configs/local-public-key.pem` 本地公钥。需要 JWKS 时写入：

```yaml
auth:
  enabled: true
  jwks-uri: https://example.com/.well-known/jwks.json
  issuer: example-issuer
  jwks-cache-seconds: 60
```

本地公钥规则：

- 本地公钥文件固定为 `configs/local-public-key.pem`
- 该路径和文件名不是配置项
- 配置了 `auth.jwks-uri` 时走 JWKS 模式，不读取本地公钥文件
- `AP_CHAT_RESOURCE_TICKET_SECRET` 只负责 ticket 签名密钥
- ticket TTL 写在 `resource.ticket-ttl-seconds`

### Paths

默认目录来自 `AP_RUNTIME_DIR`。只在目录需要脱离 runtime 根目录时，才在 `configs/runtime.yml` 写 `paths`：

```yaml
paths:
  tools-dir: /data/agent-platform/tools
  owner-dir: /data/agent-platform/owner
  agents-dir: /data/agent-platform/agents
  teams-dir: /data/agent-platform/teams
  root-dir: /data/agent-platform/root
  automations-dir: /data/agent-platform/automations
  kbase-dir: /data/agent-platform/kbase
  skills-market-dir: /data/agent-platform/skills-market
```

`AP_RUNTIME_REGISTRIES_DIR`、`AP_RUNTIME_CHATS_DIR`、`AP_RUNTIME_MEMORY_DIR`、`AP_RUNTIME_KBASE_DIR`、`AP_RUNTIME_PAN_DIR` 仍保留在 `.env` allowlist；其它低频 runtime 子目录请写入 `paths.*`，旧目录环境变量不再生效。

### Container Hub

`container-hub.base-url` 默认使用 allowlist 中的 `AP_CONTAINER_HUB_BASE_URL`：

```yaml
container-hub:
  base-url: ${AP_CONTAINER_HUB_BASE_URL:http://host.docker.internal:11960}
  auth-token:
  default-environment-id:
  request-timeout: 300
  default-sandbox-level: run
  agent-idle-timeout: 600
  destroy-queue-delay: 5
```

除 `AP_CONTAINER_HUB_BASE_URL` 外，旧 Container Hub 环境变量和高级 Container Hub 环境变量不再覆盖 runtime 配置。token、environment id、超时和 sandbox 策略请直接写入 YAML，或由部署系统在启动前渲染 YAML。

Container Hub 默认基础挂载最多为：

- `/workspace` -> `AP_RUNTIME_CHATS_DIR/<chatId>`（`rw`）
- `/root` -> `paths.root-dir`（`rw`）
- `/skills` -> `paths.agents-dir/<agentKey>/skills`（仅 `run/agent`，`global` 默认不挂载，`ro`）
- `/pan` -> `AP_RUNTIME_PAN_DIR`（`rw`）
- `/agent` -> `paths.agents-dir/<agentKey>`（`ro`，必挂载；缺失时 fail-fast）
- `/owner` -> `paths.owner-dir`（`ro`，缺失时自动创建）
- `/memory` -> `AP_RUNTIME_MEMORY_DIR/<agentKey>`（`ro`，缺失时自动创建）

`runtimeConfig.sandboxMounts` 会真实影响 Container Hub session payload：

- `platform + mode`：按需平台挂载；`agent` / `owner` / `memory` 只覆盖默认挂载模式；`skills-market` 显式挂到 `/skills-market`
- `destination + mode`：覆盖默认基础挂载模式
- `source + destination + mode`：新增非默认目标路径挂载

### Budget

通用预算写在 `configs/runtime.yml`：

```yaml
budget:
  timeout: 3600
  max-steps: 100
  model:
    max-calls: 100
    timeout: 60
    retry-count: 3
  tool:
    max-calls: 100
    timeout: 600
    retry-count: 0
  hitl:
    timeout: 600
    question:
      timeout: 600
    approval:
      timeout: 600
    form:
      timeout: 600
    plan:
      timeout: 600
```

配置项中的 HITL timeout `0` 表示未配置；事件协议中的 `awaiting.ask.timeout=0` 表示该等待项无限等待、不自动超时。`awaiting.ask.timeout>0` 由后端按真实时间独立计时，前端 observer / attach / detach 状态不会暂停或延长后端超时；`run.max-disconnected-wait` 不参与 HITL submit 等待。

Anthropic 请求的 `max_tokens` 默认值是源码内部常量 `4096`；stage 级 `maxOutputTokens` 仍可显式覆盖。

### Memory / Chat Storage

runtime memory 行为写入 `memory` 节：

```yaml
memory:
  enabled: true
  db-file-name: memory.db
  context-top-n: 5
  context-max-chars: 4000
  search-default-limit: 10
  hybrid-vector-weight: 0.7
  hybrid-fts-weight: 0.3
  dual-write-markdown: true
```

agent definition 侧另有 `memoryConfig`，用于决定某个 agent 是否注入 `memory_write`、`memory_read`、`memory_search` 以及管理类 memory tools。`memory/memory.md` 属于 agent 静态背景提示，不等同于 runtime memory store。chat 回放窗口固定为最近 20 个 run，SQLite 索引文件固定为 `chats.db`，不提供 runtime YAML 配置。

### KBASE

KBASE 是独立于 memory 的 workspace 知识库索引。runtime 负责新建 agent 的默认回答模型配置、refresh 调度和默认索引目录；具体 workspace、embedding model、include/exclude、chunk 与 retrieval 参数写在 `mode: KBASE` agent 的 `kbaseConfig` 中。KBASE 回答问题的大模型和思考程度最终仍写入 agent 顶层 `modelConfig.modelKey` 与 `modelConfig.reasoning.effort`。

```yaml
paths:
  kbase-dir: ./runtime/kbase

kbase:
  default-agent:
    modelKey: deepseek-v4-pro
    reasoningEffort: MEDIUM
  embedding:
    modelKey: babelark-text-embedding-v4
  refresh:
    debounce: 2s
    reconcile-interval: 10m
```

`default-agent.modelKey` 与 `default-agent.reasoningEffort` 只在通过 `/api/admin/agents/create` 创建 KBASE agent 且请求未显式传 `modelConfig.modelKey` / `modelConfig.reasoning.effort` 时写入 agent 配置；手写 agent 文件仍必须包含最终的 `modelConfig`。`kbase.embedding.modelKey` 会在新建 KBASE 且请求缺少完整 embedding 配置时写入 agent 配置，并在运行时解析到 `models/*.yml` 中 `type: embedding` 的 provider、真实 `modelId`、dimension、timeout 与 endpoint。旧式 `providerKey/model/dimension/timeout` 仍可读取作为 legacy fallback，但新配置推荐只写 `modelKey`。`kbaseConfig.embedding.*` 只用于知识库索引和检索，不控制回答模型或思考程度。默认 storage 使用 `${paths.kbase-dir}/${agentKey}/kbase.db` 与 `manifest.json`；agent 也可通过 `kbaseConfig.storage.location: workspace` 改用 `${runtimeConfig.workspaceRoot}/.kbase/`。KBASE 不读取 `memory` 或 `memoryConfig` 的 embedding 配置。

### Automation

Automation 策略写入 YAML：

```yaml
automation:
  enabled: true
  default-zone-id: Asia/Shanghai
  pool-size: 4
```

未在单个 automation YAML 中写 `environment.zoneId` 时，cron 会跟随这里的 `automation.default-zone-id`。只有跨时区任务或必须锁定不同业务时区的任务，才需要在 automation YAML 中显式写 `environment.zoneId`。

Logging 默认值已经源码化，不提供 runtime YAML 入口：request/auth/exception/tool/action/viewport/memory/LLM interaction 默认开启，SSE 默认关闭；LLM console 默认分类为 `request,usage`，mask 默认关闭，record 默认关闭。`AP_DEBUG_LLM_CONSOLE` 和 `AP_DEBUG_LLM_CHAT_RECORD` 是保留给现场排障的 allowlist 覆盖。

run lifecycle、WebSocket、H2A render、默认 locale、chat history 窗口和 logging 默认值都是内部常量，不提供 runtime YAML 配置入口。默认 locale 固定为英语 `en`，请求可通过 query/header/Accept-Language 显式覆盖。

## Tools / AI Tools / Prompts / Channels

工具运行时配置已合并到：

- `configs/tools.yml`：access policy、Bash、FileTools、sandbox bash
- `configs/ai-tools.yml`：`vision_recognize`、`web_fetch`、`image_generate` 等模型能力工具
- `configs/prompts.yml`：全局、plan-execute 与 memory prompt 文案
- `configs/coder-prompts.yml`：CODER 专属 prompt 文案
- `configs/kbase-prompts.yml`：KBASE 专属 prompt 文案
- `configs/channels.yml`：bridge / gateway channel 元数据与反向 WS 入口
- `configs/coder-settings.yml`：CODER 专属运行策略，例如是否读取 workspace `AGENTS.md`
- `configs/kbase-settings.yml`：KBASE 默认 agent、embedding、refresh 与文档抽取策略

旧拆分文件 `access-policy.yml`、`bash.yml`、`file-tools.yml`、`memory-prompts.yml`、`vision-recognize.yml`、`container-hub.yml`、`desktop.yml`、`cors.yml` 不再读取。

## Provider / Model Registry

provider 和 model registry 位于 `AP_RUNTIME_REGISTRIES_DIR/providers` 与 `AP_RUNTIME_REGISTRIES_DIR/models`。model registry 通过 `type` 区分调用契约，未写 `type` 时等价于 `type: chat`：

- provider `apiKey` 按明文字符串读取；未配置时保留为空值
- runtime 不再支持 provider `apiKey` 加密或解密
- provider `baseUrl` 只从 provider YAML 读取；provider 动态 base URL 环境变量不再覆盖
- 除 PROXY 与 ACP CODER 外，agent 最终必须配置 `modelConfig.modelKey`；运行时不会再通过 provider `defaultModel` 为 agent 隐式选择回答模型，管理接口新建 CODER / KBASE 时可分别用专用 settings 的 `default-agent.modelKey` 和 `default-agent.reasoningEffort` 代填模型配置。PROXY 的请求级 `model` 字段由上游代理服务解释，不要求存在于本地 model registry。
- KBASE embedding 解析优先级为 agent `embedding.modelKey` > agent 旧式 embedding 字段 > `configs/kbase-settings.yml -> embedding.modelKey` > settings 旧式 embedding 字段 > provider 顶层 `embedding.*`
- model 输入上下文字段为 `maxInputTokens`
- stage 输出上限字段为 `maxOutputTokens`
- 旧 `contextWindow` 与 `maxTokens` 配置字段不再读取

Embedding model 示例：

```yaml
key: babelark-text-embedding-v4
name: BabelArk Text Embedding v4
provider: babelark
type: embedding
modelId: text-embedding-v4
embedding:
  dimension: 1024
  timeout: 60
  endpointPath: /v1/embeddings
```

Image generation model 示例：

```yaml
key: gpt-image-1
name: GPT Image 1
provider: openai
type: image-generation
modelId: gpt-image-1
image:
  endpointPath: /v1/images/generations
  timeout: 120
  defaultSize: 1024x1024
  responseFormats:
    - b64_json
    - url
```

## Compose 目录映射

`compose.yml` 会把宿主机 runtime 根目录映射到容器内固定路径，并允许四个高频子目录单独覆盖：

| 宿主机来源 | 容器内路径 |
|---|---|
| `${AP_RUNTIME_DIR:-./runtime}` | `/opt/runtime` |
| `${AP_RUNTIME_REGISTRIES_DIR:-${AP_RUNTIME_DIR:-./runtime}/registries}` | `/opt/runtime/registries` |
| `${AP_RUNTIME_CHATS_DIR:-${AP_RUNTIME_DIR:-./runtime}/chats}` | `/opt/runtime/chats` |
| `${AP_RUNTIME_MEMORY_DIR:-${AP_RUNTIME_DIR:-./runtime}/memory}` | `/opt/runtime/memory` |
| `${AP_RUNTIME_PAN_DIR:-${AP_RUNTIME_DIR:-./runtime}/pan}` | `/opt/runtime/pan` |

容器内应用会设置 `AP_RUNTIME_DIR=/opt/runtime`，并把 runtime 子目录同步为对应 `/opt/runtime/*` 路径；容器内应用监听端口固定为 `8080`。
