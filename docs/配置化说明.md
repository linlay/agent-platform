# 配置化说明

当前 Go runtime 的配置事实源主要来自两部分：

- 代码默认值：`internal/config/config.go`
- 外部输入：环境变量与 `configs/*.yml`

`.env.example` 现在只保留推荐给普通部署者的最终用户配置入口。本页则保留完整的高级、排障和低频配置说明；工具运行时配置已统一迁移到 `configs/tools.yml`。

## 标签说明

| 标签 | 含义 |
|---|---|
| `End user` | 普通部署者通常会直接理解、填写或调整的变量 |
| `Advanced / operator` | 运维、高级部署或容量调优场景才需要改动 |
| `Debug / troubleshooting` | 主要用于排障、链路观测和临时诊断 |
| `Internal / compatibility` | 偏实现细节或兼容用途，不建议作为公开产品入口 |
| `Wired but not recommended for public use` | 代码会读取，但当前 Go runtime 并未形成稳定、清晰的终端用户能力，不建议公开暴露 |

## 推荐放进 `.env` 的最终用户变量

### Server

| 环境变量 | 默认值 | 标签 | 说明 |
|---|---|---|---|
| `SERVER_PORT` | `11949` | `End user` | 本地 `make run` 与 `compose.yml` 的外部入口端口 |

本地运行约定：

- `make run` 会先加载根目录 `.env`
- `make run` 的端口链路为 `SERVER_PORT -> 11949`
- 直接执行 `go run ./cmd/agent-platform` 不会自动加载 `.env`
- 直接 `go run` 时，应用端口链路仍然是 `SERVER_PORT -> 11949`

### Auth / Ticket

| 环境变量 | 默认值 | 标签 | 说明 |
|---|---|---|---|
| `AUTH_ENABLED` | `true` | `End user` | 是否开启 `/api/**` JWT 鉴权 |
| `AUTH_LOCAL_PUBLIC_KEY_FILE` | `configs/local-public-key.pem` | `Advanced / operator` | 本地公钥模式下的公钥文件路径 |
| `AUTH_JWKS_URI` | 空 | `End user` | JWKS 模式的公钥地址 |
| `AUTH_ISSUER` | 空 | `End user` | JWT issuer 校验值 |
| `AUTH_JWKS_CACHE_SECONDS` | `0` | `Advanced / operator` | JWKS 缓存秒数，仅在 JWKS 模式下生效 |
| `CHAT_RESOURCE_TICKET_SECRET` | 空 | `End user` | resource ticket 签名密钥；设置后启用 `/api/resource` ticket 校验 |
| `CHAT_RESOURCE_TICKET_TTL_SECONDS` | `86400` | `Advanced / operator` | resource ticket 过期时间；未设置 secret 时不生效 |

本地 JWT 公钥规则：

- 默认值是 `configs/local-public-key.pem`
- `AUTH_LOCAL_PUBLIC_KEY_FILE` 若是绝对路径，则原样使用
- 若是相对路径，则按项目根目录解析
- 若为了兼容仍写成单文件名 `local-public-key.pem`，会自动解析到 `configs/local-public-key.pem`

### Container Hub

| 环境变量 | 默认值 | 标签 | 说明 |
|---|---|---|---|
| `CONTAINER_HUB_BASE_URL` | 空 | `End user` | Container Hub 服务地址；非空即启用 Container Hub |
| `CONTAINER_HUB_AUTH_TOKEN` | 空 | `Advanced / operator` | Bearer token；运行时支持但不展示在 `.env.example` |
| `CONTAINER_HUB_DEFAULT_ENVIRONMENT_ID` | 空 | `End user` | 默认 environment id |
| `CONTAINER_HUB_REQUEST_TIMEOUT` | `300` | `Advanced / operator` | 请求超时，单位秒 |
| `CONTAINER_HUB_DEFAULT_SANDBOX_LEVEL` | `run` | `Advanced / operator` | 默认沙箱级别 |
| `CONTAINER_HUB_AGENT_IDLE_TIMEOUT` | `600` | `Advanced / operator` | agent 级沙箱闲置回收时间，单位秒 |
| `CONTAINER_HUB_DESTROY_QUEUE_DELAY` | `5` | `Advanced / operator` | 销毁队列延迟，单位秒 |

Container Hub 默认基础挂载最多为：

- `/workspace` -> `CHATS_DIR/<chatId>`（`rw`）
- `/root` -> `ROOT_DIR`（`rw`）
- `/skills` -> `AGENTS_DIR/<agentKey>/skills`（仅 `run/agent`，`global` 默认不挂载），`ro`
- `/pan` -> `PAN_DIR`（`rw`）
- `/agent` -> `AGENTS_DIR/<agentKey>`（`ro`，必挂载；缺失时 fail-fast）
- `/owner` -> `OWNER_DIR`（`ro`，缺失时自动创建）
- `/memory` -> `MEMORY_DIR/<agentKey>`（`ro`，缺失时自动创建）

`runtimeConfig.sandboxMounts` 会真实影响 Container Hub session payload；旧名 `extraMounts` 仍兼容读取：

- `platform + mode`：按需平台挂载；`agent` / `owner` / `memory` 只覆盖默认挂载模式；`skills-market` 显式挂到 `/skills-market`
- `destination + mode`：覆盖默认基础挂载模式
- `source + destination + mode`：新增非默认目标路径挂载

`configs/runtime.example.yml` 的 `budget.hitl` 节展示全局 HITL/awaiting timeout；智能体级使用 `agent.yml` 的同名 `budget.hitl` 覆盖。`container-hub` 节只默认展开 `base-url`；其它字段已有代码默认值，模板中以注释形式保留。`auth-token` 不在模板中展示，但 `CONTAINER_HUB_AUTH_TOKEN` / `container-hub.auth-token` 仍可作为客户端 Bearer token 对接 `agent-container-hub` 的 `AUTH_TOKEN`。

  model:
    timeout: 120
  tool:
    timeout: 120

    timeout: 600

      timeout: 600

      timeout: 600

      timeout: 600

      timeout: 600

### Bash 工具

Bash 工具从 `configs/tools.yml` 的 `bash` 节读取命令白名单、超时和可选 host shell 覆盖；目录权限统一从同一文件的 `access-policy` 节读取。旧 `configs/bash.yml` 不再读取。参考模板是 `configs/tools.example.yml`。

| YAML 字段 | 默认值 | 标签 | 说明 |
|---|---|---|---|
| `allowed-commands` | `ls,pwd,cat,head,tail,top,free,df,git,rg,find,...` | `Advanced / operator` | 允许执行的命令白名单 |
| `shell-timeout` | `10` | `Advanced / operator` | 单条 shell 命令默认超时时间，单位秒 |
| `max-command-chars` | `16000` | `Advanced / operator` | 单条命令字符串最大长度 |
| `working-directory` | `.` | `Advanced / operator` | 可选；未传 `cwd` 时的默认执行目录，不控制路径权限 |
| `shell-features-enabled` | `true` | `Advanced / operator` | 可选；`true` 允许 `&&`、管道、重定向等 shell 语法，`false` 只允许简单白名单命令 |
| `shell-executable` | 平台默认 | `Advanced / operator` | 可选；host shell 可执行文件。Unix/macOS 默认 `bash`，Windows 默认 `powershell.exe` |
| `shell-args` | 平台默认 | `Advanced / operator` | 可选；host shell 参数模板。Unix/macOS 默认 `-lc "{{command}}"`，Windows 默认 PowerShell 参数 |

`bash HITL` 不再有单独的 `.env` 开关。是否拦截 bash 命令完全由当前 agent 挂载的 skills 决定：规则来自 skill 目录下的 `.bash-hooks/`，未挂载任何 bash hooks 时不会触发拦截。旧字段 `bash.hitl-default-timeout-ms` 已删除；请迁移到 `budget.hitl.timeout`，具体 approval/form 可继续在 bash-hook subcommand 或 form/tool 定义上配置 `timeout`（秒）。

旧字段 `allowed-paths`、`path-checked-commands`、`path-check-bypass-commands` 已迁移到 `configs/tools.yml` 的 `access-policy`；启动时仍出现在 `bash` 节或旧 `bash.yml` 中会直接报错。

### Access Policy 目录权限

`configs/tools.yml` 的 `access-policy` 是宿主机侧工具目录权限的最终事实源，`bash` 与 FileTools 共用。旧 `configs/access-policy.yml` 不再读取。请求可传 `accessLevel`，合法值为 `default`、`auto_approve`、`full_access`，缺省为 `default`。

| YAML 字段 | 默认值 | 标签 | 说明 |
|---|---|---|---|
| `working-directory` | `@workspace` | `Advanced / operator` | 目录策略解析相对路径时的默认基准 |
| `levels.default.read-roots` | `@workspace,@chat,@agent,@skills` | `Advanced / operator` | 默认免审读取根 |
| `levels.default.write-roots` | `@workspace,@chat` | `Advanced / operator` | 默认免审写入根 |
| `levels.default.readonly-roots` | `@agent,@skills,@skills-market` | `Advanced / operator` | 即使被写根覆盖也保持只读的根 |
| `levels.default.approvals.read-outside-roots` | `hitl` | `Advanced / operator` | `file_read` / `file_glob` / `file_grep` 或 bash 读取越界路径的处理 |
| `levels.default.approvals.write-outside-roots` | `hitl` | `Advanced / operator` | `file_write` / `file_edit` 或 bash 写越界路径的处理 |
| `levels.default.approvals.bash-complex-filesystem` | `hitl` | `Advanced / operator` | 复杂 shell 无法静态证明文件访问时的处理 |
| `levels.default.approvals.bash-opaque-command` | `hitl` | `Advanced / operator` | `go`、`npm`、`node`、`python`、`make` 等不透明命令的处理 |
| `levels.default.approvals.bash-write-in-write-roots` | `allow` | `Advanced / operator` | bash 写入已允许写根时是否还需要审批 |

approval action 支持 `allow`、`hitl`、`auto`、`block`。`auto_approve` 默认继承 `default` 边界，越界读、复杂 bash 与不透明命令自动批准，越界写仍触发 HITL；自动批准会写入当前 `chatId.jsonl` 的 `messages[]` 内联 `role=user` 审计消息，消息内的 `approval.decisions[].decision` 为 `auto_approved`。`full_access` 默认把 host path 读写根设为 `/`，不产生越界审批记录，但仍保留 bashsec hard block。

### Sandbox Bash 安全覆盖

`sandbox-bash` 是 `configs/tools.yml` 的顶层配置，只对声明了 `runtimeConfig.environmentId` 的沙箱 bash 生效；host bash 不读取该配置。它用于表达 sandbox bash 相对 host bashsec 的差异，不改变 `access-policy` 的 host path 语义，也不引入 host/sandbox 双域 access-policy。

| YAML 字段 | 默认值 | 标签 | 说明 |
|---|---|---|---|
| `security.bashsec-overrides.output-redirection` | 未配置 | `Advanced / operator` | sandbox bash 输出重定向触发 `bashsec:redirections` 时的处理；支持 `allow`、`hitl`、`auto`、`block` |
| `security.bashsec-overrides.heredoc-output-redirection` | 未配置 | `Advanced / operator` | sandbox bash heredoc + 输出重定向触发 `bashsec:redirections` 时的处理；支持 `allow`、`hitl`、`auto`、`block` |
| `security.audit-auto-approvals` | `false` | `Advanced / operator` | 为 `auto` 覆盖写入 `auto_approved` 审计记录 |

### File Read / Write 文件工具

`file_read` / `file_glob` / `file_grep` / `file_write` / `file_edit` 是 Claude Code 风格的窄文件工具，行为配置从 `configs/tools.yml` 的 `file-tools` 节读取，目录权限从同一文件的 `access-policy` 节读取；参考模板是 `configs/tools.example.yml`。旧 `configs/file-tools.yml` 不再读取。它们用于替代 heredoc、重定向等容易触发 bash 安全拦截的文件读写场景，不改变 bash 安全策略。

`file_glob` 按文件名/路径 glob 返回匹配路径；`file_grep` 按文件内容正则搜索，`files_with_matches` 只返回内容命中的文件路径，不等同于纯文件名搜索。

`file_write` / `file_edit` 成功结果会包含 `lineStats`，字段为 `addedLines`、`deletedLines`、`editedLines`。统计基于写入前后完整文件内容的行级 diff；新建文件等价于从空内容写入新内容。

| YAML 字段 | 默认值 | 标签 | 说明 |
|---|---|---|---|
| `working-directory` | `.` | `Advanced / operator` | `file_read` / `file_glob` / `file_grep` / `file_write` / `file_edit` 解析相对路径时使用的工作目录；路径权限仍以 `tools.access-policy` 为准 |
| `max-read-bytes` | `1048576` | `Advanced / operator` | 单次 `file_read` 最多返回的文本文件字节数 |
| `max-write-bytes` | `1048576` | `Advanced / operator` | 单次 `file_write` 写入内容或 `file_edit` 编辑后的内容最多写入的字节数 |
| `max-batch-ops` | `20` | `Internal / compatibility` | 预留批量文件工具上限；当前文件工具仍按单次调用执行 |
| `require-write-approval` | `true` | `Advanced / operator` | 是否要求 `file_write` / `file_edit` 先走人工审批 |
| `require-read-before-write` | `true` | `Advanced / operator` | 已有文件写入或编辑前是否必须先被文件工具完整读取（fully read） |
| `read-before-write-scope` | `run` | `Advanced / operator` | `run` 只接受当前 run 的 `file_read` / 写后快照；`chat` 允许同 chat 跨 run 复用 `<chatId>/.tools/state/file-versions.json` 中的版本证明 |

默认 `accessLevel=default` 下，workspace、chat、agent、当前 agent skills 读免审，workspace 和 chat 写免审；`@chat` 解析为当前 `CHATS_DIR/<chatId>`，不依赖 `@workspace == @chat` 才可写。除此之外 read/glob/grep/write/edit 都按 `tools.access-policy` 触发审批或自动处理。旧字段 `allowed-read-paths` / `allowed-write-paths` 已移除，仍出现在 `configs/tools.yml` 的 `file-tools` 节或旧 `configs/file-tools.yml` 会导致启动失败。

### Vision Recognize 工具

`vision_recognize` 是全局配置驱动的 backend tool，用于让非 vision 主模型通过工具调用识别图片。它不会默认注入；只有 agent 在 `toolConfig.tools` 显式声明 `vision_recognize` 时才会暴露。

配置从 `configs/ai-tools.yml` 的 `vision-recognize` 节读取，参考模板是 `configs/ai-tools.example.yml`。旧 `configs/vision-recognize.yml` 不再读取。

| YAML 字段 | 默认值 | 标签 | 说明 |
|---|---|---|---|
| `enabled` | `false` | `Advanced / operator` | 是否允许调用 `vision_recognize` |
| `default-profile` | `general` | `Advanced / operator` | 未传 `profile` 时使用的 profile |
| `profiles.<name>.model-key` | 空 | `Advanced / operator` | `REGISTRIES_DIR/models` 中 `isVision: true` 的模型 key |
| `profiles.<name>.timeout` | `60` | `Advanced / operator` | 单次识别请求超时，单位秒 |
| `profiles.<name>.max-images` | `4` | `Advanced / operator` | 单次最多识别图片数 |
| `profiles.<name>.max-image-bytes` | `20971520` | `Advanced / operator` | 单张图片最大字节数 |
| `profiles.<name>.output-format` | `text` | `Advanced / operator` | 默认返回格式，支持 `text` / `json` |
| `profiles.<name>.system-prompt` | 内置通用提示 | `Advanced / operator` | 识别模型的 system prompt |

图片来源支持当前 chat 附件 `reference_name`，以及受 `tools.access-policy` 读权限约束的本地 `file_path`。v1 不支持远程 URL。

### Web Fetch 工具

`web_fetch` 是全局配置驱动的 backend tool，用于抓取公开 HTTP(S) URL，将 HTML 转成 Markdown 风格文本，并按调用参数中的 `prompt` 交给小模型抽取或总结。它不会默认注入；只有 agent 在 `toolConfig.tools` 显式声明 `web_fetch` 时才会暴露。执行入口也兼容 `WebFetch` 旧名，但工具目录只暴露 `web_fetch`。

配置从 `configs/ai-tools.yml` 的 `web-fetch` 节读取，参考模板是 `configs/ai-tools.example.yml`。网络抓取本身是工程化 HTTP GET；除预批准 host 的短 Markdown 直返外，结果处理会调用当前 profile 的 `model-key`。

| YAML 字段 | 默认值 | 标签 | 说明 |
|---|---|---|---|
| `enabled` | `false` | `Advanced / operator` | 是否允许调用 `web_fetch` |
| `default-profile` | `general` | `Advanced / operator` | 未传 `profile` 时使用的 profile |

### Memory 行为调优

以下变量会真实影响当前 Go runtime 的 memory 召回行为，但仍属于高级调优项，不建议默认暴露给最终用户：

| 环境变量 | 默认值 | 标签 | 说明 |
|---|---|---|---|
| `AGENT_MEMORY_DB_FILE_NAME` | `memory.db` | `Advanced / operator` | memory SQLite 文件名 |

agent definition 侧另有 `memoryConfig`：

- 未配置 `memoryConfig.enabled` 时，不启用 runtime memory，也不注入基础 memory tools
- `memoryConfig.enabled: true` 会注入基础 memory tools：`memory_write`、`memory_read`、`memory_search`
- `memoryConfig.managementTools: true` 会额外注入管理类 tools：`memory_update`、`memory_forget`、`memory_timeline`、`memory_promote`、`memory_consolidate`

- `memoryConfig.embedding.providerKey` 指向 `REGISTRIES_DIR/providers/*.yml` 中的 provider key；embedding 模型、维度和超时优先来自 provider 的 `memory.embedding`，也可在 agent 侧用 `memoryConfig.embedding.timeout` 覆盖，单位秒
- provider 的 `memory.embedding.timeout` 可选，默认 `15`，单位秒
- `memory/memory.md` 属于 agent 静态背景提示，不受这些环境变量控制，也不等同于 runtime memory store
- `Learn(...)` 成功后会自动执行一轮轻量 observation consolidate；显式 `memory_consolidate` 仍用于人工触发完整整理

### WebSocket

WebSocket 控制面常开，`/ws` 会随服务启动一起注册；普通 HTTP/SSE `/api/query` 继续支持。以下 `AGENT_WS_*` 仅用于深度调优，仍只保留在本参考文档中。

| 环境变量 | 默认值 | 标签 | 说明 |
|---|---|---|---|
| `AGENT_WS_MAX_MESSAGE_SIZE` | `1048576` | `Advanced / operator` | 单条消息最大字节数 |
| `AGENT_WS_PING_INTERVAL` | `30` | `Advanced / operator` | ping 心跳间隔，单位秒 |
| `AGENT_WS_WRITE_TIMEOUT` | `15` | `Advanced / operator` | 写超时，单位秒 |
| `AGENT_WS_WRITE_QUEUE_SIZE` | `256` | `Advanced / operator` | 每连接写队列大小 |
| `AGENT_WS_MAX_OBSERVES_PER_CONN` | `8` | `Advanced / operator` | 每连接最大 observe 数 |

说明：

- 默认会注册 `/ws`，不再提供环境变量关闭开关
- WebSocket handler 会复用当前 catalog、chat、query、run stream 等接口能力
- 鉴权开启时，WebSocket 也会走相同的 token 校验链路
- WebSocket 是控制面：普通上传/下载仍走 `POST /api/upload` 与 `GET /api/resource`；当前 `/ws` 上的 `/api/upload` 只接受网关发送的 `url + metadata`，platform 会通过 HTTP 旁路下载文件，不支持把文件字节直接塞进 WS JSON frame

### HTTP Polling Monitor

运行态监控通过 HTTP 轮询获取快照，不新增实时订阅通道，也不复用 `/ws` 作为监控入口。首版接口为：

- `GET /api/monitor`：总览与 WS 摘要，默认包含最近 5 条 WS 消息
- `GET /api/monitor/ws/connections`：当前/最近 WS 连接列表，支持 `limit`、`sessionId`、`source` 与 `deviceId`
- `GET /api/monitor/ws/messages`：最近 WS 消息列表，支持 `limit`、`sessionId`、`source` 与 `deviceId`

说明：

- 监控数据只保存在进程内，服务重启后清空
- 接口沿用 `/api/*` 鉴权；Auth 开启时需要 Bearer token
- WS 客户端可在握手 query 自报 `source` 与 `deviceId`，例如 `/ws?source=webclient&deviceId=device-123`；`source` 不做硬枚举，只作为排障标签
- 消息只保留元信息和最多 512 字符的脱敏 `payloadPreview`，不会保存完整 payload
- `push.heartbeat`、ping/pong/control frame 不进入最近消息列表，避免轮询结果被心跳刷屏

### Reverse WebSocket Gateway

反向 WebSocket 用于让 `agent-platform` 主动连出到一个上游智能体网关；连接建立后，网关会像普通 `/ws` client 一样发送 `request` 帧，`agent-platform` 继续返回 `response` / `stream`，并复用当前 `broadcast()` 推送 `push` 事件。Gateway 连接不再通过单个 env 配置，统一写入 `configs/channels.yml`。

说明：

- 反向连接随 WebSocket 控制面常开；具体连接条目由 `configs/channels.yml` 控制
- 握手只使用 `Authorization` header 传递 Bearer token，不会把凭据放进 URL query
- 建连成功后，网关端会先收到一条 `push.connected`
- 断线重连期间的 `broadcast` 为有损投递，当前不提供离线缓冲
- `AGENT_WS_WRITE_TIMEOUT`、`AGENT_WS_WRITE_QUEUE_SIZE`、`AGENT_WS_MAX_MESSAGE_SIZE`、`AGENT_WS_MAX_OBSERVES_PER_CONN` 这些现有 `AGENT_WS_*` 参数同样适用于反向连接

### Channel 配置

多渠道部署优先使用 `configs/channels.yml`。这个文件只描述 platform 侧关心的 channel 元数据、反向 WS 入口和 agent 准入，不存放 bridge 的外部平台凭证，也不存放 gateway 的客户端认证策略。

字段约定：

- `channels.<id>`：channel ID，同时也是 `chatId` 前缀路由键，例如 `wecom#user123` 的 channel 是 `wecom`
- `name`：展示名称
- `type`：`bridge` 或 `gateway`
- `default-agent`：仅在请求最终回退到全局默认 agent 时，才会被这个 channel 默认值覆盖
- `agents`：`"*"` 表示允许全部 agent；数组表示白名单
- `gateway.url`：platform 主动连出的完整反向 WS 地址
- `gateway.jwt-token`：用于反向 WS 握手和旁路 HTTP 请求的 Bearer token
- `gateway.base-url`：可选；未配置时从 `gateway.url` 自动派生
- `gateway.handshake-timeout`、`gateway.reconnect-min`、`gateway.reconnect-max`：可选，单位秒；未配置时默认分别为 `10`、`1`、`30`

行为说明：

- `configs/channels.yml` 缺失时不会生成 gateway 连接
- `channels.yml` 中的 gateway entry 会在启动时合成为 `config.Gateways`
- `configs/channels.example.yml` 提供了 bridge 与 gateway 两种 channel 形态示例
- `channels.yml` 只在进程启动时读取；文件变更后必须重启 runtime 才会生效

## Agent Context Tags

`context tags` 不是全局默认集合，而是每个 agent 只从 `contextConfig.tags` 读取：

```yaml
contextConfig:
  tags:
    - system
    - session
```

当前支持标签：

- `system`
- `session`
- `owner`
- `all-agents`

说明：

- `session` 负责暴露运行上下文
- `owner` 负责注入 `OWNER_DIR` 下的 markdown 内容；`mode: CODER` 不会默认启用，需要显式声明
- `sandbox` 不再属于 `context tags`；只要 agent 配置了 `runtimeConfig.environmentId`，运行时就会自动注入 sandbox context
- runtime memory context 不再属于 `context tags`；只有 agent 显式开启 `memoryConfig.enabled: true` 时，运行时才会注入 memory context

## Agent Prompt Files

目录式 agent 的 prompt 文件约定：

- 普通模式默认读取 `AGENTS.md`
- 顶层 `promptFile` 可显式覆盖普通模式 prompt
- `PLAN_EXECUTE` 默认读取 `AGENTS.plan.md`、`AGENTS.execute.md`、`AGENTS.summary.md`
