# 智能体配置说明

本文件说明 agent / team / skill 的目录化配置、prompt 文件、CODER、KBASE、memoryConfig 与 runtimeConfig。当前 Go runtime 已支持从 `configs/runtime.yml -> paths.agents-dir` 读取 `runtime/agents/*.yml` 与 `runtime/agents/<key>/agent.yml`，并将结果直接暴露给 `/api/agents` 与 query agent 选择逻辑。

## 当前状态

当前 Go 版运行时：

- `models` 会从 `AP_RUNTIME_REGISTRIES_DIR/models` 读取
- provider 连接配置会从 `AP_RUNTIME_REGISTRIES_DIR/providers` 读取
- catalog 中的 agent/team/skill/tool 已改为目录驱动
- `runtime/agents/*.yml` 与 `runtime/agents/<key>/agent.yml` 都会参与装载
- Team 只接受 `runtime/teams/<teamId>/team.yml`；平铺 `runtime/teams/*.yml|yaml` 会使启动失败

因此，这份文档当前主要承担两件事：

1. 说明与 Java 对齐后的 Agent Definition 形态
2. 标注当前 Go 版已接入与仍待补齐的细节

## 目标形态

参考仓库中的完整 Agent Definition 目标形态如下：

```yaml
key: agent_key
name: agent_name
role: 角色标签
description: 描述
icon: "emoji:🤖"
greetings:
  - 我可以帮你总结材料、拆解问题，并给出下一步行动建议。
  - 需要快速进入状态时，我会先帮你找最值得处理的切入点。
wonders:
  - 帮我总结这段材料
  - 给我 5 个可以马上执行的下一步
modelConfig:
  modelKey: qwen3-max
toolConfig:
  tools: ["bash", "datetime"]
mode: REACT
plain:
  systemPrompt: |
    系统提示词
```

长期看，Go 版也会沿用这一思路：

- 前 4 行固定 `key`、`name`、`role`、`description`
- `modelConfig.modelKey` 引用 `registries/models/*.yml`；除 PROXY 与 ACP CODER 外，所有 agent 都必须显式声明
- `toolConfig` 只声明 `tools` 工具名列表；不支持旧式 `toolConfig.overrides/backends/frontends/actions`
- `mode` 决定 agent 执行模式

## Model Sampling

agent 可在 `modelConfig.sampling` 声明默认采样参数；`PLAN-EXECUTE` 可用 `stageSettings.plan/execute/summary.modelConfig.sampling`，CODER planning 可用 `stageSettings.planning/execute.modelConfig.sampling` 按字段覆盖：

```yaml
modelConfig:
  modelKey: qwen3-max
  sampling:
    temperature: 0.7
    topP: 0.9
    presencePenalty: 0
    frequencyPenalty: 0
    seed: 42

stageSettings:
  execute:
    modelConfig:
      sampling:
        temperature: 0.4
```

继承优先级为 `stageSettings.<stage>.modelConfig.sampling` > `modelConfig.sampling` > 运行时默认。未配置时，OpenAI-compatible 主链路仍发送 `temperature: 0`，保持确定性默认行为。

字段使用 camelCase：`temperature`、`topP`、`presencePenalty`、`frequencyPenalty`、`seed`。OpenAI-compatible 协议会映射为 `temperature`、`top_p`、`presence_penalty`、`frequency_penalty`、`seed`；Anthropic 协议只发送 `temperature` 和 `top_p`。

## Agent Identity 与 SOUL.md 边界

`agent.yml` 是 agent 身份与能力元数据的唯一事实源：

- `key`
- `name`
- `role`
- `description`
- `greetings`
- `wonders`
- `mode`
- `toolConfig`
- `skillConfig`
- `runtimeConfig`

运行时会先根据 `agent.yml` 生成统一的 `Agent Identity` prompt section，再拼接其他 prompt 层。

`SOUL.md` 的职责是长期行为提示，不是配置副本：

- 允许写人格、协作方式、风险姿态、硬边界、非目标
- 允许写长期稳定的行为约束
- 不应重复 `key/name/role/description/mode`
- 不应复制 tools、skills、sandbox、环境路径或一次性任务说明

推荐结构：

```md
# Soul

## Persona

## Boundaries

## Working Style

## Long-Term Notes
```

## Greetings 与 Wonders

`greetings` 与 `wonders` 都是 UI 展示元数据，不参与 agent identity prompt 或系统提示词。

`greetings` 用来配置 agent 的开场/占位介绍，语义是“告诉人类这个智能体能做什么”。它适合展示在聊天输入框 placeholder 或空状态里，例如：

```yaml
greetings:
  - 我可以帮你练习词汇、复盘错题，也能把今天的学习拆成小步骤。
  - 你可以把单词表发给我，我会帮你设计一轮轻量测验。
```

支持配置多条，运行时会过滤空字符串，并通过 `/api/agents` 与 `/api/agent` 原样暴露给客户端；需要随机展示时，前端可以从数组中随机选一条。

单条问候语写法：

```yaml
greeting: 我可以帮你快速看懂这个项目，并给出下一步行动建议。
```

`wonders` 保持原语义：具体的 query 问题参考，也就是用户可以直接点选或提交的示例问题。

```yaml
wonders:
  - Quiz me on vocabulary
  - 帮我检查当前工作区的变更
  - |-
    帮我制定一个三步学习计划
    并给出今天可以开始的第一步
```

## Agent 契约与 mode 关系

Go 实现不使用类继承。`internal/agent` 定义 mode descriptor、profile、capabilities、system-init spec 与公共 prompt 渲染契约；`internal/agent/coder`、`internal/agent/kbase`、`internal/agent/team` 分别持有自身 mode 规则，`internal/agent/builtin` 只做静态分派。KBASE 的索引与检索运行时已下沉到独立的 `internal/kbase` 公共能力：

```text
internal/agent
├── coder                 # CODER 具体 mode
├── kbase                 # KBASE 专用 mode 的 profile/prompt/boundary/create
├── team                  # TEAM 内部 mode 与集中调度规则
└── builtin               # 静态分派，不是插件生命周期

internal/kbase            # mode 中立的 config/source/index/watch/retrieval/tools
app/kbase_adapter.go      # catalog 快照 -> kbase.AgentSpec
catalog.TeamSnapshot      # run-scoped Team/成员/协调器配置快照
```

CODER、KBASE 和 TEAM 都是 `internal/agent` 契约下的具体 mode，但 KBASE capability 不依赖 mode 或 catalog；其 `AgentSpec` 只包含 `Key`、`Enabled`、`SourceRoot`、`Requirement` 与公共配置。TEAM 只服务于目录式 Team：运行时从 `TeamSnapshot` 合成隐藏协调器，不把它注册成普通 `AgentDefinition`。普通 Agent YAML 声明 `mode: TEAM` 会校验失败，管理接口也不能创建 TEAM agent。

## Team 类型与配置

Team 配置事实源是 `runtime/teams`。只接受目录式定义；平铺 `runtime/teams/<teamId>.yml|yaml`、`defaultAgentKey` 和 `runtimeMode: legacy` 均已移除，发现即启动失败。

### Team

目录式 Team 使用以下结构：

```text
runtime/teams/research/
  team.yml
  SOUL.md       # 可选：Team 人格
  AGENTS.md     # 可选：Team 工作规则
```

`teamId` 来自目录名。最小 `team.yml`：

```yaml
name: Research Team
description: 研究、写作与复核
agentKeys:
  - researcher
  - writer
  - reviewer
orchestrator:
  modelConfig:
    modelKey: qwen3-max
    reasoning:
      enabled: true
      effort: HIGH
    sampling:
      temperature: 0.2
  budget:
    maxSteps: 40
    tool:
      maxCalls: 20
  maxParallel: 3
```

- `orchestrator.modelConfig.modelKey` 必填；`defaultAgentKey` 禁止出现。
- `maxParallel` 默认为 5，只接受 1–5 的整数；它只限制同时执行的成员数。一次 `agent_delegate` 可以列出整个冻结 roster，超出并发数时由 runtime 分波执行。
- `SOUL.md`、`AGENTS.md` 会进入协调器 system prompt，但不能覆盖 `internal/agent/team` 的硬编码路由、安全和隐私规则。
- 目录式 Team 的成员列表为空或任一成员失效时，query/automation 返回不可用，不回退到全局默认 Agent。

隐藏总控每个用户回合统一调用 `agent_delegate`：一项和多项使用相同的 `tasks[]` 契约；`agentKey` 来自本次 run 冻结 roster，`task` 可省略并默认使用原始用户消息，`taskName/files` 可选。同一次调用中成员不可重复，`tasks.maxItems` 等于 roster 大小。成员继承原始 role、scene、references 和专用 Team 历史，附加 files 会转换并去重。

复杂任务由总控先用 `plan_add_tasks` 创建任务，再用 `plan_get_tasks`、`plan_update_task` 维护状态，并按可运行任务分轮调用 `agent_delegate`。串行和分支关系不进入委派工具本身。每个用户回合至少成功发起一次委派后才能生成最终答案；所有成员结果都回注总控，根回答只由总控输出。

`agent_delegate` 是 embedded、session-local、internal-only 工具：Team 自动注入，普通 Agent 配置会校验失败，`/api/admin/tools` 不返回它，tool/reasoning 事件也不下发客户端。成员输出复用 task/content 事件，并携带 `actor`、`teamId`、成员 `agentKey` 和 `presentation:"task"`。

### 共同边界

run 启动时 catalog 会原子解析 Team、有效/失效成员、成员 `AgentDefinition`、协调器配置与 prompt，生成本次 run 使用的不可变快照。

- `teamId` 在 chat 创建后固定；已有无 Team chat 也不能补绑 Team。
- Team 请求只传 `teamId`，传入 `agentKey` 会被视为绕过协调器并返回 400。
- Team 的公开 run owner 是 `teamId`；协调器执行 key 不出现在普通 Agent catalog、常规 run 事件、chat summary 或控制接口中。
- Team 成员不能再调用 `agent_invoke`，也不能把另一个 Team 作为成员或子任务目标；ACP CODER 当前不能作为 child。
- Automation、HTTP、WebSocket、submit continuation 与子智能体准入使用同一 Team resolver。历史 chat、archive、restore 与 derive 保留原始身份字段，但历史记录不再恢复旧 Team 路由。
- 运行中的 Team 快照不受 catalog 热重载影响；下一次 run 才读取新成员、prompt 与协调器配置。

## 当前仍有边界

- TEAM 不提供委派级 DAG DSL；计划状态和后续总控轮次表达串行、分支与重试。
- `agent_delegate` 的并行 HITL 按并发波次合并为 Team-level form。
- ACP CODER 不能作为 Team/`agent_invoke` child；普通 Agent 管理接口也不提供 TEAM 创建或编辑入口。

## Context Tags

当前 Go runtime 不会为所有 agent 自动附加一组全局默认 `context tags`；每个 agent 只从 `contextConfig.tags` 读取声明：

```yaml
contextConfig:
  tags:
    - system
    - session
```

支持标签：

- `system`
- `session`
- `owner`
- `agents`

其中：

- `session` 会暴露运行时上下文
- `owner` 会注入 `configs/runtime.yml -> paths.owner-dir` 下的 markdown 内容；`mode: CODER` 不会默认启用，需要显式声明
- `agents` 会注入平台已注册 agent 的摘要上下文；只写 `tags: [agents]` 表示全部 agent，也可用 `contextConfig.agents` 指定部分 agent key
- `sandbox` 不再通过 `context tags` 控制；只要 agent 声明了 `runtimeConfig.environmentId`，运行时会自动注入 sandbox context
- runtime memory context 不再通过 `context tags` 控制；只有 agent 显式开启 `memoryConfig.enabled: true` 时，运行时才会自动注入 memory context

注入所有 agent 摘要：

```yaml
contextConfig:
  tags:
    - system
    - session
    - agents
```

只注入指定 agent 摘要：

```yaml
contextConfig:
  tags:
    - system
    - session
    - agents
  agents:
    - coder
    - planner
    - customer-service
```

`contextConfig.agents` 也支持逗号字符串或 `"*"`：

```yaml
contextConfig:
  tags:
    - agents
  agents: "coder,planner"
```

`agents` 只影响 prompt 中的 agent 摘要上下文，不影响 channel 可访问范围、`agent_invoke` 权限或 catalog 可见性。

## Prompt Files

目录式 agent 会按约定读取 prompt 文件：

- 普通模式默认读取 `AGENTS.md`
- 顶层 `promptFile` 可显式覆盖普通模式 prompt
- `PLAN-EXECUTE` 默认读取 `AGENTS.plan.md`、`AGENTS.execute.md`、`AGENTS.summary.md`
- `stageSettings.<stage>.promptFile` 可显式覆盖对应阶段
- `PLAN-EXECUTE` 阶段缺少约定文件时，先回退顶层 `promptFile`，再回退 `AGENTS.md`
- `mode: CODER` 还会按 `configs/coder-settings.yml` 读取 `runtimeConfig.workspaceRoot/AGENTS.md`，用于注入项目级 workspace 规则；配置了 `projectConfig.promptFiles` 时改按该列表读取
- `mode: KBASE` 不读取 knowledge source 中的 `AGENTS.md`，只通过 KBASE 系统提示和知识库工具工作；普通 Agent 附加 KBASE capability 时仍按自身 mode 规则加载 prompt files

`PLAN-EXECUTE` 的阶段配置推荐统一放在 `stageSettings.<stage>` 下：

```yaml
stageSettings:
  plan:
    promptFile: AGENTS.plan.md
    modelConfig:
      modelKey: qwen3-max
    toolConfig:
      tools: ["file_read", "file_grep"]
```

阶段内 `modelConfig` 支持 `modelKey`、`reasoning`、`sampling`、`maxOutputTokens` 等运行参数；阶段内 `toolConfig.tools` 覆盖该阶段可用工具。

## CODER Mode

CODER 是一等 agent mode。内建 icon、工具、context、budget、cache key、创建前缀及 capability 的代码事实源是 `internal/agent/coder`；catalog 只在 YAML 未显式声明对应字段时应用该 profile。非沙箱 CODER 要求 `runtimeConfig.workspaceRoot` 为绝对路径、`@chat`，或可展开为绝对路径的 `~`/`~/...`；声明了 `runtimeConfig.environmentId` 的沙箱 CODER 可省略宿主机 workspace。

通过管理接口 `/api/admin/agents/create` 创建 `mode: CODER` 或 `mode: KBASE` agent 时，请求只需要表达创建意图；CODER 通常传 `runtimeConfig.workspaceRoot`，ACP CODER 额外传 `runtimeConfig.acpBridgeId`，专用 KBASE 推荐传 `kbaseConfig.source.root`，旧客户端仍可传 `runtimeConfig.workspaceRoot`。创建前缀和 mode profile 分别由 `internal/agent/coder`、`internal/agent/kbase` 维护；配置文件只提供可覆盖的模型、reasoning、embedding 与 budget 输入。存在 `runtimeConfig.workspaceRoot` 时，生成流程会把缺省 `name` 设为目录 basename。手写 `runtime/agents/*.yml` 或 `runtime/agents/<key>/agent.yml` 时仍使用配置文件里的显式 `key` / `name`、最终 `modelConfig` 与最终 `budget`。

配置了 `platform_config` 工具的治理 agent 可以在不调用 HTTP 管理接口、不传递 JWT 的前提下读取同一组有效创建默认值。`action: get` 只接受 `agents.creation.coder` 与 `agents.creation.kbase` 两个精确路径，返回 `keyPrefix`、小型 `definitionDefaults`、`ready` 与 `missingFields`；它不回显 ACP、provider、MCP 或 runtime env 中的凭据。`action: validate` 可只读校验完整的 agent、team、skill 或 MCP server 候选内容，校验本身不写文件、不触发 catalog 重载。实际创建和修改仍由现有文件工具完成，并由目录 watcher 在落盘后热重载。

## KBASE 公共能力与 Mode

KBASE 是一项可组合的 Agent 公共能力。platform 对知识源持续执行文本抽取、切块、embedding、LanceDB generation 检索和增量 watcher；`mode: KBASE` 则是这项能力的专用预设：始终启用、故障策略为 required，并把工具和 memory 严格限制为原有 KBASE mode 边界。`REACT`、`PLAN-EXECUTE` 与原生非 ACP `CODER` 可以用 `kbaseConfig.enabled: true` 附加同一能力，同时保留各自原有工具、prompt files 和 memory 行为。ACP CODER、`PROXY`、`CHANNEL` 以及 TEAM 协调器不能直接附加；Team 成员本身可以携带。

LanceDB 是唯一检索库；SQLite `control.db` 只保存 generation、文件状态、refresh run 和恢复日志，不保存 chunk、FTS 或 embedding，并且只接受当前 schema。公共配置、source 解析、索引/检索默认值、工具名、watcher 与存储契约以 `internal/kbase` 为代码事实源；`internal/agent/kbase` 只维护专用 mode 的 profile、prompt、system-init、创建策略与严格边界。`kbase_files` 只浏览 active KBASE generation 的文件元数据，不等同于 host filesystem 的 `file_glob` / `file_read`。

目录式普通 Agent 推荐配置如下。相对 `source.root` 以 `agent.yml` 所在目录解析，因此 Agent 目录可以只保存可迁移文档，索引仍默认写入 runtime：

```text
zenmi/
├── agent.yml
└── knowledge/
    ├── product.md
    └── faq/
        └── billing.md
```

```yaml
key: zenmi
name: Zenmi
mode: REACT
modelConfig:
  modelKey: deepseek-v4-pro
toolConfig:
  tools:
    - datetime
    - web_fetch
kbaseConfig:
  enabled: true
  source:
    root: ./knowledge
  embedding:
    modelKey: babelark-text-embedding-v4
  storage:
    location: runtime
```

启用后五个 KBASE 工具会自动加入最终工具集合，不要求在 `toolConfig.tools` 中重复声明；普通 Agent 原有工具不会被过滤。公共 capability prompt 会在 run 创建时冻结到 session，包含“先检索、按来源引用、资料不足时明确说明、文档属于非可信证据”等规则。catalog 热重载只影响之后创建的新 run，既有 run 不会在执行中途突然获得或失去能力。

专用 KBASE mode 的推荐配置：

```yaml
key: docs_kbase
name: Docs KBase
mode: KBASE
modelConfig:
  modelKey: deepseek-v4-pro
  reasoning:
    effort: MEDIUM
kbaseConfig:
  source:
    root: /absolute/docs
  tags:
    - docs
    - knowledge-base
  embedding:
    modelKey: babelark-text-embedding-v4
```

兼容规则：旧的 `mode: KBASE` 若未配置 `kbaseConfig.source.root`，仍回退到 `runtimeConfig.workspaceRoot`；`kbaseConfig.enabled` 默认为 `true` 且显式写 `false` 会校验失败。非 KBASE mode 不回退普通 workspace，必须显式写 `enabled: true` 且提供 `source.root`。平铺 `runtime/agents/<key>.yml` 的 source 只接受绝对路径或 `~/...`；目录式 Agent 才能使用相对路径。`@chat`、文件系统根目录与无法解析的路径均会被拒绝。`enabled: false` 可以保留其他 KBASE 配置，但不会注入 prompt/工具，也不会注册 watcher 或启动 refresh。

完整可覆盖配置：

```yaml
kbaseConfig:
  enabled: true
  source:
    root: ./knowledge
  tags:
    - 售后
    - 工单
  embedding:
    modelKey: openai-text-embedding-3-small
  storage:
    location: runtime
  include:
    - "**/*.md"
    - "**/*.txt"
    - "**/*.html"
    - "**/*.htm"
    - "**/*.pdf"
    - "**/*.docx"
    - "**/*.pptx"
  exclude:
    - ".git/**"
    - ".kbase/**"
    - "node_modules/**"
  chunk:
    unit: estimatedTokens
    maxTokens: 1000
    overlapTokens: 100
  retrieval:
    topK: 8
    fusion: rrf
    rrfK: 60
    vectorWeight: 0.7
    ftsWeight: 0.3
    candidateFloor: 30
    candidateMultiplier: 4
    candidateMax: 500
```

`modelConfig.modelKey` 是 Agent 回答问题时使用的大模型；`modelConfig.reasoning.effort` 是该回答模型的思考程度。通过管理接口新建专用 KBASE agent 且请求未提供这些字段时，后端会从 `configs/kbase-settings.yml` 读取创建输入并把最终值写入 agent YAML。`kbaseConfig.tags` 是可公开的调度标签；当 enabled capability 所属 agent 通过 client channel 申报 Gateway Agent Card 时，runtime 会用这些标签生成 `kb.query.<externalAgentKey>` 隐式技能。标签不得含知识库原文、路径、凭证或未脱敏隐私。`kbaseConfig.embedding.modelKey` 是索引和检索知识库时使用的 embedding 模型业务别名，必须引用 `runtime/registries/models/*.yml` 中 `type: embedding` 的模型，不控制回答模型或思考程度。`kbaseConfig.chunk` 是每个 capability 独立的切块配置：未配置时使用 estimated token 默认值；字符切块必须显式写 `unit: chars`，再使用 camelCase 的 `maxChars` / `overlapChars`。未写 unit 的字符字段、kebab-case retrieval 键和 unit 别名均会被拒绝。storage 路径规则由 `internal/kbase` 维护；默认 `storage.location: runtime` 写入 `AP_RUNTIME_KBASE_DIR/<agentKey>`，`storage.location: workspace` 写入 knowledge source 内的 `.kbase`，一个 canonical storageDir 不允许被多个 Agent 共用。HTML 使用静态抽取，不执行脚本；抽取时会跳过脚本、样式和明显隐藏内容，只索引可见正文。`kbaseConfig.embedding` 只接受 `modelKey`，旧式 `embedding.providerKey/model/dimension/timeout` 不再读取，也不会回退到 provider 顶层 `embedding.*` 或 `provider.memory.embedding`。

故障策略由 capability requirement 决定。`mode: KBASE` 映射为 required：启动期 storage contract 无效时 Agent 以 `invalid_kbase_storage` 留在管理端并从运行 catalog 隔离，sidecar 故障使 `/healthz` 返回 unhealthy。普通 Agent 附加能力映射为 optional：相同存储、embedding 或 sidecar 故障不会移除 Agent，`status` 返回 `degraded/stale`，知识库工具返回明确的 `kbase_unavailable`，而 Agent 的非 KBASE 工具仍可工作；修复后成功 refresh 会清除启动降级状态。两种 requirement 都不迁移、不修复、也不回退旧 SQLite 检索数据。

retrieval 固定使用加权 RRF。每路先取 `candidateK = min(candidateMax, max(candidateFloor, candidateMultiplier * (offset + limit)))`，然后将 vector/FTS 权重归一化并计算 `weight / (rrfK + rank)`。对外 score 再按理论 rank-1 上限归一化并 clamp 到 `[0,1]`；只进入某一路的结果分别标记 `vector` / `fts`，两路同时命中标记 `hybrid`。同分时按 path、startLine、chunk ID 稳定排序。`matchCount` 是有限候选池的去重并集数，不是全库命中数；候选路径达到 `candidateMax` 或分页后仍有结果时 `truncated=true`。

retrieval 中的 topK/RRF/权重/候选池只进入 `queryHash`，修改后立即作用且不会全量重建。embedding 模型或 dimension、chunk、include/exclude、extraction、FTS tokenizer 等进入 `indexHash`，变化时构建新 generation。`force=true` 也构建新 generation，不原地清空 active generation。

source watcher 将 2 秒 debounce 内的事件合并为路径 change set；同一路径去重、父目录覆盖子路径，超过 1000 条或 watcher overflow 时退化为完整 reconcile。日常新增、修改和删除只处理事件路径，手工 `force=false`、启动和 10 分钟周期任务仍扫描整个 knowledge source 对账；只有首次建库、`indexHash` 变化或 `force=true` 创建新 generation。

LanceDB 使用 ICU4X word segmentation 预分词，再用 Lance FTS whitespace tokenizer 建索引，因此不需要分发 Jieba 模型。当前这仍是文本型 KBASE：PDF/DOCX/PPTX/HTML 是抽取文本后检索，没有图片、音频、视频 embedding 管线。存储、迁移、回滚和 sidecar 详情见 [KBASE LanceDB 迁移](KBASE-LanceDB迁移.md)。

## Budget

运行步数统一配置在 `budget` 下。

推荐写法：

```yaml
budget:
  timeout: 3600
  maxSteps: 240
  hitl:
    timeout: 600
    question:
      timeout: 600
    approval:
      timeout: 600
    form:
      timeout: 600
  tool:
    maxCalls: 200

stageSettings:
  plan:
    budget:
      maxSteps: 20
      tool:
        maxCalls: 40
  execute:
    budget:
      maxSteps: 20
      tool:
        maxCalls: 40
  summary:
    budget:
      maxSteps: 1
```

- `budget.maxSteps` 是整次 run 的模型轮次总上限；resolved budget 中会同步到 `model.maxCalls`。
- `tool.maxCalls` 可显式配置；未配置时，如果同级配置了 `maxSteps`，默认按 `maxSteps * 2` 推导。
- PLAN-EXECUTE 阶段预算写在 `stageSettings.<stage>.budget`。
- `budget.timeout`、`budget.model.timeout`、`budget.tool.timeout`、`budget.hitl.timeout` 等预算超时字段单位都是秒。
- `budget.timeout` 是单次 run 的执行时长上限；一个 run 是一条用户消息触发的 agent 执行，从开始到最终回答结束。CODER planning confirmation 的用户等待时间不计入该预算，其他等待或执行时间仍按既有规则计入。
- `budget.model.timeout` 是模型流式 idle 超时；首个响应 chunk 或任意两个 chunk 之间超过该秒数无响应则报错。chat model registry 顶层 `timeout` 配置后会覆盖该值，常用于 MiniMax / transit-hub 这类可能长时间不发 SSE 心跳的模型。
- `budget.tool.timeout` 是单次后端工具调用硬上限；bash/file/external 等工具不能超过该值。
- CODER 运行时预算优先级为 agent.yml 显式 `budget` > CODER profile fallback；fallback 的代码事实源是 `internal/agent/coder/profile.go`。`configs/coder-settings.yml -> default-agent.budget` 只在创建 CODER agent 时写入默认 `budget`，不会作为运行时动态覆盖；已有显式 budget 不会自动迁移。
- `budget.hitl.timeout` 是当前智能体的 HITL/awaiting 默认，会覆盖 `configs/runtime.yml` 的全局 `budget.hitl`；`question`、`approval`、`form` 可用 mode 覆盖，`approval`、`form` 还会优先使用具体等待项自己的 `timeout` 秒值（如 bash-hook subcommand 或 form/tool 定义）。未配置时内置 fallback 为 `600` 秒。CODER `finalize_planning` 的 planning confirmation 是独立机制，永久等待、不读取 `budget.hitl.plan`，并且 `awaiting.ask(mode=planning)` 不下发 `timeout`；遗留 `budget.hitl.plan` 静默忽略。CODER 使用 `stageSettings.planning` 与 `budget.stages.planning`；旧的 CODER `stageSettings.plan` / `budget.stages.plan` 会在加载时失败。`plan_*` 与 plan-tasks 是执行任务跟踪，和 planning confirmation 无关。其他事件中的 `awaiting.ask.timeout` 单位为秒：配置值 `0` 表示未配置，事件值 `0` 表示无限等待、不自动超时，正值由后端按真实时间独立计时，不受 observer / attach / detach 状态影响。
- `REACT` 使用 `react` 单阶段；普通 `CODER` 使用 `execute` 阶段；CODER planning 使用 `planning`、`execute` 两个阶段；`PLAN-EXECUTE` 支持 `plan`、`execute`、`summary` 三个预算阶段。

请求顶层 `planningMode: true` 仅允许 `mode: CODER`；其他 mode 会被拒绝。planning 阶段仅暴露 `file_read`、`file_glob`、`file_grep`、`datetime`、`regex`、`vision_recognize`、`ask_user_question`、`finalize_planning`；`finalize_planning` 只接收完整 `markdown`，后端不补标题、不重排正文，并把该 Markdown 原样写入 `AP_RUNTIME_CHATS_DIR/<chatId>/.tools/planning/<runId>_planning_<revision>.md`。live 事件按 `planning.start`、多条 `planning.delta`、`planning.end`、`awaiting.ask(mode=planning)` 的顺序发送；后者是永久等待的 planning confirmation，不下发 `timeout`，并在 `planning` 对象中携带 `planningId` 与 `planningFile`。其 `awaitingId` 与原始 `finalize_planning` tool call id 相同，并随 `_type:"react"` step 的 `awaiting[]` 持久化，不把 Markdown 正文写入 JSONL。JSONL 会保留原始 assistant `tool_calls`，并在用户 approve/reject 后追加匹配同一 `tool_call_id` / `name:"finalize_planning"` 的 `_type:"react-tool"` / `role:"tool"` 结果。`GET /api/chat` 回放时读取该文件并恢复顶层最新 `planning`，同时在对应 `awaiting.ask` 前合成 replay-only 的 `planning.snapshot` 事件供时间线使用。后端不持久化 `planning.start` / `planning.delta` / `planning.end`。

native CODER 中，用户以 `mode=planning` approve 后会结束当前 planning run：旧 live stream 先出现 `awaiting.answer`，再出现旧 run 的 `run.complete`，随后以 stream terminal `reason:"done"` 结束；旧 live stream 不追加新 run 的 `run.start`，也不继续桥接 execution 的 content/tool 事件。服务端在旧 stream done、旧 run finished 后启动新的 execution run，并通过 WebSocket push `run.started { runId, chatId, agentKey }` 让客户端发现新 `runId`；客户端应 attach 新 `runId` 读取新 run 自己的执行流。新的 execution run 首部为 execution run bootstrap `request.query(message:"Execute planning", requestId:<execution request id>)`，随后是新 run 的 `run.start`，完整执行 prompt 只写入 JSONL/model messages；execution run 关闭 `planningMode`，暴露 CODER 执行工具和 `plan_add_tasks`、`plan_get_tasks`、`plan_update_task` 作为可选轻量任务追踪工具。`plan_add_tasks` / `plan_update_task` 会把当前任务快照按 execution run 写入 `AP_RUNTIME_CHATS_DIR/<chatId>/.tools/plan-tasks/<runId>_plan.json`，`/api/chat` 读取该目录的最新快照作为顶层 `plan`；重启后同一 `chatId` 的新 query 不复活旧 run / SSE，但在 CODER 执行类阶段会从最新快照恢复 plan task 内存状态，`plan_get_tasks` 可直接读到恢复结果，后续 `plan_update_task` 会写入新 run 的快照。`plan_add_tasks` 默认追加到当前 plan；当用户开启新的目标/计划时，调用方可传 `mode:"new"` 创建新的 `planId`，并让本次 tasks 成为新 plan 的完整初始任务列表。execution 阶段的最终回复就是用户可见收尾，不再额外启动 `coder-summary` 模型调用。reject 后当前 planning 不会执行，也不会启动新 run，模型可在原 planning run 中生成下一版 planning revision 或直接结束本轮。

CODER 专属系统提示词从 `configs/coder-prompts.yml` 读取；普通 CODER 使用 `system-prompt`，planning 阶段使用 `planning-prompt`，execute 阶段使用 `system-prompt` 与 execute stage prompt。`summary-system-prompt` / `summary-user-prompt-template` 已从 CODER 专属 prompt 配置中删除；`PLAN-EXECUTE` 的 summary prompt 配置不受影响。

### ACP CODER Backend

CODER 可通过 `runtimeConfig.acpBridgeId` 改由 `configs/coder-settings.yml` 中的 ACP bridge 执行。此时 agent 仍是 `mode: CODER`，但执行链路会转发到 `codex-acp-bridge`。除 PROXY 外，ACP CODER 也允许缺省 `modelConfig.modelKey`；如果配置了 `modelConfig.modelKey`，该值会作为上游 bridge 的模型参数使用。未声明 `acpBridgeId` 时默认使用 native CODER，必须显式配置 `modelConfig.modelKey`：

```yaml
# configs/coder-settings.yml
acp-bridges:
  codex:
    base-url: http://127.0.0.1:17071
    timeout-ms: 300000
    auth-token: ${CODEX_ACP_BRIDGE_AUTH_TOKEN:}
```

```yaml
mode: CODER
modelConfig:
  modelKey: gpt-5-codex
runtimeConfig:
  acpBridgeId: codex
  workspaceRoot: /absolute/project/path
projectConfig:
  git:
    expectedBranch: main
```

ACP CODER 的模型可以只作为转发给 bridge 的外部模型参数注册，不需要配置平台原生 provider：

```yaml
# registries/models/gpt-5-codex.yml
key: gpt-5-codex
name: GPT-5 Codex
protocol: ACP_PASSTHROUGH
modelId: gpt-5-codex
```

ACP CODER 中，编程项目配置仍由 platform 的 `agent.yml` 管理，包括可选的 `modelConfig.modelKey`、`runtimeConfig.workspaceRoot` 和 `projectConfig.git.expectedBranch`。Codex CLI、网络代理、Codex app-server/exec 参数由 `codex-acp-bridge` 自身配置管理。ACP CODER 支持把平台 `planningMode` 透传给上游 ACP bridge；具体 planning mode 语义由 `codex-acp-bridge` 与 Codex app-server 实现。ACP CODER v1 不支持 `projectConfig.promptFiles`；工作区规则由 Codex 在目标工作目录中自行读取。

## CHANNEL Mode

`mode: CHANNEL` 表示“把外部 platform/adaptor 上的 agent 导入为本地 agent”。该模式不进入本地 LLM engine，也不要求 `modelConfig.modelKey`；运行时会根据 `channelConfig.channelId` 找到对应 channel，再用 `remoteAgentKey` 作为上游 agent key 发起 `platform-ws` 请求。`channel.mode` 只表示连接建立方向：`client` 由本端主动连出到 `endpoint.url`，`server` 由本端开放 `/ws/channel` 等待对端连入；两者都可以承载 CHANNEL agent 调用。

```yaml
key: remote-coder
name: Remote Coder
mode: CHANNEL
channelConfig:
  channelId: peer-a
  remoteAgentKey: coder
```

`mode: CHANNEL` 的校验规则：

- 必须配置 `channelConfig.channelId` 与 `channelConfig.remoteAgentKey`
- 禁止配置 `channelConfig.exports`
- 对应 channel 必须存在于 `configs/channels.yml`
- `mode: client` channel 必须配置可连接的 `endpoint.url`
- `mode: server` channel 必须已有活跃对端连接；未连接时 `/api/query` 返回 `503 channel <channelId> is not connected`

本地 agent 导出给外部 channel 时，不使用 `mode: CHANNEL`。本地 `REACT / CODER / PLAN-EXECUTE / PROXY` 等 agent 可以声明 `channelConfig.exports`：

```yaml
key: assistant
name: Assistant
mode: REACT
modelConfig:
  modelKey: deepseek-v4-pro
channelConfig:
  exports:
    - channelId: public-entry
      externalAgentKey: assistant
      allow:
        query: true
        submit: true
        fileTransfer: true
```

`exports[].channelId` 必填。`exports[].externalAgentKey` 可选；省略或为空时，默认使用本地 agent 的 `key` 作为外部调用名。例如 `key: kbaseOrchestrator` 的 agent 可以只写：

```yaml
channelConfig:
  exports:
    - channelId: singapore-webclient
      allow:
        query: true
```

外部调用时使用 `{"externalAgentKey": "kbaseOrchestrator"}` 即可匹配。

`exports[].allow.query` 默认 `true`；`submit`、`steer`、`interrupt`、`fileTransfer` 默认 `false`，需要显式开启。导出请求只能从匹配的 channel 进入，并按 `externalAgentKey`（或其默认值）映射到本地 agent key，再检查 allow 权限。

对于 `mode: client` channel，连接和重连成功后还会向 gateway 全量申报当前有效导出的简化 Agent Card。申报范围只认显式 `channelConfig.exports` 且 `allow.query: true` 的本地 agent；旧 `agents: "*"` 配置已移除并会导致加载失败。卡片名称和描述来自 agent 定义，技能来自最终 `skillConfig.skills`，工具来自 agent 最终生效的公开 Tools 集合（包含自动注入的 KBASE/memory 工具）；`internalOnly` 或不可公开工具会被过滤。工具调用名是卡片中的 tool `id`，技能和工具都按 ID 稳定排序、去重。

Skill 的调度标签来自 `SKILL.md` front matter：

```yaml
---
name: 售后知识库问答
description: 查询售后工单和退款知识。
metadata:
  tags:
    - 售后
    - 工单
    - 退款
---
```

目录式工具 YAML 可在根层声明公开 `tags`：

```yaml
name: kbase_search
label: 知识库搜索
description: 搜索已索引的本地知识库。
tags:
  - knowledge-base
  - search
```

Agent Card 仅发送这些公开展示字段，不读取 prompt、workspace 路径、知识库原文、工具参数、环境变量或认证信息。名称、描述和标签不得包含绝对路径、API Key、Token、Cookie、系统/隐藏提示词或未脱敏客户隐私。该卡只是 A2A 技能模型的裁剪版，`tools` 是平台扩展，不是完整 A2A 1.0 Agent Card。详细帧格式和重试语义见 [API与协议](API与协议.md#gateway-agent-card-申报)。

## PROXY Mode

PROXY agent 用于把平台请求转发给兼容的上游服务。新 platform-to-platform / adaptor 接入优先使用 `mode: CHANNEL` 与 `channelConfig`。PROXY 仍可用，只要求 `proxyConfig.baseUrl`，不要求本地 `modelConfig.modelKey`；旧配置中保留的 `modelConfig.modelKey` 仍可被加载并回显，但运行链路不会依赖它：

```yaml
key: codeAssistant.demo
name: 星码
role: CLI 代码助手
description: 透传到 CLI 模式代码助手服务
mode: PROXY
runtimeConfig:
  workspaceRoot: /absolute/project/path
proxyConfig:
  baseUrl: http://127.0.0.1:3210
```

`proxyConfig` 字段语义：

- `baseUrl` 必填，指向上游代理服务根地址
- `transport` 可选，默认 `ws`；显式设为 `sse` 时使用上游 `/api/query` SSE 链路
- `agentKey` 可选，用于覆盖转发给上游的 agent key
- `token` / `tokenEnv` 可选，仅当上游服务启用 Bearer 鉴权时配置；默认不需要 token
- `timeout` 可选，默认 `300`（秒）；通常不需要显式配置

`POST /api/query` 中的 `model` 字段在 PROXY 模式下原样透传给上游，由上游服务解释；platform 不要求该模型存在于本地 model registry，也不会把它写入本地 session/stage settings。

## Static Memory 与 Runtime Memory

agent 目录下的 `memory/memory.md` 当前作为静态背景提示装载：

- 语义：agent 的长期固定背景，不是运行时记忆库
- 注入顺序：`Agent Identity` 与 `SOUL.md` 之后、runtime memory bundle 之前
- 存储位置：随 agent 文件存在，不进入 SQLite memory store

运行时记忆来自 memory store：

- `Stable Memory`：长期稳定、可复用的 `fact`
- `Relevant Observations`：近期观察类 `observation`

`snapshot/*.md` 仅作为导出/审阅视图：

- 不参与 prompt 主链路
- 不作为 runtime memory 的事实源

## Memory Tool 注入

agent definition 可通过 `memoryConfig` 控制默认注入的 memory 工具：

```yaml
memoryConfig:
  enabled: true
  managementTools: true
  embedding:
    providerKey: babelark
  autoRemember:
    enabled: true
    modelKey: minimax-m2_7-anthropic
    timeout: 60
```

规则：

- 未配置 `memoryConfig.enabled` 或配置为 `false` 时，不启用 runtime memory，也不注入 memory tools
- `enabled: true` 时注入基础集：
  - `memory_write`
  - `memory_read`
  - `memory_search`
- `managementTools: true` 时额外注入管理集：
  - `memory_update`
  - `memory_forget`
  - `memory_timeline`
  - `memory_promote`
  - `memory_consolidate`
- `autoRemember.enabled: true` 时，run 完成后自动执行 learn / auto-remember
- `autoRemember.modelKey` 填 `AP_RUNTIME_REGISTRIES_DIR/models/*.yml` 中的 model key，用于记忆筛选、总结、合并
- `embedding.providerKey` 填 `AP_RUNTIME_REGISTRIES_DIR/providers/*.yml` 中的 provider key；embedding 模型、维度和超时优先来自 provider 的 `memory.embedding`，agent 侧也可覆盖

运行时策略：

- `Learn(...)` 成功后会自动执行一轮轻量 observation 整理
- 自动整理默认只做 stale/duplicate 收口，以及“重复出现 observation”的晋升
- 更激进的生命周期治理仍建议通过显式 `memory_consolidate` 触发

## Runtime Config

Go runtime 当前支持在 `agent.yml -> runtimeConfig` 下声明：

```yaml
runtimeConfig:
  environmentId: shell
  level: RUN
  workspaceRoot: /absolute/project/path
  hostAccess:
    readRoots:
      - "@owner"
    writeRoots:
      - "@owner"
  sandboxMounts:
    - platform: owner
      target: /owner
      mode: rw
  env:
    HTTP_PROXY: "http://127.0.0.1:7890"
    HTTPS_PROXY: "http://127.0.0.1:7890"
    TZ: "Asia/Shanghai"
```

约束与语义：

- `env` 只接受 `map[string]string`
- key 必须非空，且不能包含空白字符或 `=`
- value 必须是字面量字符串；空字符串允许并原样注入 host bash 或下发到 Container Hub
- 不支持 `${VAR}` 或其他宿主环境变量展开
- `runtimeConfig.workspaceRoot` 定义宿主机 workspace 边界，供 host bash、文件工具与 PROXY 上游工作目录使用；该字段必须是绝对路径、`@chat`，或可展开为当前用户 home 绝对路径的 `~`/`~/...`
- `runtimeConfig.hostAccess.readRoots/writeRoots` 追加单个 agent 的 host-side 可操作目录，只影响 host `bash` 与 `file_*` 工具；root 支持绝对路径、`~/...`、`@workspace`、`@chat`、`@agent`、`@skills`、`@skills-market`、`@owner`
- `runtimeConfig.sandboxMounts` 配置 Container Hub / sandbox 挂载
- `runtimeConfig.acpBridgeId` 仅对 `mode: CODER` 生效，必须引用 `configs/coder-settings.yml` 中的 `acp-bridges` 条目；存在时该 CODER 使用 ACP backend，不存在时默认使用 native backend。`timeout-ms` 的默认值为 `300000`；修改 `coder-settings.yml` 后必须重启 runtime 才会生效
- `@chat` 会在运行时解析为当前 `AP_RUNTIME_CHATS_DIR/<chatId>`
- `runtimeConfig.environmentId` 是 sandbox context 的唯一入口；不需要也不支持再在 `contextConfig.tags` 中声明 `sandbox`
- agent `runtimeConfig.env` 作为基础值，skill 目录下的 `.runtime-env.json` 会按 agent 声明顺序叠加并覆盖同名键
- `/api/agent` 与 `/api/admin/agents` 的 `sandbox` meta 不会回显 `env`，避免暴露代理地址、凭据或私有 endpoint；sandbox mounts 仍可对外暴露，因为它描述的是白名单路径而非敏感值。`/api/agents` 仅返回列表摘要，不包含 `meta`。

### Agent 专属 `.config`

目录型 agent 可在 `runtime/agents/<agentKey>/.config/` 放置工具配置。平台创建的目录型 agent 会预建该目录并设为 `0700`；手写或已有 agent 即使目录不存在也不会报错，工具会继续使用系统配置。

执行 host bash、agent terminal 与 sandbox bash 时，平台会注入 `XDG_CONFIG_HOME` 和 `AP_AGENT_CONFIG_HOME`，使遵循 XDG 的工具优先查找这个目录。host/terminal 使用实际 agent 路径；Container Hub 使用只读挂载后的 `/agent/.config`，不会把宿主机路径传入容器。`runtimeConfig.env`、skill `.runtime-env.json` 仍可显式覆盖同名环境变量。

`AP_SYSTEM_XDG_CONFIG_HOME` 仅供支持 overlay 的 builtin 保留原系统 XDG 根。当前 dbx/httpx 会在 agent 中没有同名连接/site 时回退系统配置；显式 `--config` 不回退。`.config` 本期只保存静态配置，httpx 的 secret 与 state 仍使用其原有的系统或显式目录；敏感内容不得提交。

PROXY agent 会把 `runtimeConfig.workspaceRoot` 作为上游 `params.cwd` 发送给代理服务。客户端请求不得直接携带 `params.cwd`；需要切换项目时，应修改对应 agent 的 `runtimeConfig.workspaceRoot`。

CODER 可额外声明顶层 `projectConfig`：

```yaml
projectConfig:
  promptFiles:
    - source: workspace
      path: AGENTS.md
    - source: agent
      path: AGENTS.md
  git:
    expectedBranch: main
```

- `source: workspace` 从 `runtimeConfig.workspaceRoot` 解析后的目录读取
- `source: agent` 从当前 agent 定义目录读取
- `promptFiles` 使用 `source/path` 结构声明

## 当前建议

当前建议：

- 把 `runtime/agents` / `runtime/teams` / `runtime/skills-market` 视为事实源
- 新增 agent 行为时优先修改 definition 文件，其次再补 `internal/catalog` 与 `internal/engine`
- 若某个字段尚未在 Go 中消费，保持显式报错或显式忽略，避免静默吞掉

## 后续演进原则

- 文件命名尽量与参考仓库保持一致
- 语义清晰优先于实现细节
