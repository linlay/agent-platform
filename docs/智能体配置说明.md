# 智能体配置说明

本文件说明 agent / team / skill 的目录化配置、prompt 文件、CODER、memoryConfig 与 runtimeConfig。当前 Go runtime 已支持从 `configs/runtime.yml -> paths.agents-dir` 读取 `runtime/agents/*.yml` 与 `runtime/agents/<key>/agent.yml`，并将结果直接暴露给 `/api/agents` 与 query agent 选择逻辑。

## 当前状态

当前 Go 版运行时：

- `models` 会从 `REGISTRIES_DIR/models` 读取
- provider 连接配置会从 `REGISTRIES_DIR/providers` 读取
- catalog 中的 agent/team/skill/tool 已改为目录驱动
- `runtime/agents/*.yml` 与 `runtime/agents/<key>/agent.yml` 都会参与装载

因此，这份文档当前主要承担两件事：

1. 说明与 Java 对齐后的 Agent Definition 形态
2. 标注当前 Go 版已接入与仍待补齐的细节

## 目标形态

参考仓库中的完整 Agent Definition 目标形态如下：

```yaml
key: agent_key
name: agent_name
role: 角色标签
description: 描述
icon: "emoji:🤖"
greetings:
  - 我可以帮你总结材料、拆解问题，并给出下一步行动建议。
  - 需要快速进入状态时，我会先帮你找最值得处理的切入点。
wonders:
  - 帮我总结这段材料
  - 给我 5 个可以马上执行的下一步
modelConfig:
  modelKey: qwen3-max
toolConfig:
  tools: ["bash", "datetime"]
mode: ONESHOT
plain:
  systemPrompt: |
    系统提示词
```

长期看，Go 版也会沿用这一思路：

- 前 4 行固定 `key`、`name`、`role`、`description`
- `modelConfig` 引用 `registries/models/*.yml`
- `toolConfig` 只声明 `tools` 工具名列表；不支持 `toolConfig.overrides`
- `mode` 决定 agent 执行模式

## Model Sampling

agent 可在 `modelConfig.sampling` 声明默认采样参数；`stageSettings.plan/execute/summary.modelConfig.sampling` 可按字段覆盖：

```yaml
modelConfig:
  modelKey: qwen3-max
  sampling:
    temperature: 0.7
    topP: 0.9
    presencePenalty: 0
    frequencyPenalty: 0
    seed: 42

stageSettings:
  execute:
    modelConfig:
      sampling:
        temperature: 0.4
```

继承优先级为 `stageSettings.<stage>.modelConfig.sampling` > `modelConfig.sampling` > 运行时默认。未配置时，OpenAI-compatible 主链路仍发送 `temperature: 0`，保持确定性默认行为。

字段使用 camelCase：`temperature`、`topP`、`presencePenalty`、`frequencyPenalty`、`seed`。OpenAI-compatible 协议会映射为 `temperature`、`top_p`、`presence_penalty`、`frequency_penalty`、`seed`；Anthropic 协议只发送 `temperature` 和 `top_p`。

## Agent Identity 与 SOUL.md 边界

`agent.yml` 是 agent 身份与能力元数据的唯一事实源：

- `key`
- `name`
- `role`
- `description`
- `greetings`
- `wonders`
- `mode`
- `toolConfig`
- `skillConfig`
- `runtimeConfig`

运行时会先根据 `agent.yml` 生成统一的 `Agent Identity` prompt section，再拼接其他 prompt 层。

`SOUL.md` 的职责是长期行为提示，不是配置副本：

- 允许写人格、协作方式、风险姿态、硬边界、非目标
- 允许写长期稳定的行为约束
- 不应重复 `key/name/role/description/mode`
- 不应复制 tools、skills、sandbox、环境路径或一次性任务说明

推荐结构：

```md
# Soul

## Persona

## Boundaries

## Working Style

## Long-Term Notes
```

## Greetings 与 Wonders

`greetings` 与 `wonders` 都是 UI 展示元数据，不参与 agent identity prompt 或系统提示词。

`greetings` 用来配置 agent 的开场/占位介绍，语义是“告诉人类这个智能体能做什么”。它适合展示在聊天输入框 placeholder 或空状态里，例如：

```yaml
greetings:
  - 我可以帮你练习词汇、复盘错题，也能把今天的学习拆成小步骤。
  - 你可以把单词表发给我，我会帮你设计一轮轻量测验。
```

支持配置多条，运行时会过滤空字符串，并通过 `/api/agents` 与 `/api/agent` 原样暴露给客户端；需要随机展示时，前端可以从数组中随机选一条。

单条问候语写法：

```yaml
greeting: 我可以帮你快速看懂这个项目，并给出下一步行动建议。
```

`wonders` 保持原语义：具体的 query 问题参考，也就是用户可以直接点选或提交的示例问题。

```yaml
wonders:
  - Quiz me on vocabulary
  - 帮我检查当前工作区的变更
  - |-
    帮我制定一个三步学习计划
    并给出今天可以开始的第一步
```

## Context Tags

当前 Go runtime 不会为所有 agent 自动附加一组全局默认 `context tags`；每个 agent 只从 `contextConfig.tags` 读取声明：

```yaml
contextConfig:
  tags:
    - system
    - session
```

支持标签：

- `system`
- `session`
- `owner`
- `all-agents`

其中：

- `session` 会暴露运行时上下文
- `owner` 会注入 `configs/runtime.yml -> paths.owner-dir` 下的 markdown 内容；`mode: CODER` 不会默认启用，需要显式声明
- `sandbox` 不再通过 `context tags` 控制；只要 agent 声明了 `runtimeConfig.environmentId`，运行时会自动注入 sandbox context
- runtime memory context 不再通过 `context tags` 控制；只有 agent 显式开启 `memoryConfig.enabled: true` 时，运行时才会自动注入 memory context

## Prompt Files

目录式 agent 会按约定读取 prompt 文件：

- 普通模式默认读取 `AGENTS.md`
- 顶层 `promptFile` 可显式覆盖普通模式 prompt
- `PLAN_EXECUTE` 默认读取 `AGENTS.plan.md`、`AGENTS.execute.md`、`AGENTS.summary.md`
- `stageSettings.<stage>.promptFile` 可显式覆盖对应阶段
- `PLAN_EXECUTE` 阶段缺少约定文件时，先回退顶层 `promptFile`，再回退 `AGENTS.md`
- `mode: CODER` 还会按 `configs/coder-settings.yml` 读取 `runtimeConfig.workspaceRoot/AGENTS.md`，用于注入项目级 workspace 规则；配置了 `projectConfig.promptFiles` 时改按该列表读取

`PLAN_EXECUTE` 的阶段配置推荐统一放在 `stageSettings.<stage>` 下：

```yaml
stageSettings:
  plan:
    promptFile: AGENTS.plan.md
    modelConfig:
      modelKey: qwen3-max
    toolConfig:
      tools: ["file_read", "file_grep"]
```

阶段内 `modelConfig` 支持 `modelKey`、`reasoning`、`sampling`、`maxOutputTokens` 等运行参数；阶段内 `toolConfig.tools` 覆盖该阶段可用工具。

## CODER Mode

CODER 是一等 agent mode。CODER 默认使用 `bash`、`file_read`、`file_write`、`file_edit`、`file_glob`、`file_grep`、`datetime`、`regex`、`vision_recognize`。非沙箱 CODER 要求 `runtimeConfig.workspaceRoot` 为绝对路径、`@chat`，或可展开为绝对路径的 `~`/`~/...`；声明了 `runtimeConfig.environmentId` 的沙箱 CODER 可省略宿主机 workspace。

## Budget

运行步数统一配置在 `budget` 下。

推荐写法：

```yaml
budget:
  timeout: 1800
  maxSteps: 240
  hitl:
    timeout: 600
    question:
      timeout: 600
    approval:
      timeout: 600
    form:
      timeout: 600
    plan:
      timeout: 600
  tool:
    maxCalls: 200

stageSettings:
  plan:
    budget:
      maxSteps: 20
      tool:
        maxCalls: 40
  execute:
    budget:
      maxSteps: 20
      tool:
        maxCalls: 40
  summary:
    budget:
      maxSteps: 1
```

- `budget.maxSteps` 是整次 run 的模型轮次总上限；resolved budget 中会同步到 `model.maxCalls`。
- `tool.maxCalls` 可显式配置；未配置时，如果同级配置了 `maxSteps`，默认按 `maxSteps * 2` 推导。
- PLAN_EXECUTE 阶段预算写在 `stageSettings.<stage>.budget`。
- `budget.timeout`、`budget.model.timeout`、`budget.tool.timeout`、`budget.hitl.timeout` 等预算超时字段单位都是秒。
- `budget.timeout` 是单次 run 总耗时上限；一个 run 是一条用户消息触发的 agent 执行，从开始到最终回答结束。
- `budget.model.timeout` 是模型流式 idle 超时；首个响应 chunk 或任意两个 chunk 之间超过该秒数无响应则报错。
- `budget.tool.timeout` 是单次后端工具调用硬上限；bash/file/external 等工具不能超过该值。
- `budget.hitl.timeout` 是当前智能体的 HITL/awaiting 默认，会覆盖 `configs/runtime.yml` 的全局 `budget.hitl`；`question`、`plan` 可用 mode 覆盖；`approval`、`form` 还会优先使用具体等待项自己的 `timeout` 秒值（如 bash-hook subcommand 或 form/tool 定义）。未配置时内置 fallback 为 `600` 秒。协议中的 `awaiting.ask.timeout` 字段单位为秒。配置值 `0` 表示未配置；事件协议中的 `awaiting.ask.timeout=0` 表示无限等待、不自动超时。
- `REACT` 使用 `react` 单阶段；普通 `CODER` 使用 `execute` 阶段；CODER planning 和 `PLAN-EXECUTE` 支持 `plan`、`execute`、`summary` 三个预算阶段。

请求顶层 `planningMode: true` 只对 `mode: CODER` 生效：planning 阶段仅暴露 `file_read`、`file_glob`、`file_grep`、`datetime`、`regex`、`vision_recognize`、`ask_user_question`、`finalize_planning`；`finalize_planning` 只接收完整 `markdown`，后端不补标题、不重排正文，并把该 Markdown 原样写入 `CHATS_DIR/<chatId>/.tools/plans/<runId>_planning_<revision>.md`。live 事件按 `planning.start`、多条 `planning.delta`、`planning.end`、`awaiting.ask(mode=plan)` 的顺序发送；确认计划的 `awaiting.ask(mode=plan)` 会在 `plan` 对象中携带 `planningId` 与 `planningFile`，其 `awaitingId` 与原始 `finalize_planning` tool call id 相同，并随 `_type:"react"` step 的 `awaiting[]` 持久化，不把 Markdown 正文写入 JSONL。JSONL 会保留原始 assistant `tool_calls`，并在用户 approve/reject 后追加匹配同一 `tool_call_id` / `name:"finalize_planning"` 的 `_type:"react-tool"` / `role:"tool"` 结果。`GET /api/chat` 回放时读取该文件并恢复顶层最新 `planning`，同时在对应 `awaiting.ask` 前合成 replay-only 的 `planning.snapshot` 事件供时间线使用。后端不持久化 `planning.start` / `planning.delta` / `planning.end`。用户以 `mode=plan` 确认后，execution 阶段再暴露 CODER 执行工具，不追加 `plan_update_task`；reject 后当前 plan 不会执行，模型可选择生成下一版 planning revision 或直接结束本轮。

CODER 专属系统提示词从 `configs/prompts.yml` 的 `coder` 节读取；普通 CODER 使用 `system-prompt`，planning 阶段使用 `planning-prompt`，summary 阶段使用 `summary-system-prompt` / `summary-user-prompt-template`。

### ACP CODER Backend

CODER 可通过 `runtimeConfig.acpProxyId` 改由 `configs/coder-settings.yml` 中的 ACP proxy 执行。此时 agent 仍是 `mode: CODER`，但执行链路会转发到 `proxy-acp-codex`。未声明时默认使用 native CODER：

```yaml
# configs/coder-settings.yml
acp-proxies:
  codex:
    base-url: http://127.0.0.1:3211
    timeout: 300
    auth-token: ${CODEX_ACP_PROXY_TOKEN:}
```

```yaml
mode: CODER
modelConfig:
  modelKey: gpt-5-codex
runtimeConfig:
  acpProxyId: codex
  workspaceRoot: /absolute/project/path
projectConfig:
  git:
    expectedBranch: main
```

ACP CODER 的模型可以只作为转发给 proxy 的外部模型参数注册，不需要配置平台原生 provider：

```yaml
# registries/models/gpt-5-codex.yml
key: gpt-5-codex
name: GPT-5 Codex
protocol: ACP_PASSTHROUGH
modelId: gpt-5-codex
```

ACP CODER 中，编程项目配置仍由 platform 的 `agent.yml` 管理，包括 `modelConfig.modelKey`、`runtimeConfig.workspaceRoot` 和 `projectConfig.git.expectedBranch`。Codex CLI、网络代理、Codex app-server/exec 参数由 `proxy-acp-codex` 自身配置管理。ACP CODER v1 不支持 `projectConfig.promptFiles` 和平台 `planningMode`；工作区规则由 Codex 在目标工作目录中自行读取。

## PROXY Mode

PROXY agent 用于把平台请求转发给 AGW / ACP-PROXY 上游服务。最小配置只需要声明 `mode: PROXY` 和 `proxyConfig.baseUrl`：

```yaml
key: codeAssistant.demo
name: 星码
role: CLI 代码助手
description: 透传到 CLI 模式代码助手服务
mode: PROXY
runtimeConfig:
  workspaceRoot: /absolute/project/path
proxyConfig:
  baseUrl: http://127.0.0.1:3210
```

`proxyConfig` 字段语义：

- `baseUrl` 必填，指向上游代理服务根地址
- `transport` 可选，默认 `ws`；显式设为 `sse` 时使用上游 `/api/query` SSE 链路
- `agentKey` 可选，用于覆盖转发给上游的 agent key
- `token` / `tokenEnv` 可选，仅当上游服务启用 Bearer 鉴权时配置；默认不需要 token
- `timeout` 可选，默认 `300`（秒）；通常不需要显式配置

## Static Memory 与 Runtime Memory

agent 目录下的 `memory/memory.md` 当前作为静态背景提示装载：

- 语义：agent 的长期固定背景，不是运行时记忆库
- 注入顺序：`Agent Identity` 与 `SOUL.md` 之后、runtime memory bundle 之前
- 存储位置：随 agent 文件存在，不进入 SQLite memory store

运行时记忆来自 memory store：

- `Stable Memory`：长期稳定、可复用的 `fact`
- `Relevant Observations`：近期观察类 `observation`

`snapshot/*.md` 仅作为导出/审阅视图：

- 不参与 prompt 主链路
- 不作为 runtime memory 的事实源

## Memory Tool 注入

agent definition 可通过 `memoryConfig` 控制默认注入的 memory 工具：

```yaml
memoryConfig:
  enabled: true
  managementTools: true
  embedding:
    providerKey: babelark
  autoRemember:
    enabled: true
    modelKey: minimax-m2_7-anthropic
    timeout: 60
```

规则：

- 未配置 `memoryConfig.enabled` 或配置为 `false` 时，不启用 runtime memory，也不注入 memory tools
- `enabled: true` 时注入基础集：
  - `memory_write`
  - `memory_read`
  - `memory_search`
- `managementTools: true` 时额外注入管理集：
  - `memory_update`
  - `memory_forget`
  - `memory_timeline`
  - `memory_promote`
  - `memory_consolidate`
- `autoRemember.enabled: true` 时，run 完成后自动执行 learn / auto-remember
- `autoRemember.modelKey` 填 `REGISTRIES_DIR/models/*.yml` 中的 model key，用于记忆筛选、总结、合并
- `embedding.providerKey` 填 `REGISTRIES_DIR/providers/*.yml` 中的 provider key；embedding 模型、维度和超时优先来自 provider 的 `memory.embedding`，agent 侧也可覆盖

运行时策略：

- `Learn(...)` 成功后会自动执行一轮轻量 observation 整理
- 自动整理默认只做 stale/duplicate 收口，以及“重复出现 observation”的晋升
- 更激进的生命周期治理仍建议通过显式 `memory_consolidate` 触发

## Runtime Config

Go runtime 当前支持在 `agent.yml -> runtimeConfig` 下声明：

```yaml
runtimeConfig:
  environmentId: shell
  level: RUN
  workspaceRoot: /absolute/project/path
  hostAccess:
    readRoots:
      - "@owner"
    writeRoots:
      - "@owner"
  sandboxMounts:
    - platform: owner
      target: /owner
      mode: rw
  env:
    HTTP_PROXY: "http://127.0.0.1:7890"
    HTTPS_PROXY: "http://127.0.0.1:7890"
    TZ: "Asia/Shanghai"
```

约束与语义：

- `env` 只接受 `map[string]string`
- key 必须非空，且不能包含空白字符或 `=`
- value 必须是字面量字符串；空字符串允许并原样注入 host bash 或下发到 Container Hub
- 不支持 `${VAR}` 或其他宿主环境变量展开
- `runtimeConfig.workspaceRoot` 定义宿主机 workspace 边界，供 host bash、文件工具与 PROXY 上游工作目录使用；该字段必须是绝对路径、`@chat`，或可展开为当前用户 home 绝对路径的 `~`/`~/...`
- `runtimeConfig.hostAccess.readRoots/writeRoots` 追加单个 agent 的 host-side 可操作目录，只影响 host `bash` 与 `file_*` 工具；root 支持绝对路径、`~/...`、`@workspace`、`@chat`、`@agent`、`@skills`、`@skills-market`、`@owner`
- `runtimeConfig.sandboxMounts` 配置 Container Hub / sandbox 挂载
- `runtimeConfig.acpProxyId` 仅对 `mode: CODER` 生效，必须引用 `configs/coder-settings.yml` 中的 `acp-proxies` 条目；存在时该 CODER 使用 ACP backend，不存在时默认使用 native backend
- `@chat` 会在运行时解析为当前 `CHATS_DIR/<chatId>`
- `runtimeConfig.environmentId` 是 sandbox context 的唯一入口；不需要也不支持再在 `contextConfig.tags` 中声明 `sandbox`
- agent `runtimeConfig.env` 作为基础值，skill 目录下的 `.runtime-env.json` 会按 agent 声明顺序叠加并覆盖同名键
- `/api/agents` 与 `/api/agent` 的 `sandbox` meta 不会回显 `env`，避免暴露代理地址、凭据或私有 endpoint；sandbox mounts 仍可对外暴露，因为它描述的是白名单路径而非敏感值

PROXY agent 会把 `runtimeConfig.workspaceRoot` 作为上游 `params.cwd` 发送给代理服务。客户端请求不得直接携带 `params.cwd`；需要切换项目时，应修改对应 agent 的 `runtimeConfig.workspaceRoot`。

CODER 可额外声明顶层 `projectConfig`：

```yaml
projectConfig:
  promptFiles:
    - source: workspace
      path: AGENTS.md
    - source: agent
      path: AGENTS.md
  git:
    expectedBranch: main
```

- `source: workspace` 从 `runtimeConfig.workspaceRoot` 解析后的目录读取
- `source: agent` 从当前 agent 定义目录读取
- `promptFiles` 使用 `source/path` 结构声明

## 当前建议

当前建议：

- 把 `runtime/agents` / `runtime/teams` / `runtime/skills-market` 视为事实源
- 新增 agent 行为时优先修改 definition 文件，其次再补 `internal/catalog` 与 `internal/engine`
- 若某个字段尚未在 Go 中消费，保持显式报错或显式忽略，避免静默吞掉

## 后续演进原则

- 文件命名尽量与参考仓库保持一致
- 语义清晰优先于实现细节
- 未实现的字段要么明确忽略，要么启动时报错，避免静默吞掉
