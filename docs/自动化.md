# 自动化

## 当前状态

Go runtime 已具备 automation registry、orchestrator、dispatcher 与 execution store。automation 定义目录由 `configs/runtime.yml -> paths.automations-dir` 控制，调度器随服务进程启动。

当前自动化能力仍是运行时编排的一部分，不等同于完整产品层任务管理；Desktop 侧 automation 操作可通过 desktop action bridge 暴露给模型。

## 核心流程

```text
启动加载 automations
  -> registry 注册
  -> orchestrator 计算触发
  -> dispatcher 并发执行
  -> execution store 记录结果
```

日志统一写入服务进程 stdout。排障时优先查看是否出现 `automation orchestrator started`、`[automation] registered ...`、`[automation] dispatch ...`。

## 配置与接口

- `configs/runtime.yml` 的 `paths.automations-dir`：automation 定义目录。
- `configs/runtime.yml` 的 `automation.enabled`：是否启用 orchestrator。
- `configs/runtime.yml` 的 `automation.default-zone-id`：平台默认时区；未在单个 automation YAML 中写 `environment.zoneId` 时，cron 按这个时区解释。
- `configs/runtime.yml` 的 `automation.pool-size`：dispatch 最大并发。
- `desktop_action` 支持 `desktop.automations.*` 一组动作。

默认值见 [配置化说明](配置化说明.md)。

## 约束与注意事项

- 时区优先级为：automation YAML 的 `environment.zoneId`、`configs/runtime.yml` 的 `automation.default-zone-id`、最后才回退到进程 `time.Local`。
- 日常自动化默认省略 `environment.zoneId`，表示跟随平台默认时区；需要跨时区或不随平台默认时区变化的任务，再显式写 `environment.zoneId`。
- cron 表达的是对应时区里的本地时间，例如默认时区为 `Asia/Shanghai` 时，`0 9 * * *` 表示北京时间每天 09:00，不是 UTC 09:00。
- API 返回的 `nextFireAt` 是下次触发时间的 epoch milliseconds，供排序和比较；`nextFireTime` 是按 `environment.zoneId` / 默认时区格式化后的 RFC3339 展示时间。Execution API 中 `startedAt` / `completedAt` 保持 epoch milliseconds，`startedTime` / `completedTime` 是同一时间点的 RFC3339Nano 可读展示。
- automation 首次创建出的 chat 会在 chat 摘要、`/api/chat` 详情和 `chat.created` push 中记录 `source:"automation:<automationId>"`。automation 名称不写入 chat 表；如需展示名称，可用 automationId 查询 automation registry。后续 automation 写入已存在 chat 时不会覆盖原 chat source。
- 本地 `make run` 的 automation 日志在启动服务的终端里，不在 `docker compose logs`。
- automation 配置变更遵循运行时配置规则，通常需要重启或重新装载对应 registry。
- automation 与 `agent_invoke` 是不同能力；子智能体并发调度见 [子智能体调度](子智能体调度.md)。
