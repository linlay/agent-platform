# 版本化打包方案

`agent-platform` 现已具备面向 desktop builtin 分发的 program bundle 发布链路，把纯 Go runtime、Rust `kbase-lance-engine` sidecar、配置模板、启停脚本和锁定版本的 rg/dbx/httpx 打成版本化服务包；运行态目录由启动或部署流程按配置创建。

## 1. 当前状态

目前已具备：

- 根目录 `VERSION`
- `make release` / `make release-program`
- `scripts/release.sh`
- `scripts/release-common.sh`
- `scripts/release-program.sh`
- `scripts/release-assets/builtins.lock.json`
- `scripts/stage-builtins.sh` / `scripts/stage-builtins.ps1`
- `scripts/build-kbase-lance-engine.{sh,ps1}`
- `scripts/stage-kbase-lance-engine.{sh,ps1}`
- `scripts/release-assets/program/{unix,windows}`
- `dist/release/` 版本化 bundle 输出

builtins 源目录默认为相邻的 `../agent-platform-builtins`，包含独立的 `ripgrep/`、`dbx/`、`httpx/` 子仓库。CI 可通过绝对路径 `BUILTINS_ROOT` 覆盖。`kbase-lance-engine` 在同一 lock 中登记为 `source-build` 组件：开发构建来自仓库内 `native/kbase-lance-engine`，正式打包只消费 `dist/kbase-lance-engine/<os>-<arch>/` 中已有且带 SHA-256 文件的目标产物。

当前默认产物命名：

- `agent-platform-v0.3.14-darwin-arm64.tar.gz`

如需自定义目标矩阵，可通过 `PROGRAM_TARGET_MATRIX` 或 `PROGRAM_TARGETS` 覆盖。

## 2. 发布命令

标准发布命令：

```bash
make release-program
```

先为本机生成 sidecar artifact：

```bash
make build-kbase-lance-engine
```

六个发布目标的 hook 为：

```bash
make build-kbase-lance-engine-matrix
make release-program-all
```

矩阵命令不会自动安装 Rust target、`protoc`、链接器、Apple SDK、Windows SDK 或签名工具。推荐在六个匹配平台/架构的 runner 上分别执行 `scripts/build-kbase-lance-engine.sh --os ... --arch ...`，汇总 `dist/kbase-lance-engine` 后再打包。缺少任一目标 artifact 或 `.sha256` 时 release 会明确失败。Docker builder 会在隔离的构建阶段安装 `protobuf-compiler`，最终镜像不包含它。

已有可信制品服务器时，也可直接下载单个平台的裸二进制；仓库不内置尚未发布的 URL 或伪造哈希：

```bash
KBASE_LANCE_ENGINE_URL=https://artifacts.example/kbase-lance-engine \
KBASE_LANCE_ENGINE_SHA256=<64位SHA256> \
make release-program PROGRAM_TARGET_MATRIX=linux/amd64
```

下载必须同时提供 SHA-256。macOS codesign/notarization、Windows Authenticode signing 由外部 release runner 完成，本仓库生成的 manifest 不声明二进制已经签名。

等价命令：

```bash
VERSION=$(cat VERSION) ARCH=arm64 bash scripts/release.sh
```

## 3. Bundle 结构

解压后的目录结构如下：

```text
agent-platform/
  manifest.json
  builtins.manifest.json
  .env.example
  backend/
    agent-platform
  bin/
    rg
    dbx
    httpx
    kbase-lance-engine
  licenses/
    ripgrep/
      LICENSE-MIT
      UNLICENSE
    kbase-lance-engine/
      LICENSE-APACHE-2.0
      NOTICE
      THIRD_PARTY_COMPONENTS.json  # 构建 artifact 含 cargo metadata 时生成
  start.sh
  stop.sh
  configs/
    ai-tools.example.yml
    channels.example.yml
    coder-prompts.example.yml
    coder-settings.example.yml
    kbase-prompts.example.yml
    kbase-settings.example.yml
    tools.example.yml
    prompts.example.yml
    runtime.example.yml
    local-public-key.example.pem
```

说明：

- 顶层目录固定为 `agent-platform`，供 Desktop 安装时直接解压到 `userData/services/agent-platform/<version>/`
- `scripts/release-assets/builtins.lock.json` 是组件版本和分发方式的事实源；sidecar 同时固定自身 1.0.0 与 LanceDB SDK 0.30.0
- rg/dbx/httpx 从 lock 指定产物提取；sidecar staging 验证目标 artifact 的 SHA-256。最终文件、来源、SDK 版本和实际输出哈希统一写入 `builtins.manifest.json`
- 四个 builtins 都是正式包必选项；缺失平台产物或哈希不匹配会阻止发布
- 每个 archive 旁生成 `.sha256` 与 `.sizes.json`；安装了 Syft 时同时生成 CycloneDX SBOM，release CI 可设置 `REQUIRE_RELEASE_SBOM=1` 强制要求
- sidecar 构建目录还会保存 `cargo-metadata.json`；staging 只从中提取包名、版本、license 和 registry source，生成不含构建机路径的 `THIRD_PARTY_COMPONENTS.json`。安装了 Syft 时保存并随包复制 sidecar 二进制 SBOM。release CI 可设置 `REQUIRE_KBASE_RELEASE_METADATA=1`，强制要求二者都存在
- runtime 启动后只在自身进程树内把服务包根目录的 `bin/` 前置到 `PATH`
- `start.sh` 默认以守护进程模式启动，并把 PID / 日志写入 `run/`
- release 包不再预置空 `runtime/` 目录；运行态数据目录只根据 `.env` 中的 `AP_RUNTIME_DIR` 创建，Desktop 部署阶段通过 `deploy.sh --ap-runtime-dir` 写入该环境变量
- `deploy.sh` 支持在首次生成缺失配置文件时渲染 `ai-tools.yml`、`coder-settings.yml` 与 `kbase-settings.yml`；其中 KBASE 可通过 `--kbase-model-key`、`--kbase-reasoning-effort`、`--kbase-embedding-model-key` 写入默认回答模型、reasoning effort 与 embedding modelKey，已存在的本地配置不会被覆盖

## 4. Desktop 集成约束

该 bundle 供 desktop 作为 builtin 资源消费，Desktop 宿主会在启动前自动完成：

- 启动时通过 `start.sh --port <port>` 传入监听端口
- 注入 `AP_CONTAINER_HUB_BASE_URL=http://127.0.0.1:<port>`
- 分发 `configs/local-public-key.pem`
- Auth 默认由代码和 `configs/runtime.yml` 的 `auth.enabled=true` 开启

因此 bundle 内只需要携带模板文件，不需要预置真实密钥或固定运行态数据。

## 5. 适用场景

当前推荐两种交付方式：

- Desktop builtin：使用 `make release-program` 产出 tar.gz，随后在 Desktop 宿主仓库执行 `npm run sync:assets` 同步到 Electron 资源目录
- Compose / 开发联调：继续使用 `make run`、`make test`、`make docker-up`；Compose 镜像 tag 由根目录 `VERSION` 注入

Docker 镜像使用 Debian bookworm slim/glibc，并在构建阶段从 `Cargo.lock` 编译 sidecar。最终镜像以 UID/GID 10001 的非 root 用户运行；宿主机 bind mount 的 runtime、workspace 与 KBASE storage 目录必须允许该 UID 写入。容器健康检查验证 runtime 进程与 sidecar 可执行文件，token 保护的随机 loopback 协议健康由 runtime 的 KBASE status 负责。
