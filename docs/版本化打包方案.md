# 版本化打包方案

`agent-platform` 具备面向 desktop builtin 分发的 program bundle 发布链路，把纯 Go runtime、预先同步并校验的 `kbase-lance-engine` sidecar、配置模板、启停脚本和 rg/dbx/httpx/Poppler 打成版本化服务包；运行态目录由启动或部署流程按配置创建。

## 1. 当前状态

目前已具备：

- 根目录 `VERSION`
- `make release` / `make release-program`
- `scripts/release.sh`
- `scripts/release-common.sh`
- `scripts/release-program.sh`
- `scripts/release-assets/builtins.lock.json`
- `scripts/stage-builtins.sh` / `scripts/stage-builtins.ps1`
- `scripts/stage-kbase-lance-engine.{sh,ps1}`
- `scripts/sync-local-builtins.{sh,ps1}`
- `scripts/release-assets/program/{unix,windows}`
- `dist/release/` 版本化 bundle 输出

`../agent-platform-builtins` 是 `sync-local-builtins.sh`（macOS/Linux）和 `sync-local-builtins.ps1`（Windows AMD64）的构建输入，包含独立的 `ripgrep/`、`dbx/`、`httpx/`、`kbase-lance-engine/`、`poppler-pdftotext/` 子项目；CI 可通过绝对路径 `BUILTINS_ROOT` / `-BuiltinsRoot` 覆盖。同步在隔离目录中验证每个 input artifact，生成临时 local lock 后将完整 staged bundle 原子写入 `build/builtins/<os>-<arch>/`。正式 release 只读取此 platform-owned cache，绝不直接读取 sibling collection。cache 的 `builtins.manifest.json` 记录最终 payload hash与本次实际 checkout commit；release 会在复制和 bundle verification 时复验这些 hash。sidecar cache 同时包含 Cargo metadata 派生的依赖清单、sidecar SBOM 与许可证；`poppler-pdftotext` 是 checksum-verified archive tree，包含 launcher 与隔离的原生 runtime。

当前默认产物命名：

- `agent-platform-v0.3.14-darwin-arm64.tar.gz`

如需自定义目标矩阵，可通过 `PROGRAM_TARGET_MATRIX` 或 `PROGRAM_TARGETS` 覆盖。

## 2. 发布命令

标准发布命令：

```bash
make release-program
```

本机开发先构建当前主机 builtin cache：

```bash
./scripts/sync-local-builtins.sh
```

Windows AMD64 使用全原生 PowerShell 5.1 两阶段入口，不依赖 Git Bash：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/sync-local-builtins.ps1 -Target windows/amd64
make release ARCH=amd64
```

Shell 入口默认发布当前主机 OS，PowerShell 入口无显式 matrix 时固定发布 `windows/$ARCH`；两者都从 `build/builtins/<os>-<arch>/` 装配对应 cache。若 target cache 缺失、cache platform 不一致、manifest 未覆盖所有 payload 或任一 SHA-256 不匹配，release 会立即失败。六平台批量准备仍由 `sync-local-builtins.sh --all` 和匹配 runner 负责；本轮原生 PowerShell 同步入口只接受 `windows/amd64`，不会为了 Windows 修复重建 Darwin/Linux。

`scripts/release-assets/builtins.lock.json` 只记录正式分发事实：组件版本、正式源码 commit、正式 archive path/SHA、target matrix 和 archive 内容契约。Shell 本地同步默认在 cache 激活后检查更高的本地语义版本；只有 checkout 干净、现有正式 target matrix 全部产物验证通过并在交互终端输入精确的 `yes`，才将候选版本、commit、archive path/SHA（以及 Poppler 中包含版本号的 source URL）原子写入正式 lock。非交互运行、拒绝确认、dirty checkout、版本降级或目标不完整不得修改它；PowerShell target 检测、ZIP staging、复制方式或普通本机重建 hash同样不得修改它。正式 archive 字节变化必须升级 builtin 版本，不能用同版本覆盖后批量重写其他平台 SHA。

正式 release 的干净目录回归（会删除本仓库 `dist/`）为：

```bash
make test-release-program-clean
```

Darwin ARM64 与 Windows AMD64 的对应 runner 会分别验证 cache、sidecar binary、manifest payload checksum、Cargo metadata、sidecar SBOM、bundle SBOM 与 `verify-program-bundle`。该检查不需要 Rust/Cargo，但要求对应的 local cache 已由 `sync-local-builtins.sh` 准备完成。

等价命令：

```bash
VERSION=$(cat VERSION) ARCH=arm64 bash scripts/release.sh
```

## 3. Bundle 结构

解压后的目录结构如下：

```text
agent-platform/
  manifest.json
  builtins.manifest.json
  .env.example
  backend/
    agent-platform
  bin/
    rg
    dbx
    httpx
    kbase-lance-engine
    pdftotext
  libexec/
    poppler-pdftotext/
      darwin-arm64/
        bin/pdftotext
        lib/*.dylib
  licenses/
    ripgrep/
      LICENSE-MIT
      UNLICENSE
    kbase-lance-engine/
      LICENSE-APACHE-2.0
      NOTICE
      THIRD_PARTY_COMPONENTS.json  # 构建 artifact 含 cargo metadata 时生成
  start.sh
  stop.sh
  configs/
    ai-tools.example.yml
    channels.example.yml
    coder-prompts.example.yml
    coder-settings.example.yml
    kbase-prompts.example.yml
    kbase-settings.example.yml
    tools.example.yml
    prompts.example.yml
    runtime.example.yml
    local-public-key.example.pem
```

说明：

- 顶层目录固定为 `agent-platform`，供 Desktop 安装时直接解压到 `userData/services/agent-platform/<version>/`
- `scripts/release-assets/builtins.lock.json` 是 sync input 的组件版本和分发方式事实源；sidecar 同时固定自身版本与 LanceDB SDK 版本
- sync 从 lock 指定 artifact 提取 rg/dbx/httpx/sidecar，Poppler 从 lock 指定 archive tree 提取；先校验 input SHA-256，再按声明的输出目录原子安装到 target cache。release 从 cache 复制最终文件、来源、SDK 版本、许可证与实际输出哈希，并复验 `builtins.manifest.json` 中的每一个 payload
- Poppler 当前在 lock 中启用 darwin-arm64 与 windows-amd64，两者均为包含 launcher、原生 runtime 和 `share/poppler` data 的 archive tree；其他目标不内置 PDF 解析器。Windows archive 的实际抽取 smoke 仍在对应 release runner 执行。
- 每个 archive 旁生成 `.sha256` 与 `.sizes.json`；安装了 Syft 时同时生成 CycloneDX SBOM，release CI 可设置 `REQUIRE_RELEASE_SBOM=1` 强制要求
- Unix `tar.gz` 打包会显式设置 `COPYFILE_DISABLE=1`，避免 macOS 扩展属性生成 `._agent-platform` AppleDouble 顶层条目而破坏单根目录 bundle 契约
- sidecar release archive 必须携带 `cargo-metadata.json` 和 sidecar 二进制 SBOM；staging 只从前者提取包名、版本、license 和 registry source，生成不含构建机路径的 `THIRD_PARTY_COMPONENTS.json`
- runtime 启动后只在自身进程树内把服务包根目录的 `bin/` 前置到 `PATH`
- `start.sh` 默认以守护进程模式启动，并把 PID / 日志写入 `run/`
- release 包不再预置空 `runtime/` 目录；运行态数据目录只根据 `.env` 中的 `AP_RUNTIME_DIR` 创建，Desktop 部署阶段通过 `deploy.sh --ap-runtime-dir` 写入该环境变量
- `deploy.sh` 支持在首次生成缺失配置文件时渲染 `ai-tools.yml`、`coder-settings.yml` 与 `kbase-settings.yml`；其中 KBASE 可通过 `--kbase-model-key`、`--kbase-reasoning-effort`、`--kbase-embedding-model-key` 写入默认回答模型、reasoning effort 与 embedding modelKey，已存在的本地配置不会被覆盖

## 4. Desktop 集成约束

该 bundle 供 desktop 作为 builtin 资源消费，Desktop 宿主会在启动前自动完成：

- 启动时通过 `start.sh --port <port>` 传入监听端口
- 注入 `AP_CONTAINER_HUB_BASE_URL=http://127.0.0.1:<port>`
- 分发 `configs/local-public-key.pem`
- Auth 默认由代码和 `configs/runtime.yml` 的 `auth.enabled=true` 开启

因此 bundle 内只需要携带模板文件，不需要预置真实密钥或固定运行态数据。

### 4.1 时间 API 契约的同步发布

时间 API 已切换为立即生效的破坏性契约。发布这个服务包时，必须与采用同一契约的 Desktop 和 agent-webclient 同版本交付：所有公开结构化时间点均为 epoch milliseconds `number`，而不是 ISO 字符串或 Unix 秒。

- 不提供 feature flag、双读解析或历史数据迁移。旧 runtime 中不合规的 chat JSONL、archive 或 LLM trace 会在读取时返回 `422 time_contract_violation`，不得由安装程序静默改写。
- 外部 proxy / child SSE、WebSocket producer 必须先升级为输出未加引号的 epoch-milliseconds 整数；字符串、秒级、浮点、零值或缺失的必填 `timestamp` 会终止该流。
- Desktop 必须把缺失 optional `readAt` / `expiresAt` 处理为字段不存在，而不能回填 1970 或本机当前时间。

完整 wire 规则、错误形状和 token 秒级例外见 [API与协议](API与协议.md#统一时间契约破坏性)。

## 5. 适用场景

当前推荐两种交付方式：

- Desktop builtin：使用 `make release-program` 产出 tar.gz，随后在 Desktop 宿主仓库执行 `npm run sync:assets` 同步到 Electron 资源目录
- Compose / 开发联调：继续使用 `make run`、`make test`、`make docker-up`；Compose 镜像 tag 由根目录 `VERSION` 注入

Docker 镜像使用 Debian bookworm slim/glibc，并在构建阶段从 `Cargo.lock` 编译 sidecar。最终镜像以 UID/GID 10001 的非 root 用户运行；宿主机 bind mount 的 runtime、workspace 与 KBASE storage 目录必须允许该 UID 写入。容器健康检查验证 runtime 进程与 sidecar 可执行文件，token 保护的随机 loopback 协议健康由 runtime 的 KBASE status 负责。
