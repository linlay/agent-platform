## 时间契约回归

完成一次 query 后，确认 chat detail 直接提供顶层时间，且所有结构化时间都是 JSON number（不是字符串、`0` 或 `null`）：

```bash
curl -sS "$BASE_URL/api/chat?chatId=replace-me" | jq '
  .data |
  {createdAt, updatedAt, runTimes: [.runs[]? | {startedAt, completedAt}], eventTimes: [.events[]? | .timestamp]}
'
```

`createdAt`、`updatedAt`、每个 run 的 `startedAt` / `completedAt` 与每个 event 的 `timestamp` 必须位于 `1000000000000..9007199254740991`。可选字段（例如 `readAt`、`expiresAt`、KBASE `lastIndexedAt`）未发生时必须不存在，不能以 `0` 或 `null` 表示。

对 proxy / child SSE、WebSocket adapter 注入任意一项负例：`"timestamp":"1700000000000"`、`"timestamp":1700000000`、`"timestamp":1700000000000.5` 或缺少 `timestamp`。未写出响应的 HTTP 流必须返回 `422` 和 `data.error.code:"time_contract_violation"`；已开始的流必须只追加平台本地、带合法 `timestamp` 的 `run.error` 后结束。

使用旧 JSONL、archive 或 trace 夹具时，故意保留字符串、秒级、浮点、`0` 或缺失的必填时间，确认 `/api/chat`、`/api/chat/jsonl`、archive replay、search 与 `/api/chat/llm-trace` 返回 `422 time_contract_violation`，且文件没有被重写。

## KBASE LanceDB 回归测试

以下示例假设 `docs_kbase` 是 `mode: KBASE` agent，workspace 中至少有一份中文和一份英文文档，embedding provider 可用。先在 `configs/kbase-settings.yml` 使用默认灰度模式并重启 runtime：

```yaml
storage:
  engine: auto
migration:
  enabled: true
```

检查初始状态：

```bash
curl -sS "$BASE_URL/api/kbase/docs_kbase/status"
```

可接受的中间态是 `data.engine=sqlite` 且 `stale/indexing=true`，或已完成迁移的 `data.engine=lancedb`。Lance 就绪时应检查：

- `schemaVersion="3"`、`generation.state="active"`、`sidecar.available=true`。
- `indexes.fts.ready=true`、`indexes.vector.ready=true`。
- `legacyAvailable=true`、`pendingRecoveryOperations=0`。
- `files/chunks` 与 legacy 迁移前一致，`migration.progress=1`。

执行普通 refresh：

```bash
curl -sS -X POST "$BASE_URL/api/kbase/docs_kbase/refresh" \
  -H "Content-Type: application/json" \
  -d '{"force":false}'
```

首次 `auto` refresh 可先返回 SQLite refresh 结果，同一个 refresh 会再尝试 snapshot/import/index/validation/activation。如 sidecar 或验证失败，status 的 migration 应保留 error，engine 仍为 SQLite，不应出现空 active index。

已有 active Lance generation 后记录 generation ID，再做强制重建：

```bash
curl -sS -X POST "$BASE_URL/api/kbase/docs_kbase/refresh" \
  -H "Content-Type: application/json" \
  -d '{"force":true}'
```

成功后 generation ID 应变化；构建期间连续查询应仍由旧 active generation 返回结果，不得出现“先清空后导入”的空窗。

执行中文、英文和中英混合查询，要求 agent 明确调用 `kbase_search`：

```bash
curl -N -X POST "$BASE_URL/api/query" \
  -H "Content-Type: application/json" \
  -d '{"chatId":"kbase-lance-smoke","agentKey":"docs_kbase","message":"请使用 kbase_search 检索：本地知识库 LanceDB search，并标注来源"}'
```

在 SSE 中检查 `source.publish kind=kbase`，chunk 的 `matchType` 只能是 `vector|fts|hybrid`，`score` 应位于 `[0,1]`。相同查询重复执行时，同分结果应按 path/startLine/chunkId 保持稳定顺序。

让 agent 调用 scoped search，然后在 tool result 中检查所有 path/type 均未越界：

```bash
curl -N -X POST "$BASE_URL/api/query" \
  -H "Content-Type: application/json" \
  -d '{"agentKey":"docs_kbase","message":"请调用 kbase_search 搜索 generation，限制 pathPrefix=docs、pathGlob=docs/**/*.md、type=.md、limit=8"}'
```

检查分页语义时，`count` 是当页数，`matchCount` 是 vector/FTS 有限候选池的去重 union；任一路达到 `candidateMax` 或分页后仍有结果时 `truncated=true`。

验证文件变更/删除时，在临时测试 workspace 中新建或修改文件，执行 `force:false` refresh，确认 `changedFiles` 增加且搜索不同时返回新旧 chunk；删除文件并 refresh 后，`deletedFiles` 增加且该 file ID 不再可检索。中途终止 sidecar 后再 refresh，supervisor 应退避重启，`KBASE_FILE_OPS` 的 `prepared/lance_committed` 记录在下次启动补交，status 最终回到 `pendingRecoveryOperations=0`。

Release runner 的规模门槛使用 opt-in integration gate（会生成较大的临时 1024 维数据集）：

```bash
AP_KBASE_LANCE_ENGINE=/path/to/kbase-lance-engine \
AP_KBASE_LANCE_PERF_ROWS=10000 \
go test -run '^TestLanceSidecarPerformanceGate$' -count=1 ./internal/agent/kbase

AP_KBASE_LANCE_ENGINE=/path/to/kbase-lance-engine \
AP_KBASE_LANCE_PERF_ROWS=100000 \
go test -run '^TestLanceSidecarPerformanceGate$' -count=1 ./internal/agent/kbase
```

分别要求 retrieval-only p95 不高于 200ms/500ms；100k case 会越过 50k ANN 门槛，sidecar 内部同时执行 ANN 对 flat ground truth 的 recall@10 门禁。

验证 chat replay：

```bash
curl -sS "$BASE_URL/api/chat?chatId=kbase-lance-smoke"
```

应从 `react-tool` 的 `sources.items[]` sidecar 恢复 `source.publish`，并保留 `liveSeq`；迁移影子查询不应写入正式 `source.publish`。

SQLite 紧急回滚需改为 `storage.engine: sqlite` 并重启。因为 Lance 激活后不长期 dual write，首次 search 应先显示 stale/indexing 并触发 workspace refresh，不得直接返回冻结旧数据。回滚验证完成后再切回 `auto`。

验证缺 sidecar 时，只在隔离的 release 副本中移除 `bin/kbase-lance-engine[.exe]`；不要操作生产安装目录。`auto` + legacy DB 应继续 SQLite KBASE，`lancedb` 应返回 unavailable，普通 agent/chat/memory 接口仍可用。

直接下载 staging 只接受同时提供的 URL 和 SHA-256：

```bash
KBASE_LANCE_ENGINE_URL="https://artifacts.example/kbase-lance-engine" \
KBASE_LANCE_ENGINE_SHA256="<64位 SHA-256>" \
scripts/stage-kbase-lance-engine.sh \
  --output release-local \
  --os darwin \
  --arch arm64
```

故意传入错误 SHA-256 必须失败且不把 `.download` 临时文件当成可用 sidecar。当前这些脚本是六平台 build/stage hook；还需 release CI 在 darwin/linux/windows 的 amd64/arm64 runner 执行真实 build/start/search smoke、签名与 notarization，不得因为脚本存在就记录为六平台已验证。

正式 Desktop builtin 发布不需要预先保存 `dist/kbase-lance-engine`。在 Darwin ARM64 release runner 上，从干净目录验证 Platform 自行准备 sidecar：

```bash
make test-release-program-clean
```

该命令会删除本仓库 `dist/`，再直接运行 `make release-program`，要求 Rust、`protoc`、Syft 和锁定 builtins 均已就绪。它会检查 artifact 的 binary、`.sha256`、`cargo-metadata.json`、`sbom.cdx.json` 以及最终 bundle verify；Desktop 仅调用 Platform 的 `make release`，不会传递 sidecar 配置。

## Submit / Steer / Interrupt

`/api/submit` 当前使用统一 HITL 协议：

- 顶层固定是公开 owner + `runId + awaitingId + params`；普通 Agent/legacy Team 使用 `agentKey`，orchestrated Team 使用 `teamId`
- `params` 永远是数组
- 不在 submit 里再传 `mode`
- 后端按 `awaitingId` 反查当前是 `question / approval / form`
- `params[i]` 固定对应 `awaiting.ask` 下发数组的第 `i` 项；`id` 可省略，即使携带也只作审计用途

question:

```bash
curl -X POST "$BASE_URL/api/submit" \
  -H "Content-Type: application/json" \
  -d '{"agentKey":"default_agent","runId":"replace-me","awaitingId":"tool_question","params":[{"id":"q1","answer":"Weekend"},{"id":"q2","answers":["产品更新","使用教程"]}]}'
```

也可以省略 `id`，只要顺序与 `awaiting.ask.questions` 一致：

```bash
curl -X POST "$BASE_URL/api/submit" \
  -H "Content-Type: application/json" \
  -d '{"agentKey":"default_agent","runId":"replace-me","awaitingId":"tool_question","params":[{"answer":"Weekend"},{"answers":["产品更新","使用教程"]}]}'
```

question 的每个答案必须与同索引问题的题型匹配：`multi-select` 使用非空 `answers`，其他题型使用 `answer`，不能同时传两者。题目数量、数值类型、文本非空以及未开启自由输入时的选项值也会校验；`id` 可省略，携带时只用于审计，仍按顺序配对。

例如向多选题错误提交 `answer`：

```bash
curl -X POST "$BASE_URL/api/submit" \
  -H "Content-Type: application/json" \
  -d '{"agentKey":"default_agent","runId":"replace-me","awaitingId":"tool_question","params":[{"id":"q1","answer":"Weekend"},{"id":"q2","answer":"产品更新"}]}'
```

接口仍返回 HTTP 200，但响应中应为 `data.accepted:false`、`data.status:"invalid"`；等待项保持 awaiting，可将第二项改为 `{"id":"q2","answers":["产品更新"]}` 后重新提交。

approval:

```bash
curl -X POST "$BASE_URL/api/submit" \
  -H "Content-Type: application/json" \
  -d '{"agentKey":"default_agent","runId":"replace-me","awaitingId":"await_tool_bash","params":[{"id":"tool_bash","decision":"approve_rule_run","reason":"同规则本轮一并放行"}]}'
```

form:

```bash
curl -X POST "$BASE_URL/api/submit" \
  -H "Content-Type: application/json" \
  -d '{"agentKey":"default_agent","runId":"replace-me","awaitingId":"await_tool_bash","params":[{"id":"form-1","payload":{"applicant":{"name":"Lin","department":"engineering","employee_id":"E1001"},"leave_type":"事假","start_date":"2026-04-21","end_date":"2026-04-22","duration_days":2,"reason":"family_trip","urgent_contact":"Amy","urgent_phone":"13800138000","backup_person":"E2001","notes":"请协助处理审批"}}]}'
```

## Agent Workspace File

读取当前 agent workspace 内的 Markdown / 文本文件；`path` 可以是相对 workspace 的路径，也可以是仍位于 workspace 内的绝对路径：

```bash
curl -X GET "$BASE_URL/api/file?agentKey=replace-agent&path=docs%2FREADME.md"
```

PDF、图片或下载预览使用同一个接口追加 `response=content`：

## Upload / Resource

```bash
curl -X POST "$BASE_URL/api/upload" \
  -F "requestId=upload_01" \
  -F "chatId=replace-me" \
  -F "file=@./README.md"
```

上传成功后，`data.upload.path` 是智能体可读的执行路径；使用响应中的 `data.upload.url` 访问平台资源：

```bash
curl -X GET "$BASE_URL/api/resource?file=replace-me%2FREADME.md"
```

如果部署开启了 resource ticket，需要把 chat 详情或其他接口返回的 ticket 透传到 `t` 参数：

```bash
curl -X GET "$BASE_URL/api/resource?file=replace-me%2FREADME.md&t=replace-me-ticket"
```

说明：

- `/api/file` 面向 agent workspace 文件预览；`/api/resource` 面向 chat 上传资源
- chat 上传资源的浏览器/普通客户端文件字节走 `POST /api/upload` 与 `GET /api/resource`
- WebSocket `/api/upload` 仅用于网关中转 `url + metadata`，由 platform 自己下载文件，不支持直接在 WS payload 中放 base64/二进制内容
