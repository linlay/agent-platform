## 时间契约回归

完成一次 query 后，确认 chat detail 直接提供顶层时间，且所有结构化时间都是 JSON number（不是字符串、`0` 或 `null`）：

```bash
curl -sS "$BASE_URL/api/chat?chatId=replace-me" | jq '
  .data |
  {createdAt, updatedAt, runTimes: [.runs[]? | {startedAt, completedAt}], eventTimes: [.events[]? | .timestamp]}
'
```

`createdAt`、`updatedAt`、每个 run 的 `startedAt` / `completedAt` 与每个 event 的 `timestamp` 必须位于 `1000000000000..9007199254740991`。可选字段（例如 `readAt`、`expiresAt`、KBASE `lastIndexedAt`）未发生时必须不存在，不能以 `0` 或 `null` 表示。

对 proxy / child SSE、WebSocket adapter 注入任意一项负例：`"timestamp":"1700000000000"`、`"timestamp":1700000000`、`"timestamp":1700000000000.5` 或缺少 `timestamp`。未写出响应的 HTTP 流必须返回 `422` 和 `data.error.code:"time_contract_violation"`；已开始的流必须只追加平台本地、带合法 `timestamp` 的 `run.error` 后结束。

使用旧 JSONL、archive 或 trace 夹具时，故意保留字符串、秒级、浮点、`0` 或缺失的必填时间，确认 `/api/chat`、`/api/chat/jsonl`、archive replay、search 与 `/api/chat/llm-trace` 返回 `422 time_contract_violation`，且文件没有被重写。

## KBASE LanceDB 回归测试

以下示例假设 `docs_kbase` 是 `mode: KBASE` agent，workspace 中至少有一份中文和一份英文文档，embedding provider 与 Lance sidecar 可用。`configs/kbase-settings.yml` 不得包含 `storage` 或 `migration` 段。

检查初始状态：

```bash
curl -sS "$BASE_URL/api/kbase/docs_kbase/status"
```

首次索引前应为 `data.engine=lancedb` 且 `stale=true`；Lance 就绪时应检查：

- `schemaVersion="4"`、`generation.state="active"`、`sidecar.available=true`、`sidecar.protocolVersion=2`。
- `indexes.fts.ready=true`、`indexes.vector.ready=true`。
- `pendingRecoveryOperations=0`。
- `files/chunks` 与当前 workspace 的抽取结果一致。

执行普通 refresh：

```bash
curl -sS -X POST "$BASE_URL/api/kbase/docs_kbase/refresh" \
  -H "Content-Type: application/json" \
  -d '{"force":false}'
```

首次 refresh 从 workspace 冷建 Lance generation。如 sidecar 或验证失败，返回 unavailable/refresh error，且不出现 SQLite 回退或空 active index。

已有 active Lance generation 后记录 generation ID，再做强制重建：

```bash
curl -sS -X POST "$BASE_URL/api/kbase/docs_kbase/refresh" \
  -H "Content-Type: application/json" \
  -d '{"force":true}'
```

成功后 generation ID 应变化；构建期间连续查询应仍由旧 active generation 返回结果，不得出现“先清空后导入”的空窗。

执行中文、英文和中英混合查询，要求 agent 明确调用 `kbase_search`：

```bash
curl -N -X POST "$BASE_URL/api/query" \
  -H "Content-Type: application/json" \
  -d '{"chatId":"kbase-lance-smoke","agentKey":"docs_kbase","message":"请使用 kbase_search 检索：本地知识库 LanceDB search，并标注来源"}'
```

在 SSE 中检查 `source.publish kind=kbase`，chunk 的 `matchType` 只能是 `vector|fts|hybrid`，`score` 应位于 `[0,1]`。相同查询重复执行时，同分结果应按 path/startLine/chunkId 保持稳定顺序。

让 agent 调用 scoped search，然后在 tool result 中检查所有 path/type 均未越界：

```bash
curl -N -X POST "$BASE_URL/api/query" \
  -H "Content-Type: application/json" \
  -d '{"agentKey":"docs_kbase","message":"请调用 kbase_search 搜索 generation，限制 pathPrefix=docs、pathGlob=docs/**/*.md、type=.md、limit=8"}'
```

检查分页语义时，`count` 是当页数，`matchCount` 是 vector/FTS 有限候选池的去重 union；任一路达到 `candidateMax` 或分页后仍有结果时 `truncated=true`。

验证 watcher 增量链路时，在临时测试 workspace 依次执行以下操作，每步等待超过 2 秒 debounce 后读取 status 的 `lastRun`：

- 新建一个唯一内容文件：`scope=delta`、`candidatePaths=1`、`newFiles=1`，generation ID 不变，搜索能立即命中新内容。
- 只触碰 mtime：`metadataOnlyFiles=1`、`embeddedChunks=0`。
- 修改多 chunk 文件中的一小段：`modifiedFiles=1`，`reusedChunks>0`，且 `embeddedChunks` 只包含变化或新增 chunk。
- 删除单个文件：`deletedFiles=1`、`scannedFiles=0`、`embeddedChunks=0`，generation ID 不变，搜索立即不可见。
- 重命名文件：结果等价于旧路径删除和新路径新增；删除目录时该前缀下的 active/skipped/error 记录都不再出现在默认文件列表。

随后执行一次 `force:false`，确认 `scope=reconcile` 能修复故意漏报的文件变化；执行 `force:true` 时才应更换 generation ID。中途终止 sidecar 后再 refresh，supervisor 应退避重启，`KBASE_FILE_OPS` 的 `prepared/lance_committed` 记录在下次启动补交，status 最终回到 `pendingRecoveryOperations=0`。

Release runner 应在真实 sidecar 环境执行首次索引、增量 refresh、强制 generation rebuild、generation rollback、文件操作恢复和 sidecar unavailable 的 smoke；其中任何失败都不得回退或读取遗留 SQLite 检索库。

验证 chat replay：

```bash
curl -sS "$BASE_URL/api/chat?chatId=kbase-lance-smoke"
```

应从 `react-tool` 的 `sources.items[]` sidecar 恢复 `source.publish`，并保留 `liveSeq`。

### Artifact manifest 与审计 sidecar

让 agent 生成一个文件并调用 `artifact_publish`，在 SSE 中确认 `artifact.publish.toolId` 与触发该发布的 `tool.result.toolId` 相同；Team 子任务还应带正确的 `taskId`。检查 `<chatId>/.tools/artifacts.json`：其中的 `chatId`、artifact 元数据、`runId` 与 `publishedAt` 都存在，实际文件仍位于 chat 目录外层的可访问资源路径。

检查 JSONL：只有包含该 `tool.result` 的 `_type:"react-tool"` 行携带 `artifacts.items[]`，且项目仅描述本次批量发布；后续 `react-tool` 行不得复制累计 artifact 快照。再调用 `/api/chat`，确认 `data.artifact.items[]` 来自 manifest。将 chat 归档/恢复并派生到已完成 run，确认归档恢复仍能返回 artifact，而派生 chat 的 manifest 只保留该 run 的项目且 resource URL 指向新 chat。

验证缺 sidecar 时，只在隔离的 release 副本中移除 `bin/kbase-lance-engine[.exe]`；不要操作生产安装目录。KBASE 应返回 unavailable，普通 agent/chat/memory 接口仍可用。若 KBASE storageDir 含旧 `kbase.db` 而没有当前 `control.db`，启动必须先拒绝该目录，不能读取或迁移它。

本地 build cache 由相邻项目主动构建：

```bash
./scripts/sync-local-builtins.sh
```

在 Darwin ARM64，确认 cache 的 `bin/` 包含 `rg`、`dbx`、`httpx`、`kbase-lance-engine` 与 `pdftotext`，且存在 `libexec/poppler-pdftotext/darwin-arm64/`。再次同步后，正式 `scripts/release-assets/builtins.lock.json` 的内容不得变化；`--all` 不得为未在 lock 中声明的 Poppler target 尝试构建。故意令任一构建 archive、Cargo metadata、SBOM、Poppler runtime manifest 或 payload 缺失时必须失败，且不得替换既有 `build/builtins/<target>`。同步脚本不操作 `release-local/bin`。六平台 build/start/search smoke、签名与 notarization 在外部 sidecar 项目的 release CI 执行，不得因为本地同步脚本存在就记录为已验证。

正式 Desktop builtin 发布不需要 platform 仓库保存 `dist/kbase-lance-engine`。在 Darwin ARM64 release runner 上，先确保相邻 builtin collection 中的 lock target archive 已发布，再从干净 platform 目录验证装配：

```bash
make test-release-program-clean
```

该命令会删除本仓库 `dist/`，再直接运行 `make release-program`；不要求 Rust、Cargo 或 `protoc`，但要求锁定 builtin archive 及其 Cargo metadata/SBOM 均已就绪。它会检查最终 bundle 的 sidecar、metadata 和 `verify-program-bundle`；Desktop 仅调用 Platform 的 `make release`，不会传递 sidecar 配置。

## Submit / Steer / Interrupt

`/api/submit` 当前使用统一 HITL 协议：

- 顶层固定是公开 owner + `runId + awaitingId + params`；普通 Agent 使用 `agentKey`，Team 使用 `teamId`
- `params` 永远是数组
- 不在 submit 里再传 `mode`
- 后端按 `awaitingId` 反查当前是 `question / approval / form`
- `params[i]` 固定对应 `awaiting.ask` 下发数组的第 `i` 项；`id` 可省略，即使携带也只作审计用途

question:

```bash
curl -X POST "$BASE_URL/api/submit" \
  -H "Content-Type: application/json" \
  -d '{"agentKey":"default_agent","runId":"replace-me","awaitingId":"tool_question","params":[{"id":"q1","answer":"Weekend"},{"id":"q2","answers":["产品更新","使用教程"]}]}'
```

也可以省略 `id`，只要顺序与 `awaiting.ask.questions` 一致：

```bash
curl -X POST "$BASE_URL/api/submit" \
  -H "Content-Type: application/json" \
  -d '{"agentKey":"default_agent","runId":"replace-me","awaitingId":"tool_question","params":[{"answer":"Weekend"},{"answers":["产品更新","使用教程"]}]}'
```

question 的每个答案必须与同索引问题的题型匹配：`multi-select` 使用非空 `answers`，其他题型使用 `answer`，不能同时传两者。题目数量、数值类型、文本非空以及未开启自由输入时的选项值也会校验；`id` 可省略，携带时只用于审计，仍按顺序配对。

例如向多选题错误提交 `answer`：

```bash
curl -X POST "$BASE_URL/api/submit" \
  -H "Content-Type: application/json" \
  -d '{"agentKey":"default_agent","runId":"replace-me","awaitingId":"tool_question","params":[{"id":"q1","answer":"Weekend"},{"id":"q2","answer":"产品更新"}]}'
```

接口仍返回 HTTP 200，但响应中应为 `data.accepted:false`、`data.status:"invalid"`；等待项保持 awaiting，可将第二项改为 `{"id":"q2","answers":["产品更新"]}` 后重新提交。

approval:

```bash
curl -X POST "$BASE_URL/api/submit" \
  -H "Content-Type: application/json" \
  -d '{"agentKey":"default_agent","runId":"replace-me","awaitingId":"await_tool_bash","params":[{"id":"tool_bash","decision":"approve_rule_run","reason":"同规则本轮一并放行"}]}'
```

form:

```bash
curl -X POST "$BASE_URL/api/submit" \
  -H "Content-Type: application/json" \
  -d '{"agentKey":"default_agent","runId":"replace-me","awaitingId":"await_tool_bash","params":[{"id":"form-1","payload":{"applicant":{"name":"Lin","department":"engineering","employee_id":"E1001"},"leave_type":"事假","start_date":"2026-04-21","end_date":"2026-04-22","duration_days":2,"reason":"family_trip","urgent_contact":"Amy","urgent_phone":"13800138000","backup_person":"E2001","notes":"请协助处理审批"}}]}'

```

在同一 payload 中增加 `"agentKey":"writer"` 应校验失败。

## Agent Workspace File

读取当前 agent workspace 内的 Markdown / 文本文件；`path` 可以是相对 workspace 的路径，也可以是仍位于 workspace 内的绝对路径：

```bash
curl -X GET "$BASE_URL/api/file?agentKey=replace-agent&path=docs%2FREADME.md"
```

PDF、图片或下载预览使用同一个接口追加 `response=content`：

```bash
curl -L -X GET "$BASE_URL/api/file?agentKey=replace-agent&path=docs%2Fmanual.pdf&response=content" -o /tmp/manual.pdf
```

WebSocket 可读取 metadata 和文本内容；例如 JavaScript、Go、Markdown 等可解码文本会出现在响应的 `data.content`：

```json
{"frame":"request","type":"/api/file","id":"file_01","payload":{"agentKey":"replace-agent","path":"webpack.config.js"}}
```

WS 不传文件字节：二进制文件响应只包含 metadata 与 `contentUrl`；`response: "content"` 会返回 `400 invalid_request`，应改用 HTTP `GET /api/file?...&response=content`。文本读取上限由 `tools.yml` 的 `file-tools.max-read-bytes` 控制，默认 1 MiB，超出时响应标记 `truncated: true`。

## Upload / Resource

```bash
curl -X POST "$BASE_URL/api/upload" \
  -F "requestId=upload_01" \
  -F "chatId=replace-me" \
  -F "file=@./README.md"
```

上传成功后，`data.upload.path` 是智能体可读的执行路径；使用响应中的 `data.upload.url` 访问平台资源：

```bash
curl -X GET "$BASE_URL/api/resource?file=replace-me%2FREADME.md"
```

如果部署开启了 resource ticket，需要把 chat 详情或其他接口返回的 ticket 透传到 `t` 参数：

```bash
curl -X GET "$BASE_URL/api/resource?file=replace-me%2FREADME.md&t=replace-me-ticket"
```

说明：

- `/api/file` 面向 agent workspace 文件预览；`/api/resource` 面向 chat 上传资源
- chat 上传资源的浏览器/普通客户端文件字节走 `POST /api/upload` 与 `GET /api/resource`
- WebSocket `/api/upload` 仅用于网关中转 `url + metadata`，由 platform 自己下载文件，不支持直接在 WS payload 中放 base64/二进制内容
