# 工具目录权限

## 当前状态

工具目录权限统一由 `configs/access-policy.yml` 管理。`bash` host shell 与 `file_read`、`file_grep`、`file_write`、`file_edit` 都读取同一套 `AccessPolicy`，避免 bash 拥有比结构化文件工具更宽的 host path 权限。

`configs/bash.yml` 只保留 shell、命令白名单与超时配置；`configs/file-tools.yml` 只保留大小限制、写前读、写审批与 hook 等工具行为配置。

## 核心流程

```text
请求 accessLevel
  -> 选择 AccessPolicy level
  -> 展开 @workspace / @agent / @skills / @skills-market
  -> file tools 精确校验目标路径
  -> bash 先跑 bashsec，再校验 cwd、显式 path args、redirect
  -> 越权按 policy allow / hitl / auto / block 执行
```

FileTools 还维护 run-scoped 读写闭环：已有文件写入或编辑前，默认要求同一 run 内先 `file_read`，防止模型在没有读取最新内容时覆盖用户改动。

## 配置与接口

- 目录权限配置：`configs/access-policy.yml`，模板为 `configs/access-policy.example.yml`。
- Bash 执行配置：`configs/bash.yml`，模板为 `configs/bash.example.yml`。
- FileTools 行为配置：`configs/file-tools.yml`，模板为 `configs/file-tools.example.yml`。
- `default`：workspace 写免审；workspace、agent、当前 agent skills 读免审；其他路径触发 HITL。
- `auto_approve`：沿用 `default` 边界，但可审批项自动批准，bashsec hard block 仍然阻断。
- `full_access`：host path 读写全开，仍保留 bashsec hard block。
- 请求字段 `accessLevel` 可取 `default`、`auto_approve`、`full_access`；缺省为 `default`。
- `require-write-approval` 控制写入是否默认需要审批。
- `require-read-before-write` 控制写前是否必须先读。

常见结构化错误：

- `file_unchanged`
- `file_write_not_read`
- `file_modified_since_read`
- `file_edit_not_read`
- `file_edit_modified_since_read`
- `grep_ripgrep_missing`
- `grep_no_match`
- `grep_invalid_type`

## 约束与注意事项

- 旧 `AGENT_BASH_*` 与 `AGENT_FILE_*` 环境变量会导致启动失败。
- 旧 YAML 字段 `allowed-paths`、`path-checked-commands`、`path-check-bypass-commands`、`allowed-read-paths`、`allowed-write-paths` 会导致启动失败，请迁移到 `configs/access-policy.yml`。
- 工具权限与服务鉴权不是同一层；JWT、resource ticket、CORS 见 [鉴权与安全边界](鉴权与安全边界.md)。
- `approve_rule_run` 只在当前 run 内批准同一规则，不是永久配置。
- 文件工具适合替代 heredoc、重定向等高风险 shell 写文件方式。
- sandbox bash 不走 host path policy；目录边界由 sandbox 挂载控制。
