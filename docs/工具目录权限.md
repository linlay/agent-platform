# 工具目录权限

## 当前状态

工具目录权限分为 Bash 与 FileTools 两套配置。Bash 权限只约束 `bash` host shell 工具；FileTools 权限只约束 `file_read`、`file_grep`、`file_write`、`file_edit`。两套白名单互不继承，避免 shell 权限扩大后自动扩大文件工具读写面。

## 核心流程

```text
工具调用
  -> 解析工作目录和目标路径
  -> 匹配对应工具的 allowed paths
  -> 命中则执行
  -> 越权则生成 HITL approval
  -> 用户 approve / approve_rule_run / reject
```

FileTools 还维护 run-scoped 读写闭环：已有文件写入或编辑前，默认要求同一 run 内先 `file_read`，防止模型在没有读取最新内容时覆盖用户改动。

## 配置与接口

- Bash 外部配置：`configs/bash.yml`，模板为 `configs/bash.example.yml`。
- FileTools 外部配置：`configs/file-tools.yml`，模板为 `configs/file-tools.example.yml`。
- `file_read` / `file_grep` 使用 `allowed-read-paths`。
- `file_write` / `file_edit` 使用 `allowed-write-paths`。
- `require-write-approval` 控制写入是否默认需要审批。
- `require-read-before-write` 控制写前是否必须先读。

常见结构化错误：

- `file_unchanged`
- `file_write_not_read`
- `file_modified_since_read`
- `file_edit_not_read`
- `file_edit_modified_since_read`
- `grep_ripgrep_missing`
- `grep_no_match`
- `grep_invalid_type`

## 约束与注意事项

- 旧 `AGENT_BASH_*` 与 `AGENT_FILE_*` 环境变量会导致启动失败。
- 工具权限与服务鉴权不是同一层；JWT、resource ticket、CORS 见 [鉴权与安全边界](鉴权与安全边界.md)。
- `approve_rule_run` 只在当前 run 内批准同一规则，不是永久配置。
- 文件工具适合替代 heredoc、重定向等高风险 shell 写文件方式。
