# 工具目录权限

## 当前状态

工具目录权限统一由 `configs/host-tools.yml` 中的 `access-policy` 管理。`bash` host shell 与 `file_read`、`file_glob`、`file_grep`、`file_write`、`file_edit` 都读取同一套 `AccessPolicy`，避免 bash 拥有比结构化文件工具更宽的 host path 权限。

`configs/host-tools.yml` 同时包含 `bash` shell、命令白名单与超时配置，以及 `file-tools` 大小限制、写前读、写审批与 hook 等工具行为配置。旧拆分文件不再读取。

## 核心流程

```text
请求 accessLevel
  -> 选择 AccessPolicy level
  -> 展开 @workspace / @chat / @agent / @skills / @skills-market / @owner
  -> 追加 agent runtimeConfig.hostAccess readRoots/writeRoots
  -> file tools 精确校验目标路径
  -> bash 先跑 bashsec，再校验 cwd、显式 path args、redirect
  -> 越权按 policy allow / hitl / auto / block 执行
```

FileTools 还维护 run-scoped 读写闭环：已有文件写入或编辑前，默认要求同一 run 内先 `file_read`，防止模型在没有读取最新内容时覆盖用户改动。

## 配置与接口

- Host 工具配置：`configs/host-tools.yml`，模板为 `configs/host-tools.example.yml`。
- `access-policy`：目录权限配置。
- `bash`：host shell、命令白名单与超时配置。
- `file-tools`：结构化文件工具行为配置。
- `default`：workspace、chat 写免审；workspace、chat、agent、当前 agent skills 读免审；其他路径触发 HITL。
- `auto_approve`：沿用 `default` 边界；越界读、复杂 bash 与不透明命令自动批准，越界写仍触发 HITL，bashsec hard block 仍然阻断。
- `full_access`：host path 读写全开，仍保留 bashsec hard block。
- `runtimeConfig.hostAccess.readRoots/writeRoots` 可为单个 agent 追加 host-side `bash/file_*` 读写根；该配置不影响 Container Hub / sandbox，sandbox 挂载使用 `runtimeConfig.sandboxMounts`。
- 请求字段 `accessLevel` 可取 `default`、`auto_approve`、`full_access`；缺省为 `default`。
- `/api/access-level` 可在 native run 运行中动态更新当前 run 的 `accessLevel`；该更新不写回 agent 配置，也不会影响下一次 run。
- 动态更新从下一次工具权限判断开始生效，已经开始执行的 host bash / file tool 不会被中断；若当前正等待 access-policy approval，更新后会重新评估，满足新权限时自动继续。
- PROXY / ACP CODER run 暂不支持动态更新，接口返回 `status=unsupported`。
- `auto_approve` 的自动批准会作为 `messages[]` 内联 `role=user` 审计消息写入当前 `chatId.jsonl`，消息内的 `approval.decisions[].decision` 为 `auto_approved`；`full_access` 因为路径根已扩为 `/`，不产生越界审批记录。
- `require-write-approval` 控制写入是否默认需要审批。
- `require-read-before-write` 控制写前是否必须先被文件工具观察到。
- `read-before-write-scope` 控制观察范围：`run` 仅限当前 run；`chat` 会在 `<chatId>/.tools/state/file-versions.json` 保存文件版本证明，并允许同 chat 后续 run 在 sha/mtime/size 校验通过时写入。

常见结构化错误：

- `file_unchanged`
- `file_write_not_read`
- `file_modified_since_read`
- `file_edit_not_read`
- `file_edit_modified_since_read`
- `glob_ripgrep_missing`
- `glob_invalid_path`
- `grep_ripgrep_missing`
- `grep_invalid_type`

`file_glob` 用于按文件名/路径 glob 查找文件，`file_grep` 用于按内容正则搜索。两者无命中都不作为错误返回；结果会是 `matchCount: 0`、`results: []`。

## 约束与注意事项

- 旧 YAML 字段 `bash.allowed-paths`、`bash.path-checked-commands`、`bash.path-check-bypass-commands`、`file-tools.allowed-read-paths`、`file-tools.allowed-write-paths` 会导致启动失败，请迁移到 `configs/host-tools.yml` 的 `access-policy`。
- 工具权限与服务鉴权不是同一层；JWT、resource ticket、CORS 见 [鉴权与安全边界](鉴权与安全边界.md)。
- `approve_rule_run` 只在当前 run 内批准同一规则，不是永久配置。
- 文件工具适合替代 heredoc、重定向等高风险 shell 写文件方式。
- sandbox bash 不走 host path policy；目录边界由 sandbox 挂载控制。
