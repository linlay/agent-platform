# 子智能体调度

## 当前状态

运行时有两套共享底层执行器、但入口和权限不同的调度机制：

- `agent_invoke`：普通主 Agent 显式声明的批量子智能体工具。
- TEAM 调度：orchestrated Team 的隐藏协调器通过 session-local `team_delegate` / `team_invoke` 集中调度成员。

TEAM 是 `internal/agent/team` 提供的内部 mode，不是可在普通 `agent.yml` 中声明的 Agent 类型。隐藏协调器只在 Team run 内合成，不进入 `/api/agents`、`/api/agent`、`/api/admin/tools` 或普通 `agent_invoke` 目标列表。

## `agent_invoke`

```text
普通主 Agent 调用 agent_invoke
  -> 主时间线保留 tool 事件
  -> server 编排层并发启动 1~5 个子 Agent
  -> 子流实时汇聚到主 SSE / replay
  -> 子结果按输入顺序聚合
  -> 一次 InjectToolResult 返回主 Agent
```

- 工具定义：`internal/resources/tools/agent_invoke.yml`。
- 暴露方式：Agent 在 `toolConfig.tools` 显式声明 `agent_invoke`。
- 子任务数量：1–5。
- 可用主 Agent mode：`REACT`、`ONESHOT`、`CODER`。
- 可用 child mode：`REACT`、`ONESHOT`、`CODER`、`KBASE`、`PROXY`；ACP CODER 当前不能作为 child。
- 主 chat 有 `teamId` 时，child 必须属于 run 开始时冻结的 Team 快照。

## TEAM 隐藏调度

目录式 `runtime/teams/<teamId>/team.yml` 会创建 orchestrated Team。协调器首轮模型调用固定要求从两个隐藏工具中选择：

```text
用户请求
  -> 隐藏 TEAM 协调器
     -> team_delegate direct -> 一个成员 -> 成员正文直接成为根回答
     -> team_delegate fanout -> 全体成员并行/有界并发 -> 展示成员回复 -> Team 总结
     -> team_invoke batch -> 1..maxParallel 个内部任务 -> 结果回注协调器
        -> 可继续下一批，或输出最终答案
```

成员必须具备 `RunAsChild` capability；当前 native `REACT`、`ONESHOT`、`CODER`、`KBASE` 与 `PROXY` 可走共享 child 执行路径，ACP CODER 当前明确失败且不会回退为 native CODER。

### `team_delegate`

- `mode: direct` 必须指定一个冻结 roster 中的 `memberKey`。
- `mode: fanout` 不接受 `memberKey`，并自动覆盖所有有效成员。
- direct/fanout 都把当前用户的原始 message、references、role 与 scene 交给成员，不由协调器改写任务。
- direct 成功后终止协调器回合，不生成二次改写或总结；失败结果回注协调器，由其改派或解释。
- fanout 的单成员失败不取消其他成员；成员结果全部回注后，下一次协调器调用禁用工具并生成总结。

### `team_invoke`

- `tasks[]` 每项包含 `memberKey`、`task`，可选 `taskName/files`。
- 单批上限为 Team 的 `orchestrator.maxParallel`，默认 5，最大 5。
- 同一调用中的任务并行；协调器收到结果后可以继续调用下一批，从而形成串行、分支或重试。
- task 使用协调器编写的独立任务文本，不继承共享 chat history；`files` 单独转换为该任务的 references。
- v1 没有 DAG 配置 DSL，编排顺序由协调器连续调用隐藏工具表达。

## 事件、历史与隐私

子任务沿用 `task.start`、`task.complete`、`task.cancel`、`task.error`，`runId` 始终是根 run；`taskId` 仅用于 UI 归并，不能独立 attach、steer 或 interrupt。

orchestrated Team 额外使用以下展示元数据：

- 成员 content/task 事件携带 `teamId`、成员 `agentKey`、`actor:{type:"agent",...}`。
- `presentation:"reply"` 表示 direct/fanout 的用户可见成员回复；`presentation:"task"` 表示 `team_invoke` 内部任务展示。
- direct content 不带 `taskId`，会进入根 run 最终结果；fanout/invoke task content 不会被拼接到根 `content`，根结果取 Team 总结或协调器最终正文。
- 协调器 reasoning、`team_delegate/team_invoke` 工具卡和原始工具结果不发送给客户端。

历史构建保持角色隔离：

- child `_type:"query"` 只用于 replay/system-init 定位，不会作为后续共享 user message，避免泄漏内部任务 prompt。
- direct/fanout 成员会读取 Team member 专用历史：保留自己的 tool chain，只读取其他成员最终正文和 Team 最终正文，不读取其他成员 reasoning/tool result。
- `team_invoke` 的任务 prompt、reasoning、tool chain 和中间结果不进入后续共享历史。
- 根协调器使用 actor-scoped 历史：保留用户轮次、Team 最终正文和带成员标识的成员最终正文，排除 child prompt、reasoning、tool chain 与原始中间结果。

## HITL 与限制

- 子 Agent 不注册独立 active run。direct 成员 HITL 仍发布在根 run，并带 `taskId`；public `awaitingId` 使用 `taskId:rawAwaitingId`，submit 时映射回 child 的 raw ID。
- Team-owned run 的 submit 使用 `teamId + runId + awaitingId`，不能传隐藏协调器 key。
- fanout / `team_invoke` 的多个并行 awaiting 合并为一个 Team-level form；外层字段 id 是可逆的 `taskId:rawAwaitingId`，一次 submit 经 schema 校验后分发给各隔离 child RunControl。超过 `maxParallel` 时按波次继续合并，不会因等待占用并发槽而死锁。
- 所有 child session 都移除 `agent_invoke`；即使成员配置中包含该工具，嵌套调度也会被拒绝。
- Team 不能作为另一个 Team 的成员或 child 目标。
- interrupt 作用于整个根 run；steer 写入协调器 run，不直接选择某个成员。
