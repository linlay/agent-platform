# 子智能体调度

## 当前状态

运行时有两个用途和权限不同的 embedded builtin：

- `agent_invoke`：普通主 Agent 显式声明后调用普通子智能体。
- `agent_delegate`：仅供 orchestrated Team 的隐藏总控委派当前 Team 成员。

`agent_invoke` 保持原有契约。`agent_delegate` 定义在 `internal/resources/tools/agent_delegate.yml`，由 Team 自动、强制注入；普通 `agent.yml` 声明该工具会在 catalog 加载时失败，普通 session 与 backend 直接执行也会拒绝。隐藏总控和 `agent_delegate` 均不进入普通 Agent 或 Tool catalog。

## `agent_invoke`

```text
普通主 Agent 调用 agent_invoke
  -> 主时间线保留 tool 事件
  -> server 并发启动 1~5 个普通子 Agent
  -> 子流实时汇聚到主 SSE / replay
  -> 子结果按输入顺序聚合并回注主 Agent
```

- Agent 必须在 `toolConfig.tools` 显式声明 `agent_invoke`。
- 每项必须包含 `subAgentKey` 和 `task`，可选 `taskName/files`。
- 可用主 Agent mode：`REACT`、`ONESHOT`、`CODER`。
- 可用 child mode：`REACT`、`ONESHOT`、`CODER`、`KBASE`、`PROXY`；ACP CODER 当前不能作为 child。
- child session 禁止再次调用子智能体。
- TEAM 隐藏总控不能调用 `agent_invoke`，TEAM 成员也不能携带该工具。

## TEAM 的 `agent_delegate`

目录式 `runtime/teams/<teamId>/team.yml` 创建 orchestrated Team。每个 run 从 embedded definition 克隆 `agent_delegate` schema，并按冻结 roster 动态补充成员枚举和任务上限。

隐藏总控固定拥有四个工具：

```text
agent_delegate
plan_add_tasks
plan_get_tasks
plan_update_task
```

### 参数

`agent_delegate` 只接受 `tasks`：

| 参数 | 必填 | 约束 | 语义 |
|---|---|---|---|
| `tasks` | 是 | 数组，至少 1 项；`maxItems` 等于冻结 roster 大小 | 本次调用需要委派的成员任务 |
| `tasks[].agentKey` | 是 | 必须是动态 `enum` 中的精确成员 key；同一次调用不可重复 | 目标 Team 成员 |
| `tasks[].task` | 否 | string | 自定义任务文本；省略或为空时使用原始用户消息 |
| `tasks[].taskName` | 否 | string | `task.*` 进度事件的标题 |
| `tasks[].files` | 否 | 最多 10 项 | 附加文件或当前 chat 资源，转换成 references 后与原始 references 去重 |

`additionalProperties:false` 同时作用于工具根对象和每个 task。`maxParallel` 不进入工具 schema 的任务上限：它只控制 runtime 同时运行的成员数；`tasks` 数量超过 `maxParallel` 时，仍属于同一次工具调用，由 runtime 分波执行并一次性聚合结果。

成员始终继承原始请求的 `role`、`scene`、`references` 和该成员可见的 Team 历史。自定义 `task` 只替换 message，不会丢失这些上下文。

### 场景枚举

明确由一个成员回答与给单个成员自定义任务是同一条执行路径：

```json
{"tasks":[{"agentKey":"writer"}]}
```

```json
{
  "tasks": [
    {
      "agentKey": "writer",
      "task": "只检查文档中的事实错误",
      "taskName": "事实检查"
    }
  ]
}
```

无法判断合适成员与明确选择多个成员也是同一条多项委派路径：

```json
{
  "tasks": [
    {"agentKey":"writer"},
    {"agentKey":"reviewer"},
    {"agentKey":"researcher"}
  ]
}
```

```json
{
  "tasks": [
    {"agentKey":"researcher","task":"收集事实依据"},
    {"agentKey":"reviewer","task":"检查潜在风险"}
  ]
}
```

一项和多项没有不同的终止规则。每项都产生 `task.start` 和 `task.complete/error/cancel`，全部结果都回注隐藏总控，随后由总控更新计划、继续委派或生成唯一的 Team 最终回答。

工具结果统一为：

```json
{
  "results": [
    {
      "agentKey": "reviewer",
      "taskName": "风险检查",
      "status": "completed",
      "content": "最终成员结果"
    }
  ]
}
```

失败项可增加 `error`、`errorCode`。一个成员失败不会取消同批其他成员；单成员失败和多成员部分失败也走相同的结构化返回路径，由总控决定改派、解释或继续。

## 计划与执行状态

```text
routing -> waiting_members -> coordinator -> complete
```

- 简单请求可以直接调用一次 `agent_delegate`。
- 复杂请求先用 `plan_add_tasks` 创建完整任务列表，再用 `plan_update_task` 维护 `in_progress/completed/failed`，并按当前可运行任务调用 `agent_delegate`。
- `plan_get_tasks` 用于读取当前计划；计划工具不自动启动成员。
- 串行、分支和重试由计划状态及后续总控轮次表达，`agent_delegate` 本身没有 direct、fanout、invoke、DAG 或串行模式。
- 每个用户回合至少成功发起一次 `agent_delegate` 后，总控才能输出最终答案；只调用计划工具不算完成路由。

## 事件、历史与隐私

- 成员事件携带根 `runId`、task `taskId`、`teamId`、成员 `agentKey`、`actor.type:"agent"` 和 `presentation:"task"`。
- 成员正文、reasoning 和 tool chain 不会成为根回答；根 `content`、run summary 与 `AssistantText` 只取隐藏总控的最终 Team 回答。
- 协调器 reasoning、`agent_delegate` 工具卡和原始工具结果不发送给客户端，也不进入普通工具 catalog。
- 成员读取专用 Team 历史：保留自己的 tool chain，只读取其他成员最终正文和 Team 最终正文，不读取其他成员 reasoning/tool result。
- 根总控历史保留用户轮次、Team 最终正文和带成员标识的成员最终正文，排除 child prompt、reasoning、tool chain 与原始中间结果。
- 历史回放仍识别旧 `team_delegate` / `team_invoke` 调用，但只转换为 `agent_delegate agentKeys=...` 安全摘要；新 run 不接受旧工具名。

## HITL 与限制

- 成员不注册独立 active run；public `awaitingId` 使用 `taskId:rawAwaitingId`，submit 时映射回 child raw ID。
- 同一并发波次的多个 awaiting 合并为一个 Team-level form，一次 submit 校验并分发给各 child；超过 `maxParallel` 时后续波次继续合并。
- Team-owned run 的 submit 使用 `teamId + runId + awaitingId`，不能传隐藏总控 key。
- interrupt 作用于整个根 run；steer 写入总控 run，不直接选择某个成员。
- Team 不能作为另一个 Team 的成员或 child 目标。
