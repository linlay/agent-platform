# 子智能体调度

## 当前状态

`agent_invoke` 是显式配置的批量子智能体调度原语。只有 agent 在 `toolConfig.tools` 中声明 `agent_invoke` 时，模型才会看到该能力。

## 核心流程

```text
主 agent 调用 agent_invoke
  -> 主时间线保留 tool.start / tool.args / tool.end / tool.snapshot
  -> server 编排层启动 1~5 个子 agent
  -> 子流实时汇聚回主 SSE / events replay
  -> 子结果按输入顺序聚合
  -> 单次 InjectToolResult 给主 agent
```

子 agent 使用 `task.start`、`task.complete`、`task.cancel`、`task.error` 协议。`runId` 始终保持主 RunID，前端通过 `taskId` 归并子面板，通过 `invokingToolId` 关联主时间线上的聚合卡片。`taskId` 不是 active `runId`，不能作为独立运行 attach / interrupt / steer 的目标。

子 agent 触发 HITL 时，`awaiting.ask` 仍发布在主 run 上，并额外携带 `taskId`。为了避免多个子任务中的 tool call id 冲突，前端看到的 public `awaitingId` 会写成 `taskId:rawAwaitingId`；`/api/submit` 内部会把它映射回子 agent 工具实际等待的 `rawAwaitingId`。历史兼容上，submit 如果传入 `runId=taskId`，后端会尽量解析回主 run，但新客户端应使用 `awaiting.ask.runId` 中的主 runId。

## 配置与接口

- 工具定义：`internal/resources/tools/agent_invoke.yml`
- 暴露方式：agent `toolConfig.tools` 显式声明 `agent_invoke`
- 子任务数量：`1 <= tasks <= 5`
- 可用主 agent mode：`REACT`、`ONESHOT`、`CODER`
- 可用子 agent mode：`REACT`、`ONESHOT`、`CODER`、`KBASE`、`PROXY`

这里的 mode 列表由 session capability 控制，不存在 `TEAM` mode。ACP backend 的 CODER 可以作为 Team 顶层 agent，但当前不能作为 child；调度时会返回明确的 task error，不会回退到 native CODER。

## 约束与注意事项

- 当前严格禁止嵌套，子 agent 的可用工具集中会滤掉 `agent_invoke`。
- 主 chat 有 `teamId` 时，child 必须属于 run 开始时固定的 Team 快照，并使用该快照中冻结的成员 `AgentDefinition`；本轮 catalog 热重载不会替换 child 定义。Team 本身不会授予 `agent_invoke`。
- child request 从空对象构造，只继承主 run 的 `chatId`、`runId`、固定 `teamId` 与 `accessLevel`，并设置独立 `requestId`、目标 `agentKey`、task message 和 user role。
- child 不继承父请求的 `params`、`model`、`planningMode`、`scene`、`stream`、include flags、synthetic 标记或父 references；references 只由当前 task 的 `files` 生成。
- 三流分离是硬约束：子 agent 中间 reasoning、tool 调用、content delta 只进入 SSE/events replay，不进入主 `llmRunStream.messages`，也不进入 `raw_messages`。
- 主上下文只消费子 agent 最终聚合后的 `tool.result` 文本。
- `dispatcher.activeTaskID` 仍是单值；输入事件缺少 `taskId` 时会兜底到当前活跃 task。
